## 问题场景

在做 RAG 知识库时，我们选型遇到了一个实际问题：用哪个向量数据库？

团队里有人用过 Redis，有人听过 Milvus，还有人建议用 Qdrant。各有各的道理，但没人能说清楚在我们的场景下到底该选哪个。

我们的场景是：知识库规模约 10 万条文档片段，需要支持实时更新，QPS 预估 50-100，P99 延迟要求 200ms 以内。

这篇文章记录了我们从选型到落地的完整过程，以及踩过的坑。

## 分析思路

Spring AI 的 VectorStore 抽象层屏蔽了底层实现差异，理论上切换向量库只需要换依赖和配置。但实际上每个向量库的能力差异不小，特别是在过滤查询、索引管理、运维成本方面。

选型评估维度：
- 性能：检索延迟、并发能力
- 运维：部署复杂度、监控、备份
- 功能：过滤查询、混合搜索、索引类型
- 成本：存储、内存、许可证

## 具体实现

### Embedding 生成

不管用哪个向量库，第一步都是把文本转成向量。DashScope 提供了 `text-embedding-v2` 模型：

```java
@Service
@Slf4j
@RequiredArgsConstructor
public class EmbeddingService {

    private final EmbeddingModel embeddingModel;

    /**
     * 将文本列表转为向量
     * DashScope text-embedding-v2 支持批量请求
     */
    public List<float[]> embed(List<String> texts) {
        log.info("【Embedding】生成向量, 文本数: {}", texts.size());

        EmbeddingResponse response = embeddingModel.embed(
                new EmbeddingRequest(texts, EmbeddingOptions.builder()
                        .model("text-embedding-v2")
                        .build()));

        List<float[]> vectors = response.getResults().stream()
                .map(r -> {
                    double[] doubles = r.getOutput();
                    float[] floats = new float[doubles.length];
                    for (int i = 0; i < doubles.length; i++) {
                        floats[i] = (float) doubles[i];
                    }
                    return floats;
                })
                .toList();

        log.info("【Embedding】向量维度: {}", vectors.isEmpty() ? 0 : vectors[0].length);
        return vectors;
    }

    /**
     * 单条文本的 Embedding
     */
    public float[] embedOne(String text) {
        return embed(List.of(text)).get(0);
    }
}
```

**踩坑点**：DashScope 的 embedding 接口有批量限制，单次最多 25 条文本。如果要处理大量文档，需要自己做分批请求。

### 方案一：Milvus 集成

```yaml
# application-milvus.yml
spring:
  ai:
    vectorstore:
      milvus:
        host: ${MILVUS_HOST:localhost}
        port: ${MILVUS_PORT:19530}
        database-name: default
        collection-name: knowledge_base
        index-type: IVF_FLAT
        metric-type: COSINE
        dimension: 1536
        # 建索引的参数
        nlist: 1024
```

```xml
<dependency>
    <groupId>org.springframework.ai</groupId>
    <artifactId>spring-ai-milvus-store-spring-boot-starter</artifactId>
</dependency>
```

```java
@Service
@Slf4j
@RequiredArgsConstructor
public class MilvusSearchService {

    private final VectorStore vectorStore;

    /**
     * 基础相似度搜索
     */
    public List<Document> search(String query, int topK) {
        SearchRequest request = SearchRequest.query(query)
                .withTopK(topK)
                .withSimilarityThreshold(0.6);
        return vectorStore.similaritySearch(request);
    }

    /**
     * 带元数据过滤的搜索
     * Milvus 支持标量过滤 + 向量搜索的混合查询
     */
    public List<Document> searchWithFilter(String query, String category, int topK) {
        SearchRequest request = SearchRequest.query(query)
                .withTopK(topK)
                .withSimilarityThreshold(0.5)
                // Milvus 过滤表达式语法
                .withFilterExpression("category == '" + category + "'");
        return vectorStore.similaritySearch(request);
    }

    /**
     * 范围过滤：只搜索最近 7 天更新的文档
     */
    public List<Document> searchRecent(String query, int topK) {
        long sevenDaysAgo = Instant.now().minus(7, ChronoUnit.DAYS).getEpochSecond();
        SearchRequest request = SearchRequest.query(query)
                .withTopK(topK)
                .withFilterExpression("updateTimestamp > " + sevenDaysAgo);
        return vectorStore.similaritySearch(request);
    }
}
```

### 方案切换：统一接口

由于 Spring AI 的 VectorStore 抽象，切换方案只需要改配置：

```java
@Configuration
@Profile("milvus")
public class MilvusVectorStoreConfig {
    // Milvus 的 VectorStore 由 starter 自动装配
    // 这里可以放 Milvus 特有的配置
}

@Configuration
@Profile("qdrant")
public class QdrantVectorStoreConfig {
    // Qdrant 的配置
}

@Configuration
@Profile("redis")
public class RedisVectorStoreConfig {
    // Redis 的配置
}
```

业务代码完全不用改，只需要切换 `spring.profiles.active`。

### 向量库管理工具

```java
@RestController
@RequestMapping("/api/vector")
@RequiredArgsConstructor
@Slf4j
public class VectorStoreAdminController {

    private final VectorStore vectorStore;

    /**
     * 批量添加文档（带进度跟踪）
     */
    @PostMapping("/batch-add")
    public Map<String, Object> batchAdd(@RequestBody List<DocumentDto> docs) {
        log.info("【批量入库】文档数: {}", docs.size());

        // 转换为 Document 对象
        List<Document> documents = docs.stream()
                .map(d -> new Document(d.content(), d.metadata()))
                .toList();

        // 分批入库，每批 100 条
        int batchSize = 100;
        int successCount = 0;
        for (int i = 0; i < documents.size(); i += batchSize) {
            List<Document> batch = documents.subList(i,
                    Math.min(i + batchSize, documents.size()));
            try {
                vectorStore.add(batch);
                successCount += batch.size();
                log.debug("【批量入库】进度: {}/{}", successCount, documents.size());
            } catch (Exception e) {
                log.error("【批量入库失败】批次: {}, 错误: {}", i / batchSize, e.getMessage());
            }
        }

        return Map.of("total", docs.size(), "success", successCount);
    }

    /**
     * 按元数据删除文档
     * 注意：不是所有 VectorStore 都支持按条件删除
     */
    @DeleteMapping("/by-source")
    public Map<String, Object> deleteBySource(@RequestParam String source) {
        log.info("【删除文档】来源: {}", source);
        vectorStore.delete(
                SearchRequest.query("")
                        .withFilterExpression("source == '" + source + "'")
                        .getFilterExpression());
        return Map.of("success", true, "source", source);
    }

    public record DocumentDto(String content, Map<String, Object> metadata) {}
}
```

## 选型对比结果

| 维度 | Milvus | Qdrant | Redis |
|------|--------|--------|-------|
| 检索延迟（万级数据） | 5-15ms | 5-20ms | 2-10ms |
| 部署复杂度 | 较高（需 etcd/MinIO） | 低（单 binary） | 低（已广泛部署） |
| 过滤能力 | 强（标量+向量混合） | 强 | 一般 |
| 内存占用 | 中等 | 低 | 高（全内存） |
| 数据规模上限 | 十亿级 | 千万级 | 百万级 |
| 运维工具 | Attu UI | 自带 Web UI | RedisInsight |

我们的最终选择：数据量在 50 万以下用 Qdrant（轻量好运维），超过 50 万用 Milvus（扩展性好）。Redis 方案适合已有 Redis 集群、数据量不大的团队。

## 踩坑总结

1. **维度必须匹配**：DashScope text-embedding-v2 输出 1536 维。建 collection 时维度设错了，数据能写进去但搜索结果全是乱的，而且不会报错。这是最隐蔽的 bug。

2. **COSINE 和 L2 不能混用**：创建索引时选的距离度量，跟搜索时用的必须一致。DashScope 的 embedding 模型输出是归一化的，用 COSINE 更合适。

3. **Milvus 的 nlist 参数**：IVF 索引的 nlist 越大，搜索越精确但越慢。经验公式是 `nlist = sqrt(数据量)`。万级数据设 128-256 就够了。

4. **Redis 的向量搜索性能**：Redis 的向量搜索是用 RedisSearch 模块实现的，不是原生 Redis 命令。如果没有装 RedisSearch 插件，启动就会报错。另外 Redis 全内存，10 万条 1536 维的向量大约占 600MB 内存。

5. **Qdrant 的 payload 索引**：如果要按元数据过滤搜索，需要先给对应的字段建 payload 索引，否则过滤条件不生效（也不会报错）。这一点容易被忽略。

6. **批量写入要控制节奏**：一次性写入太多数据，向量库可能 OOM 或者响应变慢。建议每批 100-500 条，中间 sleep 一下。Milvus 官方建议每批不超过 1000 条。

7. **Embedding 是性能瓶颈**：在我们的场景里，生成 Embedding 的时间比向量搜索本身长 10 倍以上。如果对延迟敏感，可以考虑在写入时预计算 Embedding 并缓存，搜索时只做向量匹配。
