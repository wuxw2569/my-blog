## 问题场景

我们的系统原来都是同步调用：下单后同步扣库存、同步发短信、同步写日志、同步推推荐。结果一个下游服务超时，上游全部阻塞，级联故障。

老板拍板："上消息队列，解耦异步化。"

然后团队就开始纠结了：RabbitMQ？RocketMQ？Kafka？网上文章众说纷纭，我们实际做了调研和压测后，做了一个适合我们团队的选择。

## 分析思路

### 1. 选型对比：不是选最好的，是选最合适的

我们团队的背景：后端8人，没有专职中间件运维，日均消息量500万-2000万，对消息延迟敏感（订单场景要求50ms内），部分场景需要事务消息。

| 维度 | RabbitMQ 3.12 | RocketMQ 5.1 | Kafka 3.6 |
|------|---------------|---------------|-----------|
| 吞吐量 | 万级/秒 | 十万级/秒 | 百万级/秒 |
| 延迟 | 微秒级 | 毫秒级 | 毫秒级 |
| 语言 | Erlang（难搞源码） | Java（友好） | Scala/Java |
| 事务消息 | 不支持（有TX模式但很弱） | 支持（半消息机制） | 支持（幂等+事务） |
| 延迟消息 | 插件支持 | 原生支持18个级别 | 不原生支持 |
| 运维难度 | 中等 | 低（nameserver简单） | 高（依赖ZK/KRaft） |
| 社区生态 | 国内少，国外多 | 国内多（阿里系） | 全球主流 |

**我们的选择：RocketMQ**

理由很实际：
1. **Java 栈友好**：团队全是 Java 开发，出问题能看源码排查
2. **事务消息原生支持**：订单支付回调场景必须有
3. **延迟消息原生支持**：30分钟未支付自动取消订单
4. **运维简单**：NameServer 比 ZooKeeper 简单太多
5. **阿里系背书**：国内中小厂用的最多，踩坑经验好找

为什么不选 Kafka？日均2000万的消息量用 Kafka 有点杀鸡用牛刀，而且 Kafka 运维复杂（虽然 KRaft 模式简化了），团队没有大数据工程师维护。为什么不选 RabbitMQ？Erlang 生态对我们不友好，出了问题看不懂源码，而且 RabbitMQ 的事务消息支持较弱。

### 2. 消息丢失的三道防线

消息丢失可能发生在三个环节：生产者 -> Broker -> 消费者。

**防线一：生产端——同步发送 + 失败重试**

```java
@Service
public class OrderMessageProducer {

    @Autowired
    private RocketMQTemplate rocketMQTemplate;

    public void sendOrderCreatedMessage(OrderDTO order) {
        Message<OrderDTO> message = MessageBuilder.withPayload(order)
            .setHeader("KEYS", order.getOrderId().toString())
            .build();

        // 同步发送，等待 Broker 确认
        // 异步发送虽然快，但出问题不好排查
        try {
            SendResult result = rocketMQTemplate.syncSend(
                "TOPIC_ORDER_CREATED:" + order.getUserId() % 16,
                message
            );

            if (result.getSendStatus() != SendStatus.SEND_OK) {
                log.error("【消息发送失败】orderId={}, status={}",
                    order.getOrderId(), result.getSendStatus());
                // 入库待重试
                saveToRetryTable(order);
            }
        } catch (Exception e) {
            log.error("【消息发送异常】orderId={}", order.getOrderId(), e);
            saveToRetryTable(order);
        }
    }
}
```

**防线二：Broker 端——同步刷盘 + 主从同步**

```properties
# broker.conf
# 同步刷盘（默认是异步，性能差一倍但不丢消息）
flushDiskType=SYNC_FLUSH
# 同步复制（默认是异步，延迟高一点但更安全）
brokerRole=SYNC_MASTER
```

**注意**：同步刷盘 + 同步复制会降低吞吐量。我们的做法是核心topic（订单、支付）用同步策略，非核心topic（日志、埋点）用异步策略。

**防线三：消费端——手动 ACK + 业务幂等**

```java
@Component
@RocketMQMessageListener(
    topic = "TOPIC_ORDER_CREATED",
    consumerGroup = "CG_STOCK_SERVICE"
)
public class StockConsumer implements RocketMQListener<OrderDTO> {

    @Override
    public void onMessage(OrderDTO order) {
        try {
            // 业务处理：扣减库存
            stockService.deductStock(order.getOrderId(), order.getItems());

            // 手动 ACK（Spring RocketMQ 默认自动 ACK，但业务异常时消息就丢了）
            // 实际上 Spring RocketMQ 不支持手动ACK，需要配置消费失败重试
        } catch (Exception e) {
            log.error("【消费失败】orderId={}", order.getOrderId(), e);
            // 抛出异常触发 RocketMQ 重试机制
            throw new RuntimeException("消费失败，触发重试", e);
        }
    }
}
```

### 3. 消息重复的处理——幂等是必须的

RocketMQ 有重试机制，网络抖动导致 ACK 丢失就会重投。业务端必须做幂等。

**方案：基于消息ID的幂等表**

```java
@Service
public class StockService {

    @Transactional
    public void deductStock(Long orderId, List<OrderItem> items) {
        // 1. 幂等检查
        String messageKey = "stock:" + orderId;
        int inserted = messageMapper.insertIfNotExist(messageKey);
        if (inserted == 0) {
            log.info("【消息已处理，跳过】orderId={}", orderId);
            return;
        }

        // 2. 业务逻辑
        for (OrderItem item : items) {
            int rows = stockMapper.deductStock(item.getSkuId(), item.getQuantity());
            if (rows == 0) {
                throw new StockInsufficientException(item.getSkuId());
            }
        }
    }
}
```

```sql
-- 消息幂等表
CREATE TABLE t_message_processed (
    message_key VARCHAR(128) PRIMARY KEY,
    processed_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_processed_at (processed_at)
) ENGINE=InnoDB;
```

### 4. 顺序消息的处理

有些场景需要严格顺序：订单状态变更（创建->支付->发货->签收）。如果乱序，"发货"消息先到，"支付"消息后到，状态就错了。

RocketMQ 的方案是 **MessageQueueSelector**，把同一个 orderId 的消息路由到同一个队列：

```java
public void sendOrderStatusChange(OrderStatusEvent event) {
    rocketMQTemplate.syncSendOrderly(
        "TOPIC_ORDER_STATUS",
        event,
        event.getOrderId().toString()  // 用orderId做hash，保证同一订单进同一队列
    );
}
```

消费端用单线程消费：

```java
@RocketMQMessageListener(
    topic = "TOPIC_ORDER_STATUS",
    consumerGroup = "CG_ORDER_STATUS",
    consumeMode = ConsumeMode.ORDERLY  // 顺序消费模式
)
public class OrderStatusConsumer implements RocketMQListener<OrderStatusEvent> {
    // 单队列单线程，保证顺序
}
```

## 踩坑总结

1. **不要用异步发送处理核心业务**。异步发送虽然快，但回调失败了不好补救。订单、支付类消息老老实实用同步发送。
2. **同步刷盘 + 同步复制只用于核心 Topic**。全部用同步策略会把吞吐量打到原来的1/3，按业务分级处理。
3. **消费端必须做幂等**。不要假设消息只投递一次，网络、重启、重试都会导致重复。幂等表 + 唯一键是最简单可靠的方案。
4. **顺序消息的代价是吞吐量**。同一队列只能单线程消费，队列数决定了消费并行度。非强顺序的场景不要用顺序消费。
5. **消息积压时不要慌**。临时扩消费者实例、跳过非关键消息、事后补偿。我们有次大促积压了800万条，临时起了一组临时消费者，2小时追平。
6. **死信队列一定要监控**。消费失败重试16次后会进死信队列，这些消息需要人工处理。我们配了钉钉告警，有消息进死信就立刻通知。
