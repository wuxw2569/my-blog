## 问题场景

客服系统上线后，用户反馈说"每次问完一个新问题，它就把前面的忘了"。原因很简单：我们之前每次请求都只发送当轮的用户消息，没有带历史对话。大模型是无状态的，你不传历史记录，它当然记不住。

"那把历史消息全带上不就行了？" -- 我一开始也这么想。但很快发现两个问题：

1. 历史越积越多，Token 费用直线上升
2. 超过模型的上下文窗口限制后，直接报错

这就需要一套对话记忆管理方案：既要让模型记住关键上下文，又不能让 Token 失控。

## 分析思路

Spring AI 提供了 `ChatMemory` 抽象，内置了几种记忆策略：

1. **MessageWindowChatMemory**：保留最近 N 条消息，丢弃更早的。简单粗暴但有效。
2. **TokenWindowChatMemory**：按 Token 数量控制窗口，更精确。
3. **SummaryChatMemory**：把早期对话压缩成摘要，既省 Token 又保留关键信息。

我们需要做的：
- 选择合适的记忆策略
- 实现多会话隔离（不同用户的对话不能串）
- 处理 Token 上限的边界情况

## 具体实现

### 第一步：基础记忆配置

```java
@Configuration
public class MemoryConfig {

    /**
     * 消息窗口记忆：保留最近 20 条消息
     * 适合大多数客服场景
     */
    @Bean
    public ChatMemory chatMemory() {
        return MessageWindowChatMemory.builder()
                .maxMessages(20)
                .build();
    }

    /**
     * 带记忆的 ChatClient
     */
    @Bean
    public ChatClient chatClient(ChatModel chatModel, ChatMemory chatMemory) {
        return ChatClient.builder(chatModel)
                .defaultSystem("你是客服助手，请根据上下文回答用户问题。")
                .defaultAdvisors(new MessageChatMemoryAdvisor(chatMemory))
                .build();
    }
}
```

### 第二步：会话隔离与对话接口

```java
@RestController
@RequestMapping("/api/chat")
@RequiredArgsConstructor
@Slf4j
public class MemoryChatController {

    private final ChatClient chatClient;
    private final ChatMemory chatMemory;

    /**
     * 带记忆的对话接口
     * conversationId 用于隔离不同会话
     */
    @PostMapping("/memory")
    public Map<String, Object> chatWithMemory(@RequestBody MemoryChatRequest request) {
        String conversationId = request.conversationId();
        if (!StringUtils.hasText(conversationId)) {
            conversationId = UUID.randomUUID().toString();
        }

        log.info("【记忆对话】会话: {}, 输入: {}", conversationId, request.message());

        String answer = chatClient.prompt()
                .user(request.message())
                .advisors(a -> a.param(ChatMemory.CONVERSATION_ID, conversationId))
                .call()
                .content();

        // 查询当前记忆的消息数量，用于监控
        List<Message> history = chatMemory.get(conversationId, Integer.MAX_VALUE);
        log.info("【记忆对话】会话: {}, 历史消息数: {}", conversationId, history.size());

        return Map.of(
                "answer", answer,
                "conversationId", conversationId,
                "historySize", history.size()
        );
    }

    /**
     * 清除指定会话的记忆
     */
    @DeleteMapping("/memory/{conversationId}")
    public Map<String, Object> clearMemory(@PathVariable String conversationId) {
        chatMemory.clear(conversationId);
        log.info("【记忆清除】会话: {}", conversationId);
        return Map.of("success", true, "conversationId", conversationId);
    }

    public record MemoryChatRequest(String message, String conversationId) {}
}
```

### 第三步：Token 窗口记忆（更精确的控制）

```java
@Configuration
public class TokenMemoryConfig {

    /**
     * Token 窗口记忆：控制总 Token 数不超过指定值
     * 比消息数量控制更精确，但需要 Token 计算
     */
    @Bean("tokenMemory")
    public ChatMemory tokenMemory() {
        return MessageWindowChatMemory.builder()
                .maxMessages(50) // 作为上限兜底
                .build();
    }

    /**
     * 自定义 Advisor：手动实现 Token 级别的记忆控制
     */
    @Bean
    public ChatClient tokenAwareChatClient(ChatModel chatModel, ChatMemory tokenMemory) {
        return ChatClient.builder(chatModel)
                .defaultSystem("你是客服助手")
                .defaultAdvisors(new TokenLimitMemoryAdvisor(tokenMemory, 3000))
                .build();
    }
}
```

自定义 Token 限制 Advisor：

```java
/**
 * 自定义 Advisor：在 Token 预算内管理记忆
 * 当历史消息的 Token 总数超过预算时，从最早的消息开始删除
 */
public class TokenLimitMemoryAdvisor implements CallAroundAdvisor {

    private final ChatMemory chatMemory;
    private final int maxInputTokens;

    public TokenLimitMemoryAdvisor(ChatMemory chatMemory, int maxInputTokens) {
        this.chatMemory = chatMemory;
        this.maxInputTokens = maxInputTokens;
    }

    @Override
    public AdvisedResponse aroundCall(AdvisedRequest request, CallAroundAdvisorChain chain) {
        String conversationId = request.adviseContext()
                .getOrDefault(ChatMemory.CONVERSATION_ID, "default");

        // 获取历史消息
        List<Message> history = chatMemory.get(conversationId, 100);

        // 粗估 Token 数（中文约 1.5 token/字，英文约 0.75 token/word）
        int totalTokens = estimateTokens(history);
        log.debug("【Token管理】会话: {}, 历史Token估计: {}, 预算: {}",
                conversationId, totalTokens, maxInputTokens);

        // 超出预算时，从最早的消息开始移除
        while (totalTokens > maxInputTokens && !history.isEmpty()) {
            Message removed = history.remove(0);
            totalTokens -= estimateTokens(List.of(removed));
            log.debug("【Token管理】移除消息，剩余Token: {}", totalTokens);
        }

        // 将修剪后的消息写回记忆
        chatMemory.clear(conversationId);
        chatMemory.add(conversationId, history);

        // 继续执行链路
        return chain.nextAroundCall(request);
    }

    @Override
    public String getName() {
        return "TokenLimitMemoryAdvisor";
    }

    @Override
    public int getOrder() {
        return Ordered.HIGHEST_PRECEDENCE + 1;
    }

    /**
     * 粗估消息的 Token 数
     * 生产环境建议用 tiktoken 库做精确计算
     */
    private int estimateTokens(List<Message> messages) {
        int totalChars = messages.stream()
                .mapToInt(m -> m.getText() != null ? m.getText().length() : 0)
                .sum();
        return (int) (totalChars * 1.5); // 中文大致 1.5 token/字符
    }
}
```

### 第四步：摘要记忆（长对话场景）

```java
@Service
@Slf4j
@RequiredArgsConstructor
public class SummaryMemoryService {

    private final ChatModel chatModel;
    private final ChatMemory chatMemory;

    /**
     * 当消息数超过阈值时，将早期对话压缩为摘要
     * 保留最近 N 条原始消息 + 之前的摘要
     */
    public void compactIfNeeded(String conversationId, int threshold) {
        List<Message> history = chatMemory.get(conversationId, Integer.MAX_VALUE);

        if (history.size() <= threshold) {
            return;
        }

        // 把前半部分消息压缩成摘要
        int splitIndex = history.size() - 10; // 保留最近 10 条
        List<Message> toSummarize = history.subList(0, splitIndex);
        List<Message> toKeep = history.subList(splitIndex, history.size());

        // 用模型生成摘要
        String conversationText = toSummarize.stream()
                .map(m -> String.format("[%s] %s",
                        m.getMessageType() == MessageType.USER ? "用户" : "助手",
                        m.getText()))
                .collect(Collectors.joining("\n"));

        String summary = chatModel.call(new Prompt(List.of(
                new SystemMessage("请将以下对话压缩为简洁的摘要，保留关键信息和结论，不超过200字："),
                new UserMessage(conversationText)
        ))).getResult().getOutput().getContent();

        log.info("【摘要压缩】会话: {}, 原始消息: {} 条, 摘要长度: {}",
                conversationId, toSummarize.size(), summary.length());

        // 重建记忆：摘要 + 最近消息
        chatMemory.clear(conversationId);
        chatMemory.add(conversationId, List.of(
                new SystemMessage("之前的对话摘要：" + summary)
        ));
        chatMemory.add(conversationId, toKeep);
    }
}
```

### 第五步：记忆持久化（数据库存储）

```java
/**
 * 基于数据库的 ChatMemory 实现
 * 会话状态不丢失，重启后还能继续对话
 */
@Component
@Slf4j
@RequiredArgsConstructor
public class JdbcChatMemory implements ChatMemory {

    private final JdbcTemplate jdbcTemplate;

    @Override
    public void add(String conversationId, List<Message> messages) {
        for (Message message : messages) {
            jdbcTemplate.update(
                    "INSERT INTO chat_memory (conversation_id, role, content, created_at) VALUES (?, ?, ?, ?)",
                    conversationId,
                    message.getMessageType().name(),
                    message.getText(),
                    Instant.now()
            );
        }
    }

    @Override
    public List<Message> get(String conversationId, int lastN) {
        return jdbcTemplate.query(
                "SELECT role, content FROM chat_memory WHERE conversation_id = ? ORDER BY id DESC LIMIT ?",
                (rs, rowNum) -> {
                    String role = rs.getString("role");
                    String content = rs.getString("content");
                    if ("USER".equals(role)) {
                        return new UserMessage(content);
                    } else if ("ASSISTANT".equals(role)) {
                        return new AssistantMessage(content);
                    } else {
                        return new SystemMessage(content);
                    }
                },
                conversationId, lastN
        ).reversed(); // 按时间正序
    }

    @Override
    public void clear(String conversationId) {
        jdbcTemplate.update("DELETE FROM chat_memory WHERE conversation_id = ?", conversationId);
    }
}
```

```sql
-- 建表语句
CREATE TABLE chat_memory (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    conversation_id VARCHAR(64) NOT NULL,
    role VARCHAR(20) NOT NULL,
    content TEXT NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_conversation (conversation_id)
);
```

## 踩坑总结

1. **Token 估算不准**：中文的 Token 数跟英文差很多。我们用 "字符数 * 1.5" 做粗估，实际偏差在 20% 左右。如果对精度要求高，建议引入 tiktoken 库做精确计算，但会增加一点性能开销。

2. **摘要压缩本身消耗 Token**：每次压缩都要调一次模型，这部分 Token 也要算在成本里。建议只在消息数超过阈值（比如 30 条）时才触发压缩。

3. **并发写入问题**：如果同一个会话有多个并发请求同时写入记忆，可能出现消息顺序混乱。解决方案是在写入时加分布式锁（用 Redis），或者用数据库的乐观锁。

4. **记忆清理策略**：不能只靠 maxMessages 做限制，还要加一个时间维度的清理。比如超过 24 小时没活动的会话自动清除记忆。否则长期不活跃的会话占着内存。

5. **System Message 不要放到记忆里**：系统提示词应该每次都从配置读取，不跟历史消息混在一起。否则压缩摘要时可能把系统提示词也压缩掉了，导致模型角色设定丢失。

6. **测试对话记忆要模拟真实场景**：开发时容易只测 3-5 轮对话，看起来没问题。但真实用户可能连续聊 50 轮以上，这时候才发现 Token 超限或者性能问题。建议测试时至少模拟 20 轮以上的长对话。
