### 问题场景

作为Java后端开发，你不可避免地要和Linux服务器打交道。特别是中小厂，没有专职运维，服务器出了问题开发得自己上。我在接手运维工作的头半年里，踩了无数坑，这篇文章把最常用的技能整理出来，都是真实场景下用到的。

### 日志分析三剑客：grep/awk/sed

**场景1：查线上报错日志**

服务出了问题，第一步就是查日志。日志文件在 `/data/logs/app/` 下面，按天滚动。

```bash
# 查看今天的ERROR日志
grep "ERROR" /data/logs/app/app.log

# 查看指定时间段的日志（假设日志格式为 2026-05-25 14:30:00）
grep "2026-05-25 14:[0-3]" /data/logs/app/app.log | grep "ERROR"

# 显示错误上下文（前后各5行）
grep -A 5 -B 5 "NullPointerException" /data/logs/app/app.log

# 统计每小时的错误数量
grep "ERROR" /data/logs/app/app.log | awk '{print substr($2,1,2)":00"}' | sort | uniq -c | sort -rn
```

**场景2：统计接口响应时间**

假设日志里有接口耗时信息，格式类似 `[user/query] 耗时:235ms`：

```bash
# 找出耗时超过1秒的慢接口
grep "耗时" /data/logs/app/app.log | awk -F'耗时:' '{print $2}' | awk -F'ms' '{if($1>1000) print}' | wc -l

# 按接口统计平均耗时
grep "耗时" /data/logs/app/app.log | awk -F'[][]' '{print $2}' | sort | uniq -c | sort -rn | head -20

# 找出最慢的10个请求
grep "耗时" /data/logs/app/app.log | sed 's/.*\[\(.*\)\].*耗时:\([0-9]*\)ms/\2 \1/' | sort -rn | head -10
```

**场景3：批量替换配置**

```bash
# 把所有properties文件中的debug改为info
sed -i 's/log.level=debug/log.level=info/g' /data/config/*.properties

# 预览修改（不加-i则只输出不写入）
sed 's/old_value/new_value/g' config.yml

# 只替换第一个匹配
sed -i '0,/old_value/s//new_value/' config.yml
```

### 进程管理

**查看Java进程**

```bash
# 查找所有Java进程
ps -ef | grep java | grep -v grep

# 查看进程的详细信息（启动时间、CPU、内存）
ps -p <PID> -o pid,lstart,%cpu,%mem,rss,cmd

# 查看进程的线程数
ps -T -p <PID> | wc -l

# 查看进程打开的文件数（排查"too many open files"）
ls -la /proc/<PID>/fd | wc -l
```

**端口占用排查**

```bash
# 查看哪个进程占用了8080端口
lsof -i :8080
# 或
ss -tlnp | grep 8080

# 强制杀掉占用端口的进程（慎用）
kill -9 $(lsof -t -i :8080)
```

**systemd管理服务**

```bash
# 查看服务状态
systemctl status our-app.service

# 重启服务
systemctl restart our-app.service

# 查看服务日志
journalctl -u our-app.service -f  # -f 实时跟踪
journalctl -u our-app.service --since "1 hour ago"
```

一个实际的systemd服务配置文件：

```ini
# /etc/systemd/system/our-app.service
[Unit]
Description=Our Java Application
After=network.target

[Service]
Type=simple
User=appuser
WorkingDirectory=/data/app
ExecStart=/usr/bin/java -Xms512m -Xmx512m -jar app.jar --spring.profiles.active=prod
ExecStop=/bin/kill -SIGTERM $MAINPID
Restart=on-failure
RestartSec=10
LimitNOFILE=65536

[Install]
WantedBy=multi-user.target
```

### 磁盘和CPU排查

**磁盘空间不足**

```bash
# 查看磁盘使用情况
df -h

# 找到占用空间最大的目录（从根目录开始）
du -sh /* | sort -rh | head -10

# 找到大于100MB的日志文件
find /data/logs -name "*.log" -size +100M -exec ls -lh {} \;

# 清理30天前的日志
find /data/logs -name "*.log.*" -mtime +30 -delete

# Docker磁盘清理（经常被遗忘的大头）
docker system df         # 查看Docker占用
docker system prune -a   # 清理未使用的镜像和容器（谨慎）
docker image prune -a --filter "until=720h"  # 清理30天前的镜像
```

**CPU占用排查**

```bash
# 查看整体CPU使用情况
top -bn1 | head -20

# 找出CPU占用最高的进程
ps aux --sort=-%cpu | head -10

# 对于Java进程，进一步看是哪个线程占CPU
# 1. 找到Java进程PID
jps -l

# 2. 找到CPU高的线程
top -Hp <PID>

# 3. 把线程ID转为16进制（假设线程ID是12345）
printf "0x%x\n" 12345

# 4. 在线程dump中查找
jstack <PID> | grep -A 20 "0x3039"
```

这是我们排查CPU飙高的标准流程，每次都用得上。

**内存排查**

```bash
# 查看系统内存
free -h

# 查看Java进程的GC情况
jstat -gcutil <PID> 1000 10  # 每秒打印一次，共10次

# 导出heap dump（线上慎用，会暂停应用）
jmap -dump:live,format=b,file=heap.hprof <PID>

# 查看系统内存使用大户
ps aux --sort=-%mem | head -10
```

### Crontab 定时任务

**场景：定时清理日志、定时备份数据库**

```bash
# 编辑当前用户的crontab
crontab -e
```

```bash
# 每天凌晨2点清理30天前的日志
0 2 * * * find /data/logs -name "*.log.*" -mtime +30 -delete >> /var/log/cron-clean.log 2>&1

# 每天凌晨3点备份数据库
0 3 * * * /data/scripts/db-backup.sh >> /var/log/db-backup.log 2>&1

# 每5分钟检查服务是否存活
*/5 * * * * /data/scripts/health-check.sh >> /var/log/health-check.log 2>&1

# 每周一早上9点发送周报
0 9 * * 1 /data/scripts/send-weekly-report.sh >> /var/log/weekly-report.log 2>&1
```

**踩坑点**：

```bash
# 查看crontab是否生效
crontab -l

# 查看cron执行日志（CentOS/RedHat）
tail -f /var/log/cron

# 查看cron执行日志（Ubuntu）
tail -f /var/log/syslog | grep CRON
```

**常见的crontab不生效原因**：
1. 环境变量问题：crontab的环境变量和登录Shell不一样，脚本里要用绝对路径
2. 权限问题：脚本没有执行权限，需要 `chmod +x`
3. 输出问题：crontab的标准输出和错误输出不会显示在终端，必须重定向到日志文件
4. 换行问题：crontab最后一行必须有换行符，否则最后一条任务不会执行

```bash
# 正确写法：用绝对路径，设置PATH
PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin
SHELL=/bin/bash

0 2 * * * /data/scripts/clean-logs.sh >> /var/log/clean.log 2>&1
```

### 踩坑总结

1. **生产环境不要用 `rm -rf *`**。设个alias：`alias rm='rm -i'`，虽然麻烦一点但能救命
2. **查日志之前先看文件大小**。`ls -lh app.log`，如果好几个GB，别直接grep，先 `tail -n 100000` 截取最近的
3. **top命令按1可以查看每个CPU核心的使用率**。如果某个核心100%其他空闲，大概率是GC线程绑核的问题
4. **crontab里的脚本一定要加日志输出**。没有日志的定时任务就像闭着眼睛开车
5. **`/var/log/messages` 或 `dmesg` 可以看到OOM Killer的日志**。Java进程突然消失了，先查这里

Linux运维技能不需要多深，但这些基础命令必须熟练。建议把常用的排查命令整理成一个脚本或文档，出问题时直接翻，比现场Google高效得多。
