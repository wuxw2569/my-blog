# Fine-tuning 微调 vs Prompt 工程，怎么选？

## 前言

这是一个被问了无数遍的问题：我想让 AI 模型更好地完成我的任务，是应该写更好的 Prompt，还是应该做微调（Fine-tuning）？

**简短的回答：90% 的情况下，先把 Prompt 工程做到极致再说。** 微调是大招，不到万不得已不要用。

这篇文章解释为什么。

## 什么是 Prompt 工程？

Prompt 工程就是通过设计更好的输入提示（Prompt），让模型更好地完成任务。

它不改变模型本身，只是改变你和模型"沟通"的方式。

**Prompt 工程的常见技巧：**

```
1. 明确角色设定
   "你是一个资深的 Java 后端工程师，擅长性能优化..."

2. 给出示例（Few-shot）
   "输入: 用户投诉快递太慢
    输出: {"category": "物流", "sentiment": "负面", "urgency": "中"}"

3. 思维链（Chain-of-Thought）
   "请一步一步思考：
    1. 首先分析问题的关键信息
    2. 然后列出可能的解决方案
    3. 最后给出推荐方案"

4. 格式约束
   "请严格按照以下 JSON 格式输出，不要包含其他内容：
    {"name": "...", "age": ..., "city": "..."}"

5. 分步指令
   "第一步：阅读以下文档
    第二步：提取关键信息
    第三步：生成摘要"
```

Prompt 工程的成本为零（不需要训练），见效快（改了 prompt 立刻就能看到效果），是性价比最高的优化手段。

## 什么是 Fine-tuning（微调）？

微调是**用你自己的数据对模型进行额外训练**，让模型"学会"你想要的行为模式。

打个比方：

- Prompt 工程 = 在考试前给学生一份"答题指南"
- 微调 = 让学生做大量的模拟题，让他形成"肌肉记忆"

微调会**改变模型的权重参数**，所以它确实能让模型在特定任务上有更好的表现。但代价也很高。

## 什么时候 Prompt 工程就够了？

以下场景，**只需要做好 Prompt 工程**，不需要微调：

| 场景 | 说明 |
|------|------|
| 通用问答 | 基础的客服、知识问答 |
| 内容生成 | 文章、邮件、广告文案 |
| 数据提取 | 从文本中提取结构化信息 |
| 翻译/改写 | 语言转换、风格调整 |
| 代码生成 | 写函数、debug、代码审查 |
| 分类任务 | 情感分析、标签分类（数据量不大时） |

**判断标准：** 如果你在 prompt 中给 3-5 个示例，模型就能达到 80% 以上的效果，那就不需要微调。

一个实际的 Prompt 优化过程：

```python
# 第一版 prompt（效果差）
prompt = "分析这段评论的情感"

# 第二版（加上角色和格式要求）
prompt = """你是一个情感分析专家。请分析以下评论的情感倾向。
输出格式：{"sentiment": "正面/负面/中性", "confidence": 0.0-1.0}
评论：{text}"""

# 第三版（加上 few-shot 示例）
prompt = """你是一个情感分析专家。请分析以下评论的情感倾向。
输出格式：{"sentiment": "正面/负面/中性", "confidence": 0.0-1.0}

示例：
评论：这个产品太好用了，强烈推荐！
输出：{"sentiment": "正面", "confidence": 0.95}

评论：退货流程太复杂了，等了一个星期
输出：{"sentiment": "负面", "confidence": 0.90}

评论：一般般吧，没什么特别的
输出：{"sentiment": "中性", "confidence": 0.85}

现在请分析：
评论：{text}"""
```

仅仅是改进 prompt，效果就可能从 60% 提升到 90% 以上。

## 什么时候才需要微调？

以下场景，**可能需要考虑微调**：

### 1. 输出格式要求极其严格

你的应用需要模型输出非常特定的格式，而且 Prompt 工程怎么调都无法保证格式一致性。比如复杂的嵌套 JSON、特定的 DSL（领域特定语言）。

### 2. 特定领域风格

你需要模型的语言风格、用词习惯完全符合特定领域。比如法律文书、医学报告、古文写作。Prompt 可以引导风格，但微调可以让风格更稳定。

### 3. 需要降低推理成本

通过微调一个小模型（如 Llama 8B），让它在特定任务上达到大模型的水平，从而降低 API 调用成本。

### 4. 需要学习复杂的隐含规则

有些任务的规则很难用语言描述，但有很多标注数据。微调可以从数据中自动学习这些规则。

### 5. 需要极度稳定的输出

在生产环境中，如果输出的一致性要求极高（偏差为零容忍），微调比 Prompt 更可靠。

## 微调的成本与流程

### 微调需要什么？

1. **训练数据**：通常需要几百到几千条"输入-输出"配对数据
2. **计算资源**：GPU 服务器（云上 A100 约 $2-4/小时）
3. **时间**：几小时到几天不等
4. **专业知识**：需要了解训练参数、评估指标等

### 微调的基本流程

```
准备训练数据（JSONL 格式）
  ↓
选择基座模型（如 Llama 3、Qwen 2.5）
  ↓
配置训练参数（学习率、epoch数、batch size等）
  ↓
开始训练（在 GPU 上跑）
  ↓
评估效果（在测试集上评估）
  ↓
效果不好 → 调参重来
效果好 → 部署上线
```

训练数据格式示例（OpenAI 格式）：

```json
{"messages": [
  {"role": "system", "content": "你是一个客服情感分析助手"},
  {"role": "user", "content": "快递一周了还没到，你们怎么办事的？"},
  {"role": "assistant", "content": "{\"sentiment\": \"负面\", \"category\": \"物流\", \"urgency\": \"高\"}"}
]}
{"messages": [
  {"role": "system", "content": "你是一个客服情感分析助手"},
  {"role": "user", "content": "产品收到了，包装很好看"},
  {"role": "assistant", "content": "{\"sentiment\": \"正面\", \"category\": \"产品包装\", \"urgency\": \"低\"}"}
]}
```

### 大致成本估算

| 方式 | 数据需求 | 时间 | 费用 |
|------|---------|------|------|
| OpenAI 微调 API | 100-10000 条 | 数小时 | $5-100 |
| LoRA 微调（自部署） | 500-5000 条 | 2-8 小时 | GPU 费用 |
| 全量微调 | 10000+ 条 | 数天 | 需多卡GPU |

## LoRA：轻量级微调方案

传统的全量微调需要更新模型的所有参数，成本非常高。**LoRA（Low-Rank Adaptation）** 是目前最流行的轻量微调方案。

LoRA 的核心思想：**不动原始模型的参数，只训练一小部分"旁路参数"。**

类比一下：

- 全量微调 = 重新建一栋楼
- LoRA = 在现有大楼旁边加建一个小配楼

LoRA 的优势：

1. **显存需求低**：7B 参数的模型，全量微调需要 ~80GB 显存，LoRA 只需要 ~16GB
2. **训练速度快**：通常几个小时就能完成
3. **可以叠加**：同一个基座模型可以挂多个 LoRA 适配器，切换不同任务
4. **原始模型不变**：LoRA 权重文件通常只有几十 MB

```python
# 使用 Hugging Face PEFT 库做 LoRA 微调的简化示例
from peft import LoraConfig, get_peft_model
from transformers import AutoModelForCausalLM

model = AutoModelForCausalLM.from_pretrained("Qwen/Qwen2.5-7B")

lora_config = LoraConfig(
    r=16,                    # LoRA 的秩（rank），越大越灵活但越慢
    lora_alpha=32,           # 缩放系数
    target_modules=["q_proj", "v_proj"],  # 对哪些层做 LoRA
    lora_dropout=0.05,
    task_type="CAUSAL_LM",
)

model = get_peft_model(model, lora_config)
# 此时可训练参数量约为原始模型的 0.1%-1%
model.print_trainable_parameters()
```

## 决策框架：三步走

面对一个具体需求，按以下顺序决策：

```
第一步：Prompt 工程能搞定吗？
  → 给 3-5 个示例，调优 prompt 结构
  → 效果达标？ → 搞定，不需要微调
  → 效果不够？ → 进入第二步

第二步：RAG 能解决吗？
  → 把相关知识/规则放进检索库
  → 效果达标？ → 搞定，不需要微调
  → 效果不够？ → 进入第三步

第三步：考虑微调
  → 准备训练数据
  → 先用 LoRA 微调小模型试效果
  → 效果达标？ → 部署
  → 效果不够？ → 考虑更大模型或更多数据
```

**记住一个原则：能用 Prompt 解决的不用 RAG，能用 RAG 解决的不用微调。** 每一层的复杂度和成本都是指数级上升的。

## 总结

- Prompt 工程是第一选择，成本低、见效快、迭代方便
- RAG 适合需要引入外部知识的场景
- 微调是"核武器"，威力大但成本高，只在 Prompt + RAG 都搞不定时才考虑
- 如果要做微调，优先考虑 LoRA 等轻量方案
- 始终用业务数据做评估，不要凭感觉判断效果
