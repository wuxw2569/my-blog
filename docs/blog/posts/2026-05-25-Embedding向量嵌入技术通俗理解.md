# Embedding 向量嵌入技术通俗理解

## 前言

如果你了解 RAG（检索增强生成），或者听说过"向量数据库"，那你一定遇到过 "Embedding" 这个词。它是 AI 应用开发中最基础也最核心的技术之一。

这篇文章用最通俗的方式，帮你理解 Embedding 到底在干什么。

## 什么是 Embedding？

一句话：**Embedding 就是把文本变成一组数字（向量），让计算机可以计算文本之间的"相似度"。**

举个生活中的例子：

假设你要在图书馆找一本关于"前端框架 React 最佳实践"的书。如果图书馆只有按字母排序的目录，你得一本一本翻。但如果每本书都被映射到了一个"知识地图"上的某个位置，而且"越相似的书在地图上离得越近"，那你只需要去"React/前端"那片区域找就行了。

**Embedding 就是这张"知识地图"的坐标。**

一段文本经过 Embedding 模型处理后，会变成一个由数字组成的向量。比如：

```
"今天天气真好" → [0.12, -0.34, 0.56, 0.78, ..., -0.91]  (1536个数字)
"今天阳光明媚" → [0.11, -0.33, 0.55, 0.80, ..., -0.89]  (很接近的向量)
"Python并发编程" → [-0.45, 0.67, 0.12, -0.23, ..., 0.34]  (差异很大的向量)
```

注意看：语义相似的文本，向量的数值模式也相似；语义不同的文本，向量差异很大。

## 为什么要变成向量？

你可能会问：用关键词搜索不就行了？为什么要这么麻烦转成向量？

关键词搜索有一个根本问题：**它只认字面，不认意思。**

| 搜索词 | 关键词搜索结果 | 向量搜索结果 |
|--------|-------------|------------|
| "苹果" | 匹配所有含"苹果"的文章 | 可能返回"iPhone"、"水果"相关内容 |
| "如何让程序跑得更快" | 不一定能匹配"性能优化" | 能匹配"性能优化"、"加速方法" |
| "JAVA内存溢出" | 只匹配完全包含这几个字的 | 能匹配"OOM排查"、"堆内存不足" |

**向量搜索是"语义搜索"，它理解意思；关键词搜索是"字面搜索"，它只认字。**

对开发者来说，这意味着：

- 用户可以用自然语言提问，不需要猜关键词
- 搜索结果更"懂"用户意图
- 多语言搜索成为可能（中文问题可以匹配英文内容）

## 向量相似度怎么算？

既然文本变成了向量，那怎么判断两段文本是否相似呢？最常用的方法叫**余弦相似度（Cosine Similarity）**。

想象两条从原点出发的射线：

- 如果两条射线方向一致（夹角为 0°），余弦相似度 = 1，表示完全相同
- 如果两条射线垂直（夹角为 90°），余弦相似度 = 0，表示完全无关
- 如果方向完全相反（夹角为 180°），余弦相似度 = -1，表示完全相反

```python
import numpy as np

def cosine_similarity(a, b):
    """计算两个向量的余弦相似度"""
    a, b = np.array(a), np.array(b)
    return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))

# 模拟 Embedding 向量（实际中维度是 1536 或更高）
vec1 = [0.12, -0.34, 0.56, 0.78]  # "今天天气真好"
vec2 = [0.11, -0.33, 0.55, 0.80]  # "今天阳光明媚"
vec3 = [-0.45, 0.67, 0.12, -0.23] # "Python并发编程"

print(cosine_similarity(vec1, vec2))  # 接近 1.0（非常相似）
print(cosine_similarity(vec1, vec3))  # 接近 0 或负数（不相似）
```

## 主流 Embedding 模型

常用的 Embedding 模型有这些：

| 模型 | 维度 | 特点 |
|------|------|------|
| OpenAI text-embedding-3-small | 1536 | 性价比高，通用性好 |
| OpenAI text-embedding-3-large | 3072 | 精度更高，价格更贵 |
| BGE-M3 | 1024 | 开源，多语言支持好，中文表现优秀 |
| GTE（阿里） | 1024 | 开源，中文优化好 |
| Cohere embed-v3 | 1024 | 多语言支持好 |

**开发者选型建议：**

- 需要快速上手、不在乎成本 → OpenAI text-embedding-3-small
- 中文为主、需要私有化 → BGE-M3 或 GTE
- 预算敏感 → 开源模型 + 自部署

## Embedding 在 RAG 中的应用

Embedding 最典型的应用场景就是 **RAG（检索增强生成）**。RAG 的基本流程中，Embedding 扮演了核心角色：

```
用户提问
  ↓
[Embedding] 把问题变成向量
  ↓
[向量检索] 在知识库中找最相似的文档片段
  ↓
把检索到的文档 + 用户问题 一起发给 LLM
  ↓
LLM 基于检索到的内容生成回答
```

**没有 Embedding，就没有高效的语义检索；没有检索，RAG 就无法工作。**

实际项目中，你需要：

1. 把你的知识库文档切分成小块（chunk）
2. 每一块通过 Embedding 模型转成向量
3. 把这些向量存到向量数据库（如 Milvus、Pinecone、Chroma 等）
4. 用户提问时，把问题转成向量，在数据库中找最近的几个向量
5. 把对应的原始文本拼到 prompt 中给 LLM 参考

## 向量数据库简介

既然有了大量向量，就需要一个高效的方式来存储和检索它们。这就是向量数据库的作用。

常用的向量数据库：

| 数据库 | 类型 | 特点 |
|--------|------|------|
| Milvus | 专用向量数据库 | 性能好、可扩展、适合大规模 |
| Chroma | 轻量级 | 简单易用、适合原型开发 |
| Pinecone | 云服务 | 免运维、按使用量付费 |
| Weaviate | 专用 | 支持混合搜索（向量+关键词）|
| pgvector | PostgreSQL 扩展 | 如果你已有 PG，加个扩展就行 |

```python
# 使用 Chroma 的简单示例
import chromadb

client = chromadb.Client()
collection = client.create_collection("my_docs")

# 添加文档（Chroma 会自动做 Embedding）
collection.add(
    documents=["React 是一个前端框架", "Vue 也是前端框架", "Python 是后端语言"],
    ids=["doc1", "doc2", "doc3"]
)

# 查询
results = collection.query(query_texts=["前端开发用什么框架"], n_results=2)
print(results["documents"])
# 输出: [["React 是一个前端框架", "Vue 也是前端框架"]]
```

## 实践中的注意事项

1. **Embedding 的质量直接决定搜索质量**。如果 Embedding 模型选得不好，后续的检索、生成都不会好。建议用你自己的数据测试不同模型。

2. **文本切分策略很重要**。切得太细会丢失上下文，切得太粗检索不精准。一般建议 300-500 个 token 一块，有适当重叠。

3. **Embedding 的成本**。Embedding 模型虽然比 LLM 便宜很多，但如果你的知识库有几百万条文档，一次性 Embedding 的成本也不容忽视。

4. **向量维度不是越大越好**。维度越大，存储和检索成本越高。在精度和性能之间找平衡。

5. **中文场景注意**。部分英文为主的 Embedding 模型在中文上表现一般，建议选择多语言模型或中文优化模型。

## 总结

Embedding 是把文本变成向量的技术，让计算机可以计算文本之间的语义相似度。它是 RAG、语义搜索等应用的基础。作为开发者，你需要理解：

- Embedding 的本质是将文本映射到向量空间
- 语义相似的文本，向量距离近
- 选对 Embedding 模型和文本切分策略，是 RAG 成功的关键
- 向量数据库是存储和检索向量的基础设施
