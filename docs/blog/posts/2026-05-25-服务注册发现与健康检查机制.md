## 服务注册发现不是配个注解就完了

很多人觉得服务注册发现就是加个 `@EnableDiscoveryClient` 的事。确实，让它跑起来很简单，但在生产环境跑稳定，踩的坑远比想象的多。

我负责的一个系统有 15 个微服务，跑在 K8s 上，用 Nacos 做服务注册发现。以下是我们从"能用"到"好用"的过程。

## Nacos vs Eureka：注册原理对比

### Eureka 的机制

Eureka 用的是 **客户端心跳** 模式：

- 服务启动时向 Eureka Server 注册
- 每 30 秒发送一次心跳
- Eureka Server 如果 90 秒没收到心跳，就把实例标记为下线
- 客户端每 30 秒拉取一次注册表（增量拉取）

**问题**：Eureka 2.x 已经不维护了，而且它的"最终一致性"模型有缺陷——服务下线后最多有 90 秒的延迟，期间流量还是会打到已下线的实例。

### Nacos 的机制

Nacos 支持两种健康检查模式：

**临时实例（Ephemeral，默认）**：类似 Eureka，客户端心跳保活，但超时时间更短（默认 15 秒）。Nacos 2.x 还增加了 gRPC 长连接，心跳更加可靠。

**永久实例（Persistent）**：服务端主动探测客户端，适合数据库、中间件这类不会主动发送心跳的服务。

```yaml
# 推荐配置：临时实例 + 较短的超时时间
spring:
  cloud:
    nacos:
      discovery:
        server-addr: ${NACOS_ADDR:127.0.0.1:8848}
        # 关键配置
        heart-beat-interval: 5000      # 心跳间隔 5 秒（默认 5 秒）
        heart-beat-timeout: 15000      # 心跳超时 15 秒（默认 15 秒）
        # 元数据，用于灰度、路由等
        metadata:
          version: 1.0.0
          env: ${SPRING_PROFILES_ACTIVE:dev}
```

## 健康检查配置

光靠 Nacos 的心跳检查是不够的——实例进程还在，但业务可能已经不健康了（比如数据库连接池满了、内存溢出）。需要配合应用层面的健康检查。

### Spring Boot Actuator

```yaml
management:
  endpoints:
    web:
      exposure:
        include: health,info,prometheus
  endpoint:
    health:
      # 显示详细的健康信息
      show-details: always
  health:
    db:
      enabled: true
    redis:
      enabled: true
    diskspace:
      enabled: true
```

自定义健康检查——检查业务层面的状态：

```java
@Component
@RequiredArgsConstructor
public class BusinessHealthIndicator implements HealthIndicator {

    private final DataSource dataSource;
    private final RedisTemplate<String, String> redisTemplate;

    @Override
    public Health health() {
        // 检查数据库连接
        try (Connection conn = dataSource.getConnection()) {
            conn.isValid(1);
        } catch (Exception e) {
            return Health.down()
                .withDetail("database", "连接失败: " + e.getMessage())
                .build();
        }

        // 检查 Redis 连接
        try {
            redisTemplate.opsForValue().get("health:check");
        } catch (Exception e) {
            return Health.down()
                .withDetail("redis", "连接失败: " + e.getMessage())
                .build();
        }

        return Health.up().build();
    }
}
```

### 和 Nacos 健康检查联动

当 Actuator 健康检查失败时，需要通知 Nacos 把实例标记为不健康：

```java
/**
 * 监听健康状态变化，主动向 Nacos 注销/注册实例
 * 坑：Nacos 的健康状态和 Actuator 的健康状态是独立的，需要手动同步
 */
@Component
@Slf4j
@RequiredArgsConstructor
public class HealthStatusListener {

    private final NacosDiscoveryProperties discoveryProperties;
    private final NamingService namingService;
    private final AtomicBoolean registered = new AtomicBoolean(true);

    @EventListener
    public void onHealthChange(HealthStatusChangedEvent event) {
        // Spring Boot 2.6+ 支持此事件
        Instance instance = buildInstance();

        if (event.getStatus() == Status.DOWN && registered.compareAndSet(true, false)) {
            try {
                // 健康检查失败，主动注销实例
                namingService.deregisterInstance(
                    discoveryProperties.getService(),
                    discoveryProperties.getGroup(),
                    instance);
                log.warn("【健康检查】实例已标记为不健康，主动注销");
            } catch (NacosException e) {
                log.error("【健康检查】注销实例失败", e);
            }
        } else if (event.getStatus() == Status.UP && registered.compareAndSet(false, true)) {
            try {
                // 恢复健康，重新注册
                namingService.registerInstance(
                    discoveryProperties.getService(),
                    discoveryProperties.getGroup(),
                    instance);
                log.info("【健康检查】实例恢复健康，重新注册");
            } catch (NacosException e) {
                log.error("【健康检查】重新注册失败", e);
            }
        }
    }

    private Instance buildInstance() {
        Instance instance = new Instance();
        instance.setIp(discoveryProperties.getIp());
        instance.setPort(discoveryProperties.getPort());
        return instance;
    }
}
```

## 实例下线的优雅处理

服务重启或发版时，如果直接 kill 进程，注册中心还没来得及摘除实例，流量还会打过来，导致部分请求失败。这是我们踩过最痛的坑。

### 方案一：Spring Boot Actuator 优雅关闭

```yaml
server:
  shutdown: graceful  # 开启优雅关闭

spring:
  lifecycle:
    timeout-per-shutdown-phase: 30s  # 最多等 30 秒
```

### 方案二：主动注销 + 延迟关闭

```java
/**
 * 服务关闭前的清理工作
 * 捕获 JVM shutdown hook，在进程退出前主动从 Nacos 注销
 */
@Component
@Slf4j
public class GracefulShutdownHook implements DisposableBean {

    private final NamingService namingService;
    private final NacosDiscoveryProperties properties;

    public GracefulShutdownHook(NamingService namingService,
                                NacosDiscoveryProperties properties) {
        this.namingService = namingService;
        this.properties = properties;
    }

    @Override
    public void destroy() throws Exception {
        log.info("【优雅关闭】开始从 Nacos 注销实例");
        namingService.deregisterInstance(
            properties.getService(),
            properties.getGroup(),
            properties.getIp(),
            properties.getPort());

        // 关键：等待一段时间，让消费者拉取到最新的注册表
        // Nacos 客户端默认每 6 秒拉取一次，等 10 秒足够覆盖
        Thread.sleep(10000);
        log.info("【优雅关闭】实例已注销，等待请求处理完成");
    }
}
```

### 方案三：K8s 环境下的 preStop Hook

```yaml
# deployment.yaml
spec:
  template:
    spec:
      containers:
        - name: order-service
          lifecycle:
            preStop:
              exec:
                # 先标记为不健康，等待负载均衡器摘除，再等待存量请求完成
                command: ["sh", "-c", "sleep 10"]
      terminationGracePeriodSeconds: 60  # 给足够的时间做优雅关闭
```

**K8s 下的完整流程**：

1. K8s 发送 SIGTERM 给 Pod
2. preStop hook 执行（sleep 10 秒），期间 Pod 已从 Endpoints 中移除
3. Spring Boot 收到 SIGTERM，触发优雅关闭：停止接收新请求，等待存量请求完成
4. 30 秒超时后强制退出

**踩坑**：`terminationGracePeriodSeconds` 必须大于 `preStop` 时间 + 优雅关闭超时时间。我们设的 `preStop=10s` + `graceful=30s` = 40s，所以 `terminationGracePeriodSeconds` 设了 60s。

### 方案四：Feign/Ribbon 的重试配合

即使做了优雅关闭，仍可能有请求打到正在关闭的实例。需要配合客户端重试：

```yaml
# Feign 配置重试
spring:
  cloud:
    openfeign:
      client:
        config:
          default:
            connectTimeout: 2000
            readTimeout: 5000

# 配置 Ribbon 重试（Spring Cloud LoadBalancer 场景）
spring:
  cloud:
    loadbalancer:
      retry:
        enabled: true
        max-retries-on-same-service-instance: 0  # 同实例不重试
        max-retries-on-next-service-instance: 1  # 换实例重试 1 次
```

## 常见问题排查

1. **服务注册了但其他服务发现不了**：检查 namespace 和 group 是否一致。我们出过一次问题：dev 环境的服务注册到了 prod 的 namespace 里，原因是 K8s ConfigMap 配置错误。

2. **实例频繁上下线**：通常是网络抖动或 GC 导致心跳超时。解决方案是适当增大心跳超时时间，并排查 GC 问题。

3. **Nacos 2.x 的 gRPC 端口**：Nacos 2.x 除了 HTTP 端口（8848），还有 gRPC 端口（9848/9849），必须在防火墙/K8s NetworkPolicy 中放行，否则心跳和配置推送会失败。

4. **服务列表缓存**：Nacos 客户端有本地缓存，即使 Nacos Server 不可用，已注册的服务仍然可以互相发现。但这也意味着新增实例可能有延迟。
