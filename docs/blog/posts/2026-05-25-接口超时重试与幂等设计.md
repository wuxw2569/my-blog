## 前言

某天下午，我们的订单服务突然 CPU 飙到 100%。排查发现，下游支付服务出现了一次约 30 秒的网络抖动。就这 30 秒，引发了雪崩：

1. 所有支付请求超时
2. 上游（订单服务）触发重试，每个请求重试 3 次
3. 上游的上游（网关层）也在重试，每个请求又重试 3 次
4. 30 秒后下游恢复，突然涌入原来 9 倍的请求量（3 x 3）
5. 下游被压垮，整个链路不可用

这就是典型的**重试风暴**。这篇文章从这次事故出发，讲讲超时、重试和幂等的最佳实践。

## 问题场景

先看事故时的代码：

```java
// 错误示范：无脑重试
@Retryable(value = Exception.class, maxAttempts = 3)
public PayResult pay(PayRequest request) {
    return restTemplate.postForObject(
            "http://pay-service/api/pay",
            request, PayResult.class);
}
```

问题有三个：重试次数太多、没有区分异常类型、没有幂等保护。

## 分析思路

重试的设计需要考虑四个维度：

| 维度 | 考虑因素 |
|------|----------|
| 什么时候重试 | 只重试"可重试"的异常 |
| 重试几次 | 考虑下游承受能力 |
| 间隔多久重试 | 指数退避避免瞬间冲击 |
| 重试的幂等性 | 同一个请求多次执行结果一致 |

## 具体实现

### 一、超时时间的选择

超时时间设得太短会误判，设得太长会拖慢响应。选择依据是**目标接口的 P99 延迟**。

```java
// 推荐公式：超时时间 = P99 延迟 * 2 ~ 3
// 如果支付接口 P99 = 500ms，超时设 1500ms
// 如果报表接口 P99 = 5s，超时设 15s
```

分层超时设计：

```yaml
# application.yml
rest:
  client:
    # 支付服务：快进快出
    pay-service:
      connect-timeout: 1000    # 连接超时 1 秒
      read-timeout: 3000       # 读取超时 3 秒
    # 报表服务：容忍慢
    report-service:
      connect-timeout: 1000
      read-timeout: 30000      # 读取超时 30 秒
    # 用户服务：内网调用，应该很快
    user-service:
      connect-timeout: 500
      read-timeout: 2000
```

```java
@Configuration
public class RestTemplateConfig {

    @Bean("payRestTemplate")
    public RestTemplate payRestTemplate() {
        return createRestTemplate(1000, 3000);
    }

    @Bean("reportRestTemplate")
    public RestTemplate reportRestTemplate() {
        return createRestTemplate(1000, 30000);
    }

    private RestTemplate createRestTemplate(int connectTimeout, int readTimeout) {
        HttpComponentsClientHttpRequestFactory factory =
                new HttpComponentsClientHttpRequestFactory();
        factory.setConnectTimeout(connectTimeout);
        factory.setReadTimeout(readTimeout);
        return new RestTemplate(factory);
    }
}
```

### 二、重试策略：指数退避

Spring Retry 支持指数退避，但默认参数不太好用，需要手动配置：

```java
@Configuration
@EnableRetry
public class RetryConfig {

    @Bean
    public RetryTemplate retryTemplate() {
        RetryTemplate template = new RetryTemplate();

        // 1. 重试策略：最多重试 2 次（总共 3 次调用）
        SimpleRetryPolicy retryPolicy = new SimpleRetryPolicy();
        retryPolicy.setMaxAttempts(3);

        // 只重试这些异常，其他异常直接抛出
        Map<Class<? extends Throwable>, Boolean> retryableExceptions = new HashMap<>();
        retryableExceptions.put(ResourceAccessException.class, true);   // 网络异常
        retryableExceptions.put(HttpServerErrorException.class, true);  // 5xx
        retryableExceptions.put(ConnectTimeoutException.class, true);   // 连接超时
        retryableExceptions.put(SocketTimeoutException.class, true);    // 读取超时
        // 不重试 4xx（客户端错误，重试没意义）
        retryableExceptions.put(HttpClientErrorException.class, false);
        // 不重试业务异常
        retryableExceptions.put(BizException.class, false);

        ExceptionClassifierRetryPolicy classifier = new ExceptionClassifierRetryPolicy();
        classifier.setPolicyMap(retryableExceptions);
        template.setRetryPolicy(classifier);

        // 2. 退避策略：指数退避 + 随机抖动
        ExponentialBackOffPolicy backOff = new ExponentialBackOffPolicy();
        backOff.setInitialInterval(200);    // 首次等待 200ms
        backOff.setMultiplier(2.0);         // 每次翻倍
        backOff.setMaxInterval(5000);       // 最大等待 5 秒

        // 加随机抖动，避免所有客户端同一时刻重试
        UniformRandomBackOffPolicy randomBackOff = new UniformRandomBackOffPolicy();
        randomBackOff.setMinBackOffPeriod(100);
        randomBackOff.setMaxBackOffPeriod(500);

        template.setBackOffPolicy(backOff);

        return template;
    }
}
```

使用方式：

```java
@Service
@Slf4j
public class PaymentService {

    @Autowired
    private RetryTemplate retryTemplate;
    @Autowired
    @Qualifier("payRestTemplate")
    private RestTemplate payRestTemplate;

    public PayResult pay(PayRequest request) {
        return retryTemplate.execute(context -> {
            int retryCount = context.getRetryCount();
            if (retryCount > 0) {
                log.warn("【支付重试】第{}次重试, 订单号={}", retryCount, request.getOrderNo());
            }
            return payRestTemplate.postForObject(
                    "http://pay-service/api/pay",
                    request, PayResult.class);
        }, context -> {
            // 所有重试都失败后的降级逻辑
            log.error("【支付失败】重试{}次后仍失败, 订单号={}",
                    context.getRetryCount(), request.getOrderNo());
            throw new BizException("支付服务暂时不可用，请稍后重试");
        });
    }
}
```

### 三、幂等键设计

重试的前提是接口幂等。同一个请求执行多次，结果应该和执行一次一样。

**幂等键的设计原则**：业务唯一标识 + 操作类型。

```java
// 数据库表设计
CREATE TABLE idempotent_record (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    idempotent_key VARCHAR(128) NOT NULL UNIQUE,  -- 幂等键
    request_hash VARCHAR(64) NOT NULL,             -- 请求体哈希（防参数篡改）
    result TEXT,                                    -- 首次执行的结果
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    expire_at DATETIME NOT NULL,                   -- 过期时间
    INDEX idx_key (idempotent_key)
);
```

```java
@Component
@Slf4j
public class IdempotentHelper {

    @Autowired
    private IdempotentRecordMapper mapper;

    /**
     * 幂等执行
     * @param idempotentKey 幂等键，如 "pay:order:20260525001"
     * @param requestHash 请求体的哈希值
     * @param supplier 实际执行逻辑
     * @param ttlMinutes 幂等记录保留时间
     */
    public <T> T execute(String idempotentKey, String requestHash,
                         Supplier<T> supplier, int ttlMinutes) {
        // 1. 查询是否已执行过
        IdempotentRecord existing = mapper.selectByKey(idempotentKey);
        if (existing != null) {
            // 校验请求参数是否一致（防止用相同 key 不同参数调用）
            if (!existing.getRequestHash().equals(requestHash)) {
                throw new BizException("幂等键冲突：参数不一致");
            }
            log.info("【幂等命中】key={}", idempotentKey);
            return JSON.parseObject(existing.getResult(), new TypeReference<T>() {});
        }

        // 2. 未执行过，先插入记录（占位）
        IdempotentRecord record = new IdempotentRecord();
        record.setIdempotentKey(idempotentKey);
        record.setRequestHash(requestHash);
        record.setExpireAt(LocalDateTime.now().plusMinutes(ttlMinutes));

        try {
            mapper.insert(record);
        } catch (DuplicateKeyException e) {
            // 并发场景：另一个线程已经插入了，直接返回结果
            existing = mapper.selectByKey(idempotentKey);
            return JSON.parseObject(existing.getResult(), new TypeReference<T>() {});
        }

        // 3. 执行实际逻辑
        T result = supplier.get();

        // 4. 保存结果
        record.setResult(JSON.toJSONString(result));
        mapper.updateResult(record);

        return result;
    }
}
```

使用示例：

```java
public PayResult pay(PayRequest request) {
    // 幂等键 = 操作类型 + 业务唯一标识
    String idempotentKey = "pay:" + request.getOrderNo();
    String requestHash = DigestUtils.md5Hex(JSON.toJSONString(request));

    return idempotentHelper.execute(
            idempotentKey,
            requestHash,
            () -> doPay(request),
            60  // 保留 60 分钟
    );
}
```

### 四、重试风暴防范

针对开头提到的重试风暴，我们做了以下防护：

**1. 限制单机重试总量**

```java
@Component
public class RetryLimiter {

    // 滑动窗口计数器：10 秒内最多 100 次重试
    private final RateLimiter retryLimiter = RateLimiter.create(10);

    public boolean allowRetry() {
        return retryLimiter.tryAcquire();
    }
}
```

**2. 引入断路器（Resilience4j）**

```java
@Configuration
public class CircuitBreakerConfig {

    @Bean
    public CircuitBreaker payServiceCircuitBreaker() {
        CircuitBreakerConfig config = CircuitBreakerConfig.custom()
                // 失败率超过 50% 时打开断路器
                .failureRateThreshold(50)
                // 慢调用超过 60% 也算失败
                .slowCallRateThreshold(60)
                .slowCallDurationThreshold(Duration.ofSeconds(3))
                // 断路器打开 10 秒后进入半开状态
                .waitDurationInOpenState(Duration.ofSeconds(10))
                // 半开状态允许 5 次调用
                .permittedNumberOfCallsInHalfOpenState(5)
                // 滑动窗口大小
                .slidingWindowSize(20)
                .build();

        return CircuitBreaker.of("payService", config);
    }
}
```

**3. 给重试请求打标记**

```java
// 在请求头中加入重试标记
interceptor.execute(request -> {
    request.getHeaders().add("X-Retry-Count",
            String.valueOf(context.getRetryCount()));
    // 下游可以通过这个标记决定是否拒绝
    return execution.execute(request, body);
});
```

## 踩坑总结

1. **不是所有异常都应该重试**。4xx 不要重试，业务异常不要重试，只重试网络超时和服务端 5xx。

2. **重试必须配合幂等**。没有幂等保护的重试，比不重试更危险。

3. **指数退避 + 随机抖动**。避免所有客户端在同一时刻重试，否则会造成"惊群效应"。

4. **跨服务重试次数应该是乘法关系**。如果网关重试 3 次，订单服务重试 3 次，支付服务收到的请求量最多是 9 倍。所以链路中每个环节的重试次数要递减。

5. **超时 > P99**。超时时间设在 P99 以内，意味着大部分正常请求会被误杀。

6. **断路器是最后一道防线**。即使有重试策略，也要加断路器。当整体失败率过高时，直接短路返回，保护下游。
