### 问题场景

线上Java服务突然变慢，CPU飙到90%。日志里没有明显的报错，重启之后过一阵又飙上去。传统的排查方式——加日志、重新部署——太慢了。Arthas是阿里开源的Java诊断工具，可以在不重启、不修改代码的情况下，实时诊断线上问题。这篇文章记录了我们用Arthas排查几个真实线上问题的过程。

### 安装和连接

```bash
# 下载arthas-boot.jar
curl -O https://arthas.aliyun.com/arthas-boot.jar

# 启动，自动列出Java进程供选择
java -jar arthas-boot.jar

# 如果有多个Java进程，直接指定PID
java -jar arthas-boot.jar 12345

# Docker容器中使用
docker exec -it <container_id> /bin/bash
# 先下载arthas-boot.jar，然后连接
java -jar arthas-boot.jar 1
```

连接成功后会进入Arthas的交互式命令行。

### 常用命令实战

**dashboard：全局视图**

```
[arthas@12345]$ dashboard
```

一目了然：CPU使用率、内存使用、GC情况、活跃线程。这是我们排查问题的第一步。

**thread：线程分析**

```bash
# 查看所有线程
thread

# 查看CPU占用最高的N个线程
thread -n 5

# 查看指定线程的堆栈（假设线程ID是42）
thread 42

# 查看BLOCKED状态的线程（排查死锁）
thread -b
```

**实战案例**：某天下午3点，告警CPU持续高于80%。

```bash
[arthas@12345]$ thread -n 3
```

输出显示3个线程CPU占用最高，堆栈都指向同一个方法：

```
"pool-3-thread-1" Id=42 cpuUsage=28% RUNNABLE
    at com.ourcompany.service.DataProcessService.parseLargeFile(DataProcessService.java:156)
    at com.ourcompany.service.DataProcessService.process(DataProcessService.java:89)
```

定位到是批量导入功能中，解析大文件时正则表达式出现了灾难性回溯（catastrophic backtracking）。

**watch：方法返回值监控**

```bash
# 监控方法的入参和返回值
watch com.ourcompany.service.OrderService getOrder "{params, returnObj}" -x 3

# 只看返回值
watch com.ourcompany.service.OrderService getOrder "{returnObj}" -x 2

# 只在方法抛异常时捕获
watch com.ourcompany.service.OrderService getOrder "{params, throwExp}" -e

# 同时观察耗时（#cost > 100表示超过100ms）
watch com.ourcompany.service.OrderService getOrder "{params, returnObj}" '#cost > 100' -x 2
```

`-x 3` 表示展开3层对象深度，不加的话只显示对象的toString()。

**trace：方法调用链路耗时**

```bash
# 追踪方法内部每个步骤的耗时
trace com.ourcompany.service.OrderService createOrder

# 过滤耗时超过200ms的调用
trace com.ourcompany.service.OrderService createOrder '#cost > 200'

# 包含JDK方法（默认不包含）
trace com.ourcompany.service.OrderService createOrder --skipJDKMethod false
```

**输出示例**：

```
`---ts=2026-05-25 14:30:15;thread_name=http-nio-8080-exec-3;cost=356ms
    +---[2ms] com.ourcompany.service.OrderService.validateParams()
    +---[125ms] com.ourcompany.service.InventoryService.deductStock()   # 这个最慢
    +---[8ms] com.ourcompany.dao.OrderMapper.insert()
    +---[218ms] com.ourcompany.service.PaymentService.createPayment()   # 这个也慢
    `---[3ms] com.ourcompany.service.NotificationService.sendOrderNotice()
```

一眼看出：扣库存调用耗时125ms，创建支付单耗时218ms。两个外部调用加起来就占了95%的耗时。

**monitor：方法调用统计**

```bash
# 每10秒统计一次方法的调用次数、成功/失败率、平均耗时
monitor -c 10 com.ourcompany.service.OrderService createOrder

# 统计某个类下所有方法
monitor -c 10 com.ourcompany.service.OrderService
```

输出：

```
 timestamp       class                      method   total  success  fail  avg-rt(ms)
 2026-05-25      OrderService               createOrder  523    520      3     89.45
```

**sc/sm：查看类和方法信息**

```bash
# 搜索类（支持通配符）
sc com.ourcompany.service.*Order*

# 查看类的详细信息（字段、方法、类加载器）
sc -d com.ourcompany.service.OrderService

# 查看类的方法列表
sm com.ourcompany.service.OrderService

# 查看方法的具体信息
sm -d com.ourcompany.service.OrderService createOrder
```

### 反编译热修复

线上发现Bug，但来不及走发布流程。Arthas可以直接反编译并热替换类。

**步骤1：反编译查看当前代码**

```bash
# 反编译类并输出到文件
jad --source-only com.ourcompany.service.OrderService > /tmp/OrderService.java
```

**步骤2：修改代码并编译**

```bash
# 编辑修复后的代码
vim /tmp/OrderService.java

# 编译修改后的类
mc -c <ClassLoaderHash> /tmp/OrderService.java -d /tmp/classes/
```

**步骤3：热替换**

```bash
# 用修改后的类替换线上的类
retransform /tmp/classes/com/ourcompany/service/OrderService.class
```

**重要警告**：
- 热修复只能修改方法体，不能添加/删除方法或字段
- 重启后失效，热修复只是临时方案，代码必须提交到仓库
- 操作前先做heap dump备份：`heapdump /tmp/before-fix.hprof`

### 火焰图

火焰图是分析CPU和方法调用的利器。Arthas内置了profiler命令。

```bash
# 开始采集（默认CPU火焰图）
profiler start

# 查看采集状态
profiler status

# 采集30秒后停止，生成HTML火焰图
profiler stop --format html --file /tmp/flamegraph.html

# 采集内存分配火焰图
profiler start --event alloc

# 采集阻塞/等待火焰图
profiler start --event wall

# 指定采集时长
profiler start --duration 60
```

**火焰图怎么看**：
- X轴：采样占比，越宽表示耗时越多
- Y轴：调用栈深度，从下往上是调用链
- 找最宽的"平顶"就是CPU消耗最大的方法

### CPU占用排查完整流程

这是我们在实际工作中排查CPU飙高的标准SOP：

```bash
# Step 1：dashboard看整体
dashboard

# Step 2：找CPU高的线程
thread -n 5

# Step 3：看线程堆栈，定位代码位置
thread <高CPU线程ID>

# Step 4：如果怀疑是GC导致，看GC情况
jvm

# Step 5：用trace确认方法耗时
trace com.ourcompany.xxx.YyyMethod '#cost > 50'

# Step 6：用profiler生成火焰图做全局分析
profiler start
# 等待30秒
profiler stop --format html --file /tmp/cpu-flame.html
```

**GC导致CPU高的判断**：

在dashboard中看到GC频繁、GC耗时长：

```
 gc.cms.count                    0
 gc.ps_scavenge.count           45
 gc.ps_scavenge.time(ms)        12345  # 这个值很大说明GC很频繁
 gc.ps_marksweep.count           3
 gc.ps_marksweep.time(ms)        8901
```

如果是GC问题，用Arthas的heapdump命令导出堆快照：

```bash
heapdump /tmp/heap-20260525.hprof
```

然后用MAT（Memory Analyzer Tool）或VisualVM分析哪些对象占了大量内存。

### 其他实用命令

```bash
# 查看系统属性
sysprop

# 查看环境变量
sysenv

# 查看类加载器信息（排查类冲突）
classloader

# 搜索已加载的类中包含特定注解的
sc -d * @org.springframework.stereotype.Service

# 临时修改日志级别（不用重启）
logger --name ROOT --level DEBUG
# 恢复
logger --name ROOT --level INFO

# OGNL表达式直接执行
ognl '@com.ourcompany.config.AppConfig@getMaxRetryCount()'
```

**临时修改日志级别**是排查问题的利器。线上出问题时，临时把日志级别调到DEBUG，看详细日志，排查完再调回去。

### 踩坑总结

1. **Arthas会消耗少量性能**。长期挂在生产环境可以，但profiler采集和trace会明显影响性能，用完及时停止
2. **watch的-x参数很重要**。不加 `-x` 只显示对象地址和toString()，看不到内部字段。但层数不要设太大，3-4层就够了
3. **反编译热修复是双刃剑**。救急可以，但一定要同步修复代码并走正常发布流程
4. **火焰图要采样足够长时间**。5秒以内的采样可能有偶然性，建议至少30秒
5. **Arthas本身的jar包不要放在应用的classpath里**。独立下载运行，避免版本冲突
6. **Docker容器中使用Arthas**：确保基础镜像里有JDK而不仅是JRE，否则无法运行Arthas

Arthas是Java后端开发的瑞士军刀。不需要每次都用，但出问题时能快速定位问题，比翻日志高效得多。建议团队里每个人都在测试环境练习过一遍常用命令，不要等线上出问题了才第一次用。
