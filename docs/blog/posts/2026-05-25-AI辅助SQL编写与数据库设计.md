## 场景引入

作为后端开发，日常工作中写SQL的时间不少：新功能要建表写查询，线上要排查慢查询，数据分析要写统计SQL。这些任务技术含量不低但重复性高，AI能不能帮忙提效？

这篇文章基于我在MySQL 8.0 + Spring Boot项目中的实际使用经验，告诉你AI在SQL领域的强项和软肋。

## AI擅长的场景

### 1. 基础CRUD SQL生成

这是AI的绝对强项。给出表结构和业务需求，AI能快速生成正确的SQL：

```
表结构：
CREATE TABLE orders (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id BIGINT NOT NULL,
    total_amount DECIMAL(10,2) NOT NULL,
    status TINYINT NOT NULL DEFAULT 0 COMMENT '0待付款1已付款2已发货3已完成4已取消',
    create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    update_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_user_id (user_id),
    INDEX idx_status (status),
    INDEX idx_create_time (create_time)
);

需求：查询某用户最近3个月的已付款订单，按金额降序，分页
```

AI生成（质量很好，直接可用）：
```sql
SELECT id, total_amount, status, create_time
FROM orders
WHERE user_id = #{userId}
  AND status IN (1, 2, 3)
  AND create_time >= DATE_SUB(NOW(), INTERVAL 3 MONTH)
ORDER BY total_amount DESC
LIMIT #{offset}, #{pageSize};
```

### 2. 统计查询

数据分析类的SQL，AI比大多数人写得快：

```
需求：按月统计各状态订单的数量和总金额
```

```sql
SELECT 
    DATE_FORMAT(create_time, '%Y-%m') AS month,
    status,
    COUNT(*) AS order_count,
    SUM(total_amount) AS total_amount
FROM orders
WHERE create_time >= '2026-01-01'
GROUP BY DATE_FORMAT(create_time, '%Y-%m'), status
ORDER BY month DESC, status;
```

### 3. 建表语句生成

给出业务需求，AI生成的建表语句通常很规范：

```
需求：设计一个电商系统的购物车表，支持多规格商品
```

```sql
CREATE TABLE cart_items (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id BIGINT NOT NULL COMMENT '用户ID',
    product_id BIGINT NOT NULL COMMENT '商品ID',
    sku_id BIGINT NOT NULL COMMENT 'SKU ID',
    quantity INT NOT NULL DEFAULT 1 COMMENT '数量',
    selected TINYINT NOT NULL DEFAULT 1 COMMENT '是否选中',
    create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    update_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    UNIQUE KEY uk_user_sku (user_id, sku_id),
    INDEX idx_user_id (user_id),
    INDEX idx_product_id (product_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='购物车';
```

## AI的局限

### 1. 复杂JOIN查询

涉及3张以上表的复杂查询，AI经常出错：

```
需求：查询每个用户的最近一笔订单的商品名称和数量
```

AI可能生成：
```sql
-- AI的方案：子查询方式，性能可能不好
SELECT u.name, o.id, p.name, oi.quantity
FROM users u
JOIN orders o ON u.id = o.user_id
JOIN order_items oi ON o.id = oi.order_id
JOIN products p ON oi.product_id = p.id
WHERE o.create_time = (
    SELECT MAX(create_time) FROM orders WHERE user_id = u.id
);
```

问题：
- 子查询在大数据量下性能差
- 一个用户可能有多笔同一时间创建的订单
- 没有考虑分页

更好的写法需要用窗口函数，但AI不一定能想到：
```sql
WITH ranked_orders AS (
    SELECT *, ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY create_time DESC) AS rn
    FROM orders
)
SELECT u.name, ro.id, p.name, oi.quantity
FROM ranked_orders ro
JOIN users u ON u.id = ro.user_id
JOIN order_items oi ON ro.id = oi.order_id
JOIN products p ON oi.product_id = p.id
WHERE ro.rn = 1;
```

### 2. 性能优化

给AI一段慢SQL让它优化，AI的建议通常停留在"加索引"层面：

```
-- AI的典型建议
1. 给WHERE条件的字段添加索引
2. 避免使用SELECT *
3. 避免在WHERE中使用函数
```

这些是对的，但不够。真正复杂的优化需要：
- 分析执行计划（EXPLAIN）
- 理解数据分布和基数
- 考虑索引选择性和联合索引顺序
- 评估不同写法的IO差异

**建议**：先用EXPLAIN分析执行计划，把结果给AI，让它基于执行计划给出建议：

```
以下SQL执行时间为3秒，请分析EXPLAIN结果并给出优化建议：

SQL: [粘贴SQL]

EXPLAIN结果:
[粘贴EXPLAIN输出]

表数据量：orders表约500万行
索引情况：[列出已有索引]
```

这种方式下AI的建议质量会好很多。

### 3. 数据迁移SQL

涉及数据迁移的SQL，AI很容易忽略数据一致性：

```sql
-- AI可能生成的迁移SQL
UPDATE orders SET status = 5 WHERE status = 3 AND create_time < '2025-01-01';
-- 问题：没有考虑并发写入、没有备份、没有分批执行
```

安全的迁移方式：
```sql
-- 1. 先备份
CREATE TABLE orders_backup_20260525 AS 
SELECT * FROM orders WHERE status = 3 AND create_time < '2025-01-01';

-- 2. 分批执行
UPDATE orders SET status = 5 
WHERE status = 3 AND create_time < '2025-01-01' 
LIMIT 1000;

-- 3. 验证数据
SELECT COUNT(*) FROM orders WHERE status = 5 AND create_time < '2025-01-01';
```

AI很少会主动提醒你做备份和分批执行。

### 4. MyBatis动态SQL

AI对MyBatis的动态SQL（XML方式）支持一般：

```xml
<!-- AI生成的动态SQL，条件判断经常遗漏 -->
<select id="queryOrders" resultType="Order">
    SELECT * FROM orders
    <where>
        <if test="userId != null">
            AND user_id = #{userId}
        </if>
        <if test="status != null">
            AND status = #{status}
        </if>
        <!-- AI可能遗漏日期范围的处理 -->
    </where>
</select>
```

## 总结

AI在SQL领域的定位是**"快速起手，人工优化"**。它能帮你：
- 快速生成基础SQL，省去手写的时间
- 提供优化思路，但需要结合实际执行计划验证
- 生成规范的建表语句

它不能帮你：
- 做精确的性能调优（需要EXPLAIN和实际测试）
- 处理复杂的数据迁移（需要考虑一致性、回滚方案）
- 做分库分表的架构决策

把AI当作"SQL助手"，让它帮你打草稿，你来定稿和优化。
