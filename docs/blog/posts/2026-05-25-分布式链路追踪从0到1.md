## 没有链路追踪的日子

之前排查一个线上慢请求，用户反馈"下单卡了 10 秒"。我们有 5 个微服务参与下单流程：网关 -> 订单服务 -> 库存服务 -> 支付服务 -> 通知服务。

排查过程：登录 5 台服务器分别 grep 日志，靠时间和参数人工关联。光找到是哪个服务慢就花了 2 小时，最后发现是支付服务调用银行接口超时。

从那以后，我们开始建设链路追踪。

## 选型对比：SkyWalking vs Zipkin vs Jaeger

| 维度 | SkyWalking | Zipkin | Jaeger |
|------|-----------|--------|--------|
| Java Agent 探针 | 原生支持，字节码增强，无需改代码 | 需要 Brave SDK 或 Spring Cloud Sleuth | 需要 OpenTelemetry SDK |
| 存储后端 | Elasticsearch / BanyanDB / H2 | Elasticsearch / MySQL / Cassandra | Elasticsearch / Cassandra |
| UI | 功能丰富，拓扑图、Trace、指标都有 | 基础的 Trace 查看 | 功能尚可，但不如 SW 完善 |
| 性能损耗 | 低（<3%，官方数据） | 中等 | 中等 |
| 社区活跃度 | Apache 顶级项目，国内社区活跃 | Spring 生态绑定，社区一般 | CNCF 项目，社区活跃 |
| 上手难度 | 配置简单，Java Agent 加个参数就行 | 需要代码埋点或 Sleuth | 需要 OpenTelemetry |

我们选了 **SkyWalking**，核心原因：

1. Java Agent 无侵入接入，不用改一行代码
2. 自带拓扑图、JVM 监控、慢 SQL 检测，功能开箱即用
3. 团队之前用 ELK，SkyWalking 支持 Elasticsearch 存储，复用现有 ES 集群

## SkyWalking 接入步骤

### 1. 部署 OAP Server 和 UI

用 Docker Compose 快速部署：

```yaml
version: '3'
services:
  oap:
    image: apache/skywalking-oap-server:9.7.0
    environment:
      SW_STORAGE: elasticsearch
      SW_STORAGE_ES_CLUSTER_NODES: elasticsearch:9200
      SW_CORE_RECORD_DATA_TTL: 7      # Trace 数据保留 7 天
      SW_CORE_METRICS_DATA_TTL: 30    # 指标数据保留 30 天
    ports:
      - "11800:11800"  # gRPC 上报端口
      - "12800:12800"  # REST 查询端口

  ui:
    image: apache/skywalking-ui:9.7.0
    environment:
      SW_OAP_ADDRESS: http://oap:12800
    ports:
      - "8080:8080"
```

### 2. Java 应用接入

启动时加 Java Agent 参数：

```bash
java -javaagent:/opt/skywalking/agent/skywalking-agent.jar \
     -Dskywalking.agent.service_name=order-service \
     -Dskywalking.collector.backend_service=skywalking-oap:11800 \
     -jar order-service.jar
```

Dockerfile 里加一行就行：

```dockerfile
COPY agent/ /opt/skywalking/agent/
ENV JAVA_AGENT_OPTS="-javaagent:/opt/skywalking/agent/skywalking-agent.jar"
ENTRYPOINT ["sh", "-c", "java $JAVA_AGENT_OPTS -jar app.jar"]
```

**零代码改动**，SkyWalking Agent 通过字节码增强自动埋点 HTTP 请求、数据库查询、RPC 调用。

### 3. 自定义 TraceId 打入日志

链路追踪最有价值的功能之一是：**通过 TraceId 把一次请求在所有服务的日志串起来**。

需要在 logback 配置里加 TraceId：

```xml
<!-- logback-spring.xml -->
<appender name="CONSOLE" class="ch.qos.logback.core.ConsoleAppender">
    <encoder>
        <!-- %tid 是 SkyWalking Agent 注入的 TraceId 占位符 -->
        <pattern>%d{yyyy-MM-dd HH:mm:ss.SSS} [%tid] [%thread] %-5level %logger{36} - %msg%n</pattern>
    </encoder>
</appender>
```

注意：`%tid` 占位符需要 SkyWalking Agent 6.6+ 版本，它会自动注入 logback 的 MDC。

如果没有用 SkyWalking 的 tid（比如某些场景下 Agent 没覆盖到），可以手动传递：

```java
// 手动设置 TraceId 到 MDC
MDC.put("tid", TraceContext.traceId());
```

## 跨线程和异步场景处理

这是最容易踩坑的地方。

### 场景一：@Async 异步方法

```java
// 坑：直接用 @Async，TraceId 会丢失
@Async
public void sendNotification(Long orderId) {
    // 这里的 TraceId 是空的
    log.info("发送通知, orderId={}", orderId);
}
```

SkyWalking Agent 会自动处理 `@Async` 和 `ThreadPoolExecutor` 的上下文传播，**但前提是线程池必须是 Spring 管理的**。

```java
@Configuration
@EnableAsync
public class AsyncConfig {
    // 必须用 Spring 管理的线程池，SkyWalking Agent 才能正确传播上下文
    @Bean("eventExecutor")
    public Executor eventExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(5);
        executor.setMaxPoolSize(10);
        executor.setQueueCapacity(100);
        executor.setThreadNamePrefix("event-");
        executor.initialize();
        return executor;
    }
}
```

### 场景二：手动创建的线程池

```java
// 如果必须手动创建线程池，用 SkyWalking 的 API 手动传播上下文
ExecutorService executor = Executors.newFixedThreadPool(10);

// 包装 Runnable，传播 Trace 上下文
Runnable task = RunnableWrapper.of(() -> {
    log.info("异步任务执行"); // TraceId 可以正确传递
});
executor.submit(task);
```

### 场景三：CompletableFuture

```java
// CompletableFuture 的上下文传播
CompletableFuture.runAsync(() -> {
    // SkyWalking Agent 4.x+ 已经支持自动传播
    log.info("CompletableFuture 内部, traceId={}",
        MDC.get("tid"));
}, eventExecutor);
```

**踩坑记录**：我们在用 `CompletableFuture.allOf()` 并行调用多个服务时，发现只有第一个 future 能正确关联 trace，其余的 trace 是独立的。原因是 SkyWalking 对 `CompletableFuture` 的增强在某些组合操作（如 `allOf`、`thenCombine`）上有 bug。最终我们换成了 `@Async` + `CountDownLatch` 的方式。

## 生产环境调优

### 采样率

全量采集 Trace 数据对性能和存储的压力都很大。我们日均请求量 500 万，全量存储每天要 50GB 的 ES 空间。

```yaml
# agent.config
agent.sample_n_per_3_secs=100  # 每 3 秒最多采集 100 条 Trace
```

后来改用**尾部采样策略**：只保存慢请求（>1s）和错误请求的完整 Trace，正常请求按 10% 采样。这个需要 OAP Server 端配置：

```yaml
# config/application.yml
agent.sample rate: 10  # 10% 采样
```

### 存储优化

ES 索引策略：
- Trace 数据 TTL 设为 7 天
- 指标数据 TTL 设为 30 天
- 开启 ES 的 ILM（Index Lifecycle Management）自动清理过期索引

### 和日志系统打通

在 Kibana 里通过 TraceId 搜索所有服务的日志，这是排查问题最高效的手段：

```json
{
  "query": {
    "match": {
      "tid": "abc123def456"
    }
  },
  "sort": [{"@timestamp": "asc"}]
}
```

## 踩坑总结

1. **Agent 版本和 Java 版本的兼容性**：SkyWalking Agent 8.x 不完全支持 Java 17+ 的 sealed class。我们升级 Java 17 后有几个类报 `VerifyError`，最后升到 Agent 9.5 才解决。

2. **gRPC 端口冲突**：如果应用本身也用 gRPC（比如 Dubbo），要注意 OAP 的 gRPC 端口不要和业务端口冲突。我们用的是 11800。

3. **K8s 环境下的 Agent 挂载**：用 Init Container 把 Agent JAR 挂载进去，不要在每个业务镜像里打包 Agent，否则升级 Agent 版本要重新构建所有镜像。

4. **不要用 SkyWalking 的日志收集功能**：它有这个功能，但性能很差。日志还是走 ELK 体系，SkyWalking 只负责 Trace 和 Metrics，各司其职。

链路追踪从 0 到 1 的投入其实不大，2 天就能把 OAP 部署好、所有服务接入 Agent。但它带来的排查效率提升是数量级的——以前 2 小时的问题现在 5 分钟就能定位。
