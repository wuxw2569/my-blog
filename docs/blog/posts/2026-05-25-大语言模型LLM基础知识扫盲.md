# 大语言模型（LLM）基础知识扫盲

## 前言

如果你是一个后端开发、前端开发或者全栈工程师，最近一两年你一定被各种"AI"、"大模型"、"LLM"的新闻刷屏了。但说实话，很多科普文章要么太学术、要么太营销，看完之后你可能还是不太清楚这东西到底是什么、能干什么、不能干什么。

这篇文章就站在一个"有编程基础但AI知识薄弱的开发者"的角度，把 LLM 最核心的概念讲清楚。

## LLM 到底是什么？

**LLM（Large Language Model），大语言模型。** 你可以把它理解成一个"超级自动补全器"。

就像你手机输入法会根据你已打的字猜测下一个字一样，LLM 也是在做类似的事情——根据前面的文本，预测下一个最可能出现的词。只不过它的规模大了几个数量级，训练数据覆盖了互联网上大量的文本，所以它"补全"出来的内容看起来像是一个知识渊博的人在回答问题。

用一个更具体的类比：

- **传统程序**：你写好规则，它按规则执行 → 规则是确定的
- **LLM**：你给它一段话，它基于概率生成下一段话 → 输出是有随机性的

所以 LLM 本质上是一个**概率模型**，不是数据库，也不是搜索引擎。它不"知道"答案，而是"推测"出一个合理的回答。

## Token：LLM 世界的"货币单位"

你调用 LLM API 的时候，会看到 `token` 这个概念。Token 是 LLM 处理文本的最小单位。

**Token 不等于汉字。**

- 英文中，一个单词可能被拆成 1-3 个 token（如 "unbelievable" → "un" + "believ" + "able"）
- 中文中，一个汉字通常是 1-2 个 token
- 大致经验值：1 个 token 约等于 0.75 个英文单词，或者 1.5 个中文字符

Token 为什么重要？因为：

1. **计费单位**：API 调用按 token 计费，输入和输出分别算
2. **上下文窗口的度量**：模型一次能处理多少 token 是有限制的
3. **响应速度**：token 数越多，生成越慢

```python
# 使用 tiktoken 库估算 token 数（GPT 系列）
import tiktoken

encoder = tiktoken.encoding_for_model("gpt-4o")
text = "你好，世界！Hello World!"
tokens = encoder.encode(text)
print(f"文本长度: {len(text)}, Token数: {len(tokens)}")
# 大约输出: 文本长度: 16, Token数: 6
```

## 上下文窗口：模型的"短期记忆"

上下文窗口（Context Window）是指模型一次能"看到"多少 token 的内容。你可以把它理解为模型的"工作记忆"或"短期记忆"。

| 模型 | 上下文窗口（约） |
|------|-----------------|
| GPT-4o | 128K tokens |
| Claude 3.5 Sonnet | 200K tokens |
| Gemini 1.5 Pro | 1M-2M tokens |
| DeepSeek-V3 | 128K tokens |
| 通义千问 | 128K tokens |

**超过上下文窗口的内容，模型是"看不见"的。** 这就像你同时看一本 300 页的书和一本 30 页的书，你对 30 页的内容记忆肯定更清晰。

对开发者的实际影响：

- 如果你的 prompt 太长（比如塞了一整本书的内容），可能超出窗口限制
- 超长上下文不代表模型能精准利用每一个细节，它对中间部分的内容关注度往往较低（学术上叫 "Lost in the Middle"）
- 在实际应用中，与其把一大堆内容塞进 prompt，不如先做一轮筛选和摘要

## Temperature 和 Top-P：控制模型的"创造力"

这两个参数决定了模型输出的随机性。

**Temperature（温度）**：

- 范围通常是 0-2，默认值一般是 1
- Temperature 越低，模型越"保守"，倾向于选概率最高的那个词
- Temperature 越高，模型越"放飞自我"，会选一些概率较低但可能更有创意的词

用类比来说：

- Temperature = 0：像一个严谨的公务员写报告，措辞标准、少有惊喜
- Temperature = 1：像一个正常的聊天，有逻辑但也有一些变化
- Temperature = 1.5：像喝了两杯酒之后聊天，有趣但可能跑偏

**Top-P（核采样）**：

- 范围 0-1，默认值一般是 1
- Top-P = 0.1：只从概率最高的前 10% 的候选词中选择
- Top-P = 1：从所有候选词中选择

**开发者怎么选？**

| 场景 | 建议设置 |
|------|---------|
| 代码生成、数据提取、分类任务 | Temperature = 0 或很低 |
| 正常对话、邮件撰写 | Temperature = 0.7 |
| 创意写作、头脑风暴 | Temperature = 1.0+ |

一般建议：**调一个就行**，不要同时调 Temperature 和 Top-P，容易互相干扰。

## LLM 能干什么？不能干什么？

作为开发者，你需要清楚 LLM 的能力边界：

**LLM 擅长的事：**

- 文本理解与生成（写文章、翻译、摘要）
- 代码生成与解释（写函数、debug、代码审查）
- 逻辑推理（简单的数学、因果推断）
- 格式转换（JSON ↔ XML、Markdown ↔ HTML）
- 知识问答（在训练数据范围内）

**LLM 不擅长或做不到的事：**

- **精确计算**：它不是计算器，做复杂算术会出错
- **实时信息**：它有训练数据截止日期，不知道"今天"发生了什么
- **保持一致性**：同样的问题问两次，可能给出不同的答案
- **严格遵循规则**：复杂的业务规则很难通过 prompt 完全约束
- **专业领域深度知识**：在非常垂直的领域可能产生"一本正经地胡说八道"（即幻觉 Hallucination）

## 一些给开发者的实用建议

1. **不要把 LLM 当万能工具**。它是一个强大的文本处理引擎，但不是数据库、不是搜索引擎、不是计算器。该用传统工具的地方就用传统工具。

2. **Prompt 是你的新"代码"**。和 LLM 交互的核心方式就是写好 prompt。一个好的 prompt 和一个差的 prompt，结果差距可能是天壤之别。

3. **理解概率性本质**。不要期望 LLM 的输出 100% 准确。在你的应用中需要有兜底方案和校验机制。

4. **关注 token 消耗**。在生产环境中，不必要的 token 消耗就是真金白银。学会精简 prompt、合理设置 max_tokens。

## 总结

LLM 本质上是一个超大规模的文本概率模型，通过 Token 处理文本，受上下文窗口限制，通过 Temperature 等参数控制输出风格。它在文本理解和生成方面非常强大，但在精确计算、实时信息、严格一致性方面有明显短板。理解这些基本概念，是你在实际项目中用好 AI 的第一步。
