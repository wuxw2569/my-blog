## 我们不需要"完整的DDD"

DDD（Domain-Driven Design）在技术圈被神化了。很多人一上来就要搞四层架构（接口层、应用层、领域层、基础设施层）、CQRS、Event Sourcing，结果业务代码没写几行，架构代码堆了一大堆。

我在一家 100 人的 SaaS 公司做后端，用了一种"够用的 DDD"——只取核心思想，不搞形式主义。以下是我们团队的真实实践。

## 聚合根：保护业务不变量

聚合根是 DDD 里最有用的概念之一。**核心思想是：通过聚合根来保证业务规则的一致性，不允许绕过聚合根直接操作内部对象。**

### 实际案例：订单聚合

```java
/**
 * 订单聚合根 - 所有对订单的操作都必须通过这个类
 */
public class Order {
    private Long id;
    private Long userId;
    private OrderStatus status;
    private Money totalAmount;
    private List<OrderItem> items; // 内部实体
    private Address shippingAddress; // 值对象
    private List<DomainEvent> events = new ArrayList<>();

    /**
     * 下单 - 坑：不能先创建订单再校验库存，校验逻辑要内聚在聚合根里
     * （当然库存查询是外部依赖，这里只做订单维度的校验）
     */
    public static Order create(Long userId, List<OrderItemCommand> itemCommands,
                               Address address) {
        if (itemCommands == null || itemCommands.isEmpty()) {
            throw new BizException(ErrorCode.PARAM_ERROR, "订单商品不能为空");
        }
        if (address == null) {
            throw new BizException(ErrorCode.PARAM_ERROR, "收货地址不能为空");
        }

        Order order = new Order();
        order.userId = userId;
        order.status = OrderStatus.CREATED;
        order.shippingAddress = address;
        order.items = itemCommands.stream()
            .map(cmd -> new OrderItem(cmd.getProductId(),
                cmd.getProductName(), cmd.getPrice(), cmd.getQuantity()))
            .collect(Collectors.toList());
        order.totalAmount = order.calculateTotal();

        // 发布领域事件
        order.events.add(new OrderCreatedEvent(order.id, order.userId,
            order.totalAmount));
        return order;
    }

    /**
     * 支付完成 - 状态流转有严格的前置条件
     * 坑：不要在 Service 里写 if(status == CREATED) 然后直接 setStatus
     */
    public void paySuccess(String payNo, Money paidAmount) {
        if (this.status != OrderStatus.CREATED) {
            throw new BizException("订单状态不正确，无法支付确认");
        }
        if (!this.totalAmount.equals(paidAmount)) {
            throw new BizException("支付金额与订单金额不一致");
        }
        this.status = OrderStatus.PAID;
        this.events.add(new OrderPaidEvent(this.id, payNo, paidAmount));
    }

    /**
     * 取消订单 - 只有待支付状态才能取消
     */
    public void cancel(String reason) {
        if (this.status != OrderStatus.CREATED) {
            throw new BizException("只有待支付的订单才能取消");
        }
        this.status = OrderStatus.CANCELLED;
        this.events.add(new OrderCancelledEvent(this.id, reason));
    }

    private Money calculateTotal() {
        return items.stream()
            .map(OrderItem::getSubtotal)
            .reduce(Money.ZERO, Money::add);
    }
}
```

**关键点**：所有改变订单状态的操作都通过 `Order` 这个聚合根的方法来完成，方法内部做业务规则校验。`OrderItem` 不能脱离 `Order` 被单独修改。

### 值对象：用 Money 替代 BigDecimal

值对象没有唯一标识，通过属性值判断相等。我们用得最多的是 `Money`：

```java
@Value
public class Money {
    BigDecimal amount;
    String currency;

    public static final Money ZERO = new Money(BigDecimal.ZERO, "CNY");

    public Money(BigDecimal amount, String currency) {
        if (amount.compareTo(BigDecimal.ZERO) < 0) {
            throw new BizException("金额不能为负");
        }
        this.amount = amount.setScale(2, RoundingMode.HALF_UP);
        this.currency = currency;
    }

    public Money add(Money other) {
        assertSameCurrency(other);
        return new Money(this.amount.add(other.amount), this.currency);
    }

    public Money multiply(int quantity) {
        return new Money(this.amount.multiply(BigDecimal.valueOf(quantity)),
            this.currency);
    }

    private void assertSameCurrency(Money other) {
        if (!this.currency.equals(other.currency)) {
            throw new BizException("币种不一致，无法计算");
        }
    }
}
```

**为什么值对象有用**：之前我们直接用 `BigDecimal`，到处都有 `new BigDecimal("0")` 和 `null` 的判断。封装成 `Money` 后，金额相关的 bug 减少了至少 70%。

## 领域事件：解耦聚合根之间的通信

订单创建后需要扣减库存、发送通知、记录日志。如果在 `OrderService` 里直接调用这些服务，就是强耦合。

```java
// 领域事件基类
@Data
public abstract class DomainEvent {
    private final String eventId = UUID.randomUUID().toString();
    private final LocalDateTime occurredOn = LocalDateTime.now();
}

// 订单已创建事件
@Getter
@AllArgsConstructor
public class OrderCreatedEvent extends DomainEvent {
    private final Long orderId;
    private final Long userId;
    private final Money totalAmount;
}
```

事件发布和消费：

```java
@Service
@RequiredArgsConstructor
public class OrderApplicationService {
    private final OrderRepository orderRepository;
    private final ApplicationEventPublisher eventPublisher;

    @Transactional
    public Long createOrder(CreateOrderCommand command) {
        // 用 ApplicationEventPublisher 发布，和 Spring 生态无缝衔接
        Order order = Order.create(command.getUserId(),
            command.getItems(), command.getAddress());
        orderRepository.save(order);

        // 发布所有领域事件
        order.getDomainEvents().forEach(eventPublisher::publishEvent);
        order.clearEvents();
        return order.getId();
    }
}

// 事件监听器 - 各个关注方自己订阅
@Component
@Slf4j
public class OrderEventListener {

    @EventListener
    @Async("eventExecutor") // 异步处理，不阻塞主流程
    public void onOrderCreated(OrderCreatedEvent event) {
        log.info("【订单事件】收到订单创建事件, orderId={}", event.getOrderId());
        // 扣减库存
        inventoryService.deduct(event.getOrderId());
        // 发送通知
        notificationService.sendOrderCreated(event.getUserId(), event.getOrderId());
    }
}
```

**踩坑**：`@EventListener` 默认是同步的，如果监听器里有耗时操作，一定要加 `@Async`。我们一开始没加，导致下单接口 P99 延迟从 200ms 涨到 2s。另外，异步事件要注意事务边界——如果主事务还没提交，监听器就开始执行了，可能会查不到数据。解决方案：

```java
// 使用事务同步，在事务提交后再执行事件监听
@TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
public void onOrderCreated(OrderCreatedEvent event) {
    // 事务已提交，可以安全地查询订单数据
}
```

## Repository 模式：领域层不依赖数据库

```java
// 领域层定义接口
public interface OrderRepository {
    Order findById(Long id);
    void save(Order order);
}

// 基础设施层实现
@Repository
@RequiredArgsConstructor
public class OrderRepositoryImpl implements OrderRepository {
    private final OrderMapper orderMapper;
    private final OrderItemMapper orderItemMapper;

    @Override
    public Order findById(Long id) {
        OrderPO po = orderMapper.selectById(id);
        List<OrderItemPO> items = orderItemMapper.selectByOrderId(id);
        return OrderConverter.toDomain(po, items);
    }

    @Override
    public void save(Order order) {
        OrderPO po = OrderConverter.toPO(order);
        if (po.getId() == null) {
            orderMapper.insert(po);
            order.setId(po.getId());
        } else {
            orderMapper.updateById(po);
        }
        // 简化处理：全量覆盖子项（实际生产环境可以做 diff 更新）
        orderItemMapper.deleteByOrderId(order.getId());
        order.getItems().forEach(item -> {
            OrderItemPO itemPO = OrderConverter.toItemPO(order.getId(), item);
            orderItemMapper.insert(itemPO);
        });
    }
}
```

## 不需要 DDD 的地方

说实话，不是所有业务都适合 DDD。以下场景我们没有用：

- **CRUD 管理后台**：后台配置类的操作，简单 Service + Mapper 就够了
- **报表查询**：读多写少的场景，直接 SQL 查询，不需要聚合根
- **批量导入导出**：纯数据处理，没有复杂的业务规则

**我们的原则**：有复杂业务规则和状态流转的地方用聚合根保护；纯 CRUD 的地方保持简单。不要为了"架构统一"把简单事情搞复杂。

## 简化 DDD 的落地清单

1. 包结构按领域划分，不按技术层划分：`order/`、`product/`、`inventory/`
2. 聚合根用纯 Java 对象，不加框架注解（不用 JPA Entity）
3. 值对象用 `@Value`（Lombok），不可变
4. 领域事件用 Spring 的 `ApplicationEventPublisher`，不引入 MQ
5. Repository 接口在领域层，实现在基础设施层
6. ApplicationService 做事务编排，不放业务逻辑

这套"简化版 DDD"在我们团队跑了 1 年多，新同事上手基本没有障碍，而业务规则的内聚性比以前好很多。
