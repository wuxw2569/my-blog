## 前言

去年双十一大促前，我们一个核心交易服务突然大面积超时。排查后发现，对外部支付网关的 HTTP 调用全部卡住，Tomcat 线程池被打满。根本原因很简单：**RestTemplate 使用默认配置，没有设置连接池**。每次请求都新建 TCP 连接，高并发下端口耗尽，连接全部阻塞在 TIME_WAIT 状态。

这篇文章就从这次事故出发，讲讲 HTTP 连接池的配置、调优和踩坑经验。

## 问题场景

我们有一个订单服务，通过 RestTemplate 调用下游支付服务。代码大概是这样的：

```java
@Configuration
public class RestTemplateConfig {

    // 错误示范：直接 new，没有配置连接池
    @Bean
    public RestTemplate restTemplate() {
        return new RestTemplate();
    }
}
```

平时流量小，看不出问题。大促时 QPS 从 200 飙到 3000，问题暴露了：

- `netstat` 显示大量 TIME_WAIT 状态的连接（约 28000 个）
- 支付网关侧日志显示连接建立成功但请求没到
- 本机端口资源耗尽，新连接建不起来

## 分析思路

HTTP 客户端连接管理有三个核心参数需要关注：

| 参数 | 含义 | 影响 |
|------|------|------|
| maxTotal | 连接池最大连接数 | 决定整个客户端的并发上限 |
| maxPerRoute | 单个 host 的最大连接数 | 决定对单个服务的并发上限 |
| connectionTimeout | 建立连接的超时时间 | 过长会导致线程阻塞 |
| socketTimeout | 等待数据的超时时间 | 过长会拖慢请求响应 |

不同客户端实现的默认值差异很大，这也是踩坑的根源。

## 具体实现

### 方案一：RestTemplate + Apache HttpClient 连接池

这是 Spring 生态中最常见的组合，适合同步调用场景。

```java
@Configuration
public class RestTemplateConfig {

    @Bean
    public HttpClient httpClient() {
        PoolingHttpClientConnectionManager connectionManager =
                new PoolingHttpClientConnectionManager();
        // 总连接数，经验值：并发线程数的 1.5~2 倍
        connectionManager.setMaxTotal(200);
        // 单个 host 的最大连接数，下游单服务时与 maxTotal 设为一样
        connectionManager.setDefaultMaxPerRoute(200);

        RequestConfig requestConfig = RequestConfig.custom()
                // 连接超时：3秒，超过则认为服务不可用
                .setConnectTimeout(3000)
                // 读取超时：根据下游 P99 耗时设置，一般 5~10 秒
                .setSocketTimeout(10000)
                // 从连接池获取连接的超时：避免无限等待
                .setConnectionRequestTimeout(2000)
                .build();

        return HttpClientBuilder.create()
                .setConnectionManager(connectionManager)
                .setDefaultRequestConfig(requestConfig)
                // 开启空闲连接驱逐，避免使用已被服务端关闭的连接
                .evictExpiredConnections()
                .evictIdleConnections(30, TimeUnit.SECONDS)
                .build();
    }

    @Bean
    public RestTemplate restTemplate(HttpClient httpClient) {
        HttpComponentsClientHttpRequestFactory factory =
                new HttpComponentsClientHttpRequestFactory(httpClient);
        return new RestTemplate(factory);
    }
}
```

### 方案二：OkHttp 连接池

OkHttp 在 Android 开发中用得多，但后端场景也很合适，特别是它的连接复用机制做得更好。

```java
@Configuration
public class OkHttpConfig {

    @Bean
    public OkHttpClient okHttpClient() {
        ConnectionPool pool = new ConnectionPool(
                100,           // 最大空闲连接数
                5,             // 空闲连接存活时间
                TimeUnit.MINUTES
        );

        return new OkHttpClient.Builder()
                .connectionPool(pool)
                .connectTimeout(3, TimeUnit.SECONDS)
                .readTimeout(10, TimeUnit.SECONDS)
                .writeTimeout(10, TimeUnit.SECONDS)
                // 失败不自动重试，由业务层控制
                .retryOnConnectionFailure(false)
                .build();
    }
}
```

### 方案三：WebClient（Spring WebFlux）

如果项目是 Spring Boot 2.7+ 且有异步需求，WebClient 是更好的选择。

```java
@Configuration
public class WebClientConfig {

    @Bean
    public WebClient webClient() {
        HttpClient httpClient = HttpClient.create()
                .option(ChannelOption.CONNECT_TIMEOUT_MILLIS, 3000)
                .responseTimeout(Duration.ofSeconds(10))
                .connectionProvider(ConnectionProvider.builder("custom")
                        .maxConnections(200)
                        // 每个 host 等待连接的最大等待时间
                        .pendingAcquireTimeout(Duration.ofSeconds(3))
                        // 空闲连接回收
                        .maxIdleTime(Duration.ofSeconds(30))
                        .build());

        return WebClient.builder()
                .clientConnector(new ReactorClientHttpConnector(httpClient))
                .build();
    }
}
```

### 连接泄漏排查

连接池配置好之后，我们又遇到一个问题：**连接数持续增长，不释放**。排查步骤如下：

**第一步：通过 JMX 暴露连接池指标**

```java
// 注册 HttpClient 连接池的 MBean
HttpClientConnectionManager cm = httpClient.getConnectionManager();
if (cm instanceof PoolingHttpClientConnectionManager poolManager) {
    // 日志定期打印
    log.info("【连接池状态】total={}, available={}, leased={}",
            poolManager.getTotalStats().getMax(),
            poolManager.getTotalStats().getAvailable(),
            poolManager.getTotalStats().getLeased());
}
```

**第二步：arthas 在线诊断**

```bash
# 查看当前 HTTP 连接的调用栈
watch org.apache.http.impl.conn.PoolingHttpClientConnectionManager \
  requestConnection '{params, throwExp}' -n 5

# 查看连接池中的连接对象
ognl '@org.apache.http.impl.conn.PoolingHttpClientConnectionManager@getDefaultMaxPerRoute()'
```

**第三步：常见泄漏原因**

1. **response body 没有关闭**：最常见的坑

```java
// 错误写法：没有消费 response body
ResponseEntity<String> response = restTemplate.getForEntity(url, String.class);
String body = response.getBody();
// 如果 body 为 null，底层连接可能不会被释放

// 正确写法：使用 exchange 并确保 body 被消费
restTemplate.execute(url, HttpMethod.GET, null, ClientHttpResponse::getBody);
```

2. **异常路径没有释放连接**：try-catch 处理不当导致连接没有归还池。

## 踩坑总结

| 踩坑点 | 现象 | 解决方案 |
|--------|------|----------|
| 不配置连接池 | 高并发下端口耗尽 | 必须配置 PoolingHttpClientConnectionManager |
| maxPerRoute 太小 | 对单个服务并发上不去 | 设为与 maxTotal 相同或合理比例 |
| 没有空闲驱逐 | 使用被服务端关闭的连接导致异常 | 开启 evictExpiredConnections |
| socketTimeout 不合理 | 大文件传输被误断 | 按接口特性分别设置超时 |
| connectionRequestTimeout 不设 | 连接池满时无限等待 | 设置合理的等待上限 |
| response body 没消费 | 连接泄漏 | 确保 response body 被读取或关闭 |

**最后给一个参数选择的经验公式**：

- `maxTotal`：预估并发数 x 1.5（留出缓冲）
- `maxPerRoute`：如果只调一个下游，和 maxTotal 设一样；调多个下游则按比例分配
- `connectTimeout`：3 秒（内网环境一般足够）
- `socketTimeout`：看接口 P99 延迟，通常设为 P99 的 2~3 倍
- `connectionRequestTimeout`：2~5 秒（连接池排队上限）

连接池调优不是一次性的工作，需要结合监控数据持续调整。我们最终用 Prometheus + Grafana 接了连接池指标，在大促前做了压测验证，才算真正解决了这个问题。
