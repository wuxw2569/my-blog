## 问题场景

我们的核心MySQL实例（16核64G SSD）扛不住了。读请求占了80%的流量，写请求只占20%，但全都打在同一个实例上。监控显示CPU长期在70%以上，高峰期到90%。

方案很明确：**读写分离**。一主两从，读走从库，写走主库。

听起来简单，但真正落地后才发现——主从延迟是读写分离最大的敌人。

## 分析思路

### 1. 中间件选型：MyCat 还是 ShardingSphere？

| 方案 | 优点 | 缺点 |
|------|------|------|
| MyCat | 独立部署，应用无感知 | 社区维护弱，文档老旧，2.x版本不稳定 |
| ShardingSphere-JDBC | 客户端分片，零额外运维；与分库分表统一技术栈 | 侵入性强 |
| 代码层手动切换 | 完全可控 | 容易出错，每个Service都要记得指定数据源 |

**我们选了 ShardingSphere-JDBC**，和分库分表用同一套，降低技术栈复杂度。配置如下：

```yaml
spring:
  shardingsphere:
    datasource:
      names: master,slave0,slave1
      master:
        type: com.zaxxer.hikari.HikariDataSource
        jdbc-url: jdbc:mysql://10.0.1.100:3306/order_db
        username: order_user
        password: ${DB_PASSWORD}
        hikari:
          maximum-pool-size: 20
      slave0:
        type: com.zaxxer.hikari.HikariDataSource
        jdbc-url: jdbc:mysql://10.0.1.101:3306/order_db
        username: order_read
        password: ${DB_READ_PASSWORD}
        hikari:
          maximum-pool-size: 30
      slave1:
        type: com.zaxxer.hikari.HikariDataSource
        jdbc-url: jdbc:mysql://10.0.1.102:3306/order_db
        username: order_read
        password: ${DB_READ_PASSWORD}
        hikari:
          maximum-pool-size: 30
    rules:
      readwrite-splitting:
        data-sources:
          order_ds:
            write-data-source-name: master
            read-data-source-names: slave0,slave1
            load-balancer-name: round_robin
        load-balancers:
          round_robin:
            type: ROUND_ROBIN
```

### 2. 主从延迟到底有多严重？

先搞清楚主从延迟的原因。MySQL 主从复制是**异步**的：

```
主库写binlog -> 从库IO线程拉取binlog -> 从库SQL线程回放
```

延迟可能发生在任何一个环节。我们的监控数据：

- 正常情况下延迟 < 1秒
- 大促期间延迟最高到 15秒
- 大表DDL（如加字段）期间延迟到 30秒+

**延迟的典型影响场景**：

```
1. 用户下单 -> 写主库 -> 立刻查订单列表 -> 走从库 -> 没查到 -> "我的订单怎么没了？"
2. 用户改密码 -> 写主库 -> 用新密码登录 -> 走从库验证 -> "密码没改成功？"
3. 后台创建商品 -> 写主库 -> 前台搜索 -> 走ES/从库 -> "商品搜不到？"
```

### 3. 解决方案：分级策略

我们把场景分为三类，每类用不同的策略：

**第一类：必须走主库的场景**

写后立刻读的场景（下单后查订单、登录验证、支付回调确认）。

方案一：注解标记 + AOP拦截

```java
/**
 * 标记当前方法强制走主库
 */
@Target({ElementType.METHOD, ElementType.TYPE})
@Retention(RetentionPolicy.RUNTIME)
public @interface MasterRoute {
}
```

```java
@Aspect
@Component
@Order(Ordered.HIGHEST_PRECEDENCE)  // 最高优先级，确保在事务注解之前执行
public class MasterRouteAspect {

    private static final ThreadLocal<Boolean> MASTER_FLAG = new ThreadLocal<>();

    public static boolean isForceMaster() {
        return Boolean.TRUE.equals(MASTER_FLAG.get());
    }

    public static void clear() {
        MASTER_FLAG.remove();
    }

    @Around("@annotation(masterRoute)")
    public Object around(ProceedingJoinPoint point, MasterRoute masterRoute) throws Throwable {
        MASTER_FLAG.set(true);
        try {
            return point.proceed();
        } finally {
            MASTER_FLAG.remove();
        }
    }
}
```

```java
@Service
public class OrderService {

    @MasterRoute
    public Order getOrderAfterCreate(Long orderId) {
        // 强制走主库
        return orderMapper.selectById(orderId);
    }
}
```

方案二：利用 ShardingSphere 的 Hint 强制路由

```java
@Component
public class ShardingSphereHintHelper {

    /**
     * 强制后续查询走主库
     */
    public static void forceMaster() {
        HintManager hintManager = HintManager.getInstance();
        hintManager.setWriteRouteOnly();
        // 注意：使用完后要在finally中关闭
    }
}

// 使用方式
public Order getOrderForceMaster(Long orderId) {
    HintManager hintManager = HintManager.getInstance();
    try {
        hintManager.setWriteRouteOnly();
        return orderMapper.selectById(orderId);
    } finally {
        hintManager.close();
    }
}
```

**第二类：可以容忍延迟的场景**

商品列表、评论列表、统计报表等场景，延迟几秒完全可接受。直接走默认的读写分离即可，不需要特殊处理。

**第三类：需要平衡一致性和性能的场景**

订单列表（非刚创建的）、账户余额查询等。我们用了一个巧妙的方案——**延迟时间窗口**：

```java
/**
 * 基于延迟时间窗口的主从路由
 * 如果是最近N秒内修改过的数据，走主库；否则走从库
 */
@Component
public class DelayAwareRoutingService {

    @Autowired
    private StringRedisTemplate redisTemplate;

    private static final String DELAY_KEY_PREFIX = "db:recent_write:";
    private static final long DELAY_SECONDS = 5;

    /**
     * 标记某用户最近有写操作
     */
    public void markRecentWrite(Long userId) {
        String key = DELAY_KEY_PREFIX + userId;
        redisTemplate.opsForValue().set(key, "1", DELAY_SECONDS, TimeUnit.SECONDS);
    }

    /**
     * 判断是否需要走主库
     */
    public boolean shouldUseMaster(Long userId) {
        String key = DELAY_KEY_PREFIX + userId;
        return Boolean.TRUE.equals(redisTemplate.hasKey(key));
    }
}
```

```java
// 使用
public List<Order> getUserOrders(Long userId) {
    if (delayAwareRoutingService.shouldUseMaster(userId)) {
        ShardingSphereHintHelper.forceMaster();
    }
    return orderMapper.selectByUserId(userId);
}
```

在写操作完成后标记：

```java
@Service
public class OrderService {

    @Transactional
    public void createOrder(Order order) {
        orderMapper.insert(order);
        // 标记该用户最近有写操作，后续5秒内的查询走主库
        delayAwareRoutingService.markRecentWrite(order.getUserId());
    }
}
```

### 4. 从库故障处理

从库挂了一个怎么办？ShardingSphere 支持自动检测和摘除：

```yaml
spring:
  shardingsphere:
    rules:
      readwrite-splitting:
        data-sources:
          order_ds:
            # ... 其他配置
            load-balancer-name: round_robin
    props:
      # 从库故障自动摘除（需要配合健康检查）
      sql-show: false
```

但 ShardingSphere-JDBC 的故障摘除能力有限，我们额外加了一层保障：

```java
@Component
public class DataSourceHealthChecker {

    @Autowired
    private List<DataSource> slaveDataSources;

    @Scheduled(fixedRate = 10000)
    public void checkHealth() {
        for (DataSource ds : slaveDataSources) {
            try (Connection conn = ds.getConnection()) {
                conn.isValid(3);
            } catch (Exception e) {
                log.error("【从库健康检查失败】dataSource={}", ds, e);
                // 发钉钉告警
                dingTalkService.sendAlert("从库连接失败: " + e.getMessage());
            }
        }
    }
}
```

## 踩坑总结

1. **读写分离不能解决所有性能问题**。如果主库写入本身就慢（大量锁竞争），读写分离帮不了你。先优化写入SQL。
2. **主从延迟是客观存在的**，不是配置问题。哪怕用半同步复制，延迟也可能有几百毫秒。业务层必须有处理延迟的策略。
3. **强制走主库要最小化范围**。只在确实需要强一致性的场景使用，否则就白做读写分离了。我们通过 `@MasterRoute` 注解标记，审计时一搜就能看到哪些方法走了主库。
4. **延迟时间窗口方案很实用**。用户写后5秒内读走主库，5秒后走从库。这5秒的主库额外压力完全可以接受，用户体验也保障了。
5. **从库至少两个**。一个从库挂了还有另一个顶上。我们用 ROUND_ROBIN 轮询，自动负载均衡。
6. **监控主从延迟是运维第一要务**。`SHOW SLAVE STATUS` 看 `Seconds_Behind_Master`，配合 Prometheus 持续监控。延迟超过10秒就告警。
