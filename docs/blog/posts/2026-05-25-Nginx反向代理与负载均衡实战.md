## 前言

Nginx 可能是 Java 后端开发接触最多的基础设施组件了。不管是做反向代理、负载均衡还是静态资源服务，几乎每个项目都离不开它。但"会用"和"用好"之间的差距，往往体现在出问题时才暴露出来。

这篇文章分享我在三个不同项目中遇到的 Nginx 实际问题和解决方案。

## 问题场景

### 场景一：发布期间用户报 502

我们的用户服务有 4 台机器，用 Nginx 做负载均衡。每次发布（逐台滚动重启）时，总有少量用户报 502 Bad Gateway。运维说"发布时有个别请求失败是正常的"，但产品不接受。

### 场景二：某个接口偶尔超时 30 秒

一个导入功能的接口偶尔需要 20~30 秒处理大量数据。正常情况下 Nginx 默认 60 秒超时没问题，但实际测试发现经常 30 秒就断了。

### 场景三：ip_hash 导致流量不均

用了 ip_hash 做会话保持，但客户大多在同一个 NAT 后面，导致某台机器被打爆。

## 分析思路

Nginx 负载均衡的核心配置就两个部分：`upstream`（定义后端服务器组）和 `proxy_pass`（转发请求）。先把最常用的配置梳理清楚，再解决上述问题。

## 具体实现

### 基础 upstream 配置

```nginx
# /etc/nginx/conf.d/upstream.conf

upstream user_service {
    # 负载均衡策略选择（见下文分析）
    least_conn;

    server 10.0.1.10:8080 weight=3 max_fails=3 fail_timeout=30s;
    server 10.0.1.11:8080 weight=3 max_fails=3 fail_timeout=30s;
    server 10.0.1.12:8080 weight=2 max_fails=3 fail_timeout=30s;
    server 10.0.1.13:8080 weight=2 max_fails=3 fail_timeout=30s;

    # 长连接复用：后端是 HTTP/1.1 时开启
    keepalive 32;
    keepalive_requests 1000;
    keepalive_timeout 60s;
}
```

### 负载均衡策略选择

| 策略 | 适用场景 | 坑 |
|------|----------|----|
| 轮询（默认） | 后端机器配置相同 | 不考虑服务器当前负载 |
| weight（加权） | 机器配置不同 | 权重需要手动调整 |
| ip_hash | 需要会话保持 | NAT 后面的用户会被分到同一台 |
| least_conn | 后端处理时间差异大 | 需要 Nginx 1.3.1+ |

**选型建议**：大多数场景用 `least_conn` 就对了。它会把请求发给当前连接数最少的后端，天然适配后端处理时间不均的场景。需要会话保持的话优先考虑用 Redis 共享 session，而不是依赖 ip_hash。

### 场景一的解决方案：优雅发布

502 的原因很简单：Nginx 把请求发到了正在重启的机器上，连接被拒绝。

```nginx
# 方案：配合发布系统，先摘流量再重启
upstream user_service {
    server 10.0.1.10:8080;
    server 10.0.1.11:8080;
    # 发布时通过 API 或脚本将目标机器 weight 设为 0
    # 或者直接从 upstream 中移除
}
```

更优雅的方案是用 Nginx 的 `max_fails` 和 `fail_timeout`：

```nginx
upstream user_service {
    # max_fails=3：连续失败3次后标记为不可用
    # fail_timeout=30s：标记不可用后30秒内不再转发，30秒后重新尝试
    server 10.0.1.10:8080 max_fails=3 fail_timeout=30s;
    server 10.0.1.11:8080 max_fails=3 fail_timeout=30s;
}
```

但这还不够。`max_fails` 检测的是 Nginx 自己的被动健康检查，响应慢。更推荐主动健康检查（Nginx Plus 有内置功能，开源版可以用第三方模块 `nginx_upstream_check_module`）：

```nginx
upstream user_service {
    server 10.0.1.10:8080;
    server 10.0.1.11:8080;

    check interval=3000 rise=2 fall=3 timeout=1000 type=http;
    check_http_send "GET /actuator/health HTTP/1.0\r\n\r\n";
    check_http_expect_alive http_2xx http_3xx;
}
```

最终我们采用的方案是：**发布系统与 Nginx 联动，发布前先将目标机器 weight 设为 0 并等待已有连接处理完成，发布后再恢复**。

```bash
#!/bin/bash
# 发布脚本中的关键步骤
TARGET_SERVER="10.0.1.10:8080"

# 1. 摘流量
curl -X POST http://nginx-admin-api/upstream/user_service/server \
  -d '{"server": "'$TARGET_SERVER'", "weight": 0}'

# 2. 等待已有连接完成（根据最长接口响应时间设定）
sleep 15

# 3. 发布新版本
ssh $TARGET_SERVER "systemctl restart user-service"

# 4. 健康检查通过后恢复流量
sleep 10
curl -X POST http://nginx-admin-api/upstream/user_service/server \
  -d '{"server": "'$TARGET_SERVER'", "weight": 3}'
```

### 场景二的解决方案：超时配置

排查发现 Nginx 的 `proxy_read_timeout` 默认是 60 秒，但实际被一个中间层的配置覆盖成了 30 秒。

```nginx
location /api/import {
    proxy_pass http://user_service;

    # 代理超时配置
    proxy_connect_timeout 5s;     # 连接后端的超时
    proxy_send_timeout 60s;       # 发送请求到后端的超时
    proxy_read_timeout 120s;      # 等待后端响应的超时（重要！）

    # 如果后端处理慢，可以先返回一个中间结果
    # proxy_buffering off;        # 流式代理，不缓冲
}
```

**关键教训**：`proxy_read_timeout` 要根据业务接口的最大耗时来设，不能用默认值。特别是文件导入、报表生成这类接口。

### 场景三的解决方案：放弃 ip_hash

客户的出口 IP 集中在几个 NAT 地址后，ip_hash 把这些流量全部打到同一台机器。

```nginx
# 修改前：ip_hash 导致流量不均
upstream user_service {
    ip_hash;
    server 10.0.1.10:8080;
    server 10.0.1.11:8080;
}

# 修改后：用 cookie 做会话保持，配合 least_conn
upstream user_service {
    least_conn;
    server 10.0.1.10:8080;
    server 10.0.1.11:8080;
}

# 会话通过 Redis 共享，Nginx 不再负责会话粘性
```

## 完整的 location 配置模板

把上面的经验汇总，这是我常用的生产配置模板：

```nginx
server {
    listen 80;
    server_name api.example.com;

    # 请求体大小限制
    client_max_body_size 50m;

    location /api/ {
        proxy_pass http://user_service;

        # 必须设置的 Header
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # 超时配置
        proxy_connect_timeout 5s;
        proxy_read_timeout 60s;
        proxy_send_timeout 30s;

        # 长连接复用（配合 upstream 中的 keepalive）
        proxy_http_version 1.1;
        proxy_set_header Connection "";

        # 错误处理：后端一台失败自动切下一台
        proxy_next_upstream error timeout http_502 http_503;
        proxy_next_upstream_tries 2;
        proxy_next_upstream_timeout 10s;

        # 缓冲配置
        proxy_buffer_size 8k;
        proxy_buffers 16 8k;
    }
}
```

## 踩坑总结

1. **proxy_http_version 必须设为 1.1**。默认是 1.0，不支持长连接。配了 `keepalive` 但没设版本号等于没配。

2. **proxy_set_header Connection ""**。配合 keepalive 使用，去掉客户端传过来的 `Connection: close`，让 Nginx 与后端复用连接。

3. **proxy_next_upstream 要慎用**。默认只在连接错误和超时时重试。不要加上 `http_500`，否则 POST 请求会被重复提交。

4. **upstream 中的 server 不能带 URI**。`server 10.0.1.10:8080/api/` 是错的，URI 应该写在 `proxy_pass` 中。

5. **发布时一定要先摘流量**。Nginx 的被动健康检查有延迟，靠它扛不住滚动发布。

6. **别忘了日志**。在 location 中加上 `access_log` 和 `error_log`，出问题时有据可查：

```nginx
access_log /var/log/nginx/user_service_access.log combined buffer=32k flush=5s;
error_log /var/log/nginx/user_service_error.log warn;
```

这些配置看起来不起眼，但每一个都是真实踩过的坑。特别是 Nginx 的超时和连接复用配置，直接决定了系统的稳定性和性能。
