## 前言

凌晨三点被告警叫醒，原因是主站 HTTPS 证书过期了。这种事故在行业里屡见不鲜，每家公司都有类似的故事。我们那次是因为证书自动续期的 cron job 配置错误，续期脚本把新证书生成到了错误的目录。

痛定思痛，我们花了一周时间把证书管理、TLS 配置全面梳理了一遍。这篇文章就是这次梳理的成果。

## 问题场景

除了证书过期这种低级错误，HTTPS 相关的问题其实还不少：

- **TLS 握手慢**：外网用户首次访问延迟高，TCP 连接建好后 TLS 握手还要 1~2 个 RTT
- **证书链不完整**：部分浏览器或旧版 Android 报"不安全"，抓包发现是中间证书缺失
- **TLS 版本不兼容**：升级到 TLS 1.3 后，某些老客户端连不上

## 具体实现

### 一、Let's Encrypt 证书自动续期

我们用 certbot 管理 Let's Encrypt 证书。正确的自动续期配置如下：

```bash
# 安装 certbot
apt install certbot python3-certbot-nginx

# 首次申请证书
certbot --nginx -d api.example.com -d www.example.com

# 测试续期
certbot renew --dry-run

# 自动续期 cron job（每天凌晨 3 点检查）
# /etc/cron.d/certbot-renew
0 3 * * * root certbot renew --quiet --deploy-hook "systemctl reload nginx"
```

**踩坑教训**：`--deploy-hook` 很关键。证书续期后必须 reload Nginx 才能生效。我们之前只配了 `renew`，没配 `deploy-hook`，导致新证书生成了但 Nginx 还在用旧证书。

更安全的做法是写个检查脚本：

```bash
#!/bin/bash
# /opt/scripts/check-cert.sh
DOMAIN="api.example.com"
EXPIRE_DATE=$(echo | openssl s_client -servername $DOMAIN -connect $DOMAIN:443 2>/dev/null \
  | openssl x509 -noout -enddate | cut -d= -f2)
EXPIRE_TS=$(date -d "$EXPIRE_DATE" +%s)
NOW_TS=$(date +%s)
DAYS_LEFT=$(( ($EXPIRE_TS - $NOW_TS) / 86400 ))

if [ $DAYS_LEFT -lt 7 ]; then
    echo "【证书告警】${DOMAIN} 证书将在 ${DAYS_LEFT} 天后过期" | \
      mail -s "HTTPS证书即将过期" ops@example.com
fi

if [ $DAYS_LEFT -lt 0 ]; then
    echo "【证书已过期】${DOMAIN}" | \
      mail -s "HTTPS证书已过期！" ops@example.com
fi
```

配合 Prometheus + Alertmanager 监控：

```yaml
# Prometheus alert rule
groups:
  - name: ssl
    rules:
      - alert: SSLCertExpiringSoon
        expr: probe_ssl_earliest_cert_expiry - time() < 86400 * 7
        for: 1h
        labels:
          severity: warning
        annotations:
          summary: "SSL证书将在7天内过期"
```

### 二、TLS 版本选择：1.2 vs 1.3

**我们的选择：同时支持 TLS 1.2 和 TLS 1.3**。

原因很实际：虽然 TLS 1.3 更快更安全，但还有一部分老客户端（Android 4.x、旧版 Java 8 等）不支持。直接只开 TLS 1.3 会导致这些用户无法访问。

Nginx 配置：

```nginx
server {
    listen 443 ssl http2;
    server_name api.example.com;

    # 证书
    ssl_certificate /etc/letsencrypt/live/api.example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/api.example.com/privkey.pem;

    # TLS 版本：同时支持 1.2 和 1.3
    ssl_protocols TLSv1.2 TLSv1.3;

    # TLS 1.2 密码套件（去除弱密码）
    ssl_ciphers 'ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384';
    ssl_prefer_server_ciphers on;

    # TLS 1.3 的 0-RTT（需谨慎开启）
    # ssl_early_data on;  # 有重放攻击风险，支付类接口不要开

    # 启用 OCSP Stapling
    ssl_stapling on;
    ssl_stapling_verify on;
    ssl_trusted_certificate /etc/letsencrypt/live/api.example.com/chain.pem;
    resolver 8.8.8.8 8.8.4.4 valid=300s;
    resolver_timeout 5s;
}
```

**TLS 1.2 vs 1.3 的性能差异**：

| 指标 | TLS 1.2 | TLS 1.3 |
|------|---------|---------|
| 握手 RTT | 2 RTT | 1 RTT |
| 支持 0-RTT | 否 | 是 |
| 密码套件数量 | 多（有安全隐患的） | 少（只有安全的） |
| 浏览器兼容性 | 99%+ | 95%+（不含 IE） |

### 三、TLS Session 复用

TLS 握手的性能瓶颈在于密钥协商的计算开销和网络 RTT。Session 复用可以让后续连接跳过完整握手：

```nginx
# Session 缓存（推荐）
ssl_session_cache shared:SSL:50m;
ssl_session_timeout 1d;

# Session Ticket（配合 Session Cache 使用）
ssl_session_tickets on;
ssl_session_ticket_key /etc/nginx/ssl/ticket.key;
```

**踩坑点**：Session Ticket 的密钥需要定期轮换。如果不轮换，就失去了前向保密性（Forward Secrecy）。

```bash
# 生成并轮换 Session Ticket 密钥
#!/bin/bash
TICKET_KEY="/etc/nginx/ssl/ticket.key"
# 生成新密钥
openssl rand 80 > ${TICKET_KEY}.new
mv ${TICKET_KEY}.new $TICKET_KEY
nginx -s reload
```

我们在 cron 中每 24 小时轮换一次。

### 四、证书链问题排查

证书链不完整是最隐蔽的问题。有些浏览器（如 Chrome）会自动下载缺失的中间证书，所以开发者自己测不出问题。但旧版 Android、某些爬虫、Java HttpClient 会直接报错。

**排查方法**：

```bash
# 检查证书链完整性
openssl s_client -connect api.example.com:443 -servername api.example.com </dev/null 2>/dev/null | \
  openssl x509 -noout -text | grep -A2 "Issuer"

# 查看完整证书链
openssl s_client -connect api.example.com:443 -servername api.example.com -showcerts </dev/null

# 在线检测（最全面）
# https://www.ssllabs.com/ssltest/
```

**常见的证书链错误**：

1. **只配了域名证书，没配中间证书**：
   - 错误：`ssl_certificate /path/to/domain.crt;`
   - 正确：`ssl_certificate /path/to/fullchain.pem;`（Let's Encrypt 的 fullchain.pem 已包含中间证书）

2. **证书顺序错误**：fullchain.pem 中证书的顺序应该是：域名证书 → 中间证书 → 根证书（可选）

### 五、Java 客户端的 TLS 配置

服务间调用如果用 HTTPS，Java 客户端也需要正确配置：

```java
@Configuration
public class HttpsConfig {

    @Bean
    public RestTemplate restTemplate() throws Exception {
        // 信任自签名证书的场景（仅限内网测试环境）
        SSLContext sslContext = SSLContextBuilder.create()
                .loadTrustMaterial(null, (chain, authType) -> true)
                .build();

        // 生产环境：信任系统默认的 CA 证书
        SSLContext prodSslContext = SSLContext.getInstance("TLSv1.2");
        prodSslContext.init(null, null, null);

        HttpClient httpClient = HttpClientBuilder.create()
                .setSSLContext(prodSslContext)
                // 只允许 TLS 1.2 和 1.3
                .setSSLHostnameVerifier(NoopHostnameVerifier.INSTANCE) // 仅测试环境
                .build();

        return new RestTemplate(new HttpComponentsClientHttpRequestFactory(httpClient));
    }
}
```

**注意**：`NoopHostnameVerifier` 只能在测试环境用，生产环境必须启用主机名验证，否则有中间人攻击风险。

## 踩坑总结

1. **证书续期必须有 deploy-hook**。续期不 reload 等于没续期。

2. **监控证书过期时间**。用 Blackbox Exporter 定期探测，设置 7 天预警。

3. **用 fullchain.pem 而不是 domain.crt**。`fullchain.pem` 包含完整的证书链，`domain.crt` 只有域名证书。

4. **TLS 1.2 + 1.3 并存**。不要激进地只开 1.3，除非你能控制所有客户端。

5. **开启 OCSP Stapling**。不开启的话，客户端需要单独查询 OCSP 服务器，增加延迟。

6. **Session 复用要开**。对高 QPS 站点，Session Cache + Session Ticket 能显著降低 CPU 开销。

7. **定期轮换 Session Ticket 密钥**。保持前向保密性。

8. **生产环境不要用自签名证书**。Let's Encrypt 免费又自动续期，没有理由用自签名。
