## 先泼一盆冷水：不是所有系统都该微服务

我在一家 150 人的公司做后端架构，系统从单体到 20+ 个微服务走了两年。回头看，至少有 3 个服务不该拆，纯属跟风。这篇文章把我们踩过的坑和总结的方法论写出来，希望对同行有参考价值。

## 什么时候该拆，什么时候不该拆

### 该拆的信号

**团队协作冲突严重**。我们的订单和库存原来在一个服务里，订单团队和库存团队经常因为同一个 JAR 改代码冲突。一个发版另一个得等着，这才是拆的真正理由——**解耦团队，不是解耦代码**。

**某个模块的伸缩需求完全不同**。我们的商品搜索（Elasticsearch 查询）需要大内存、高 QPS，而商品管理后台是低频操作。混在一起时，搜索把内存吃满了管理后台也跟着挂。

**故障爆炸半径太大**。一个支付回调处理的 bug 导致整个电商服务 OOM，商品浏览、订单查询全部不可用。拆开之后，支付服务挂了至少商品服务还能正常浏览。

### 不该拆的信号

**团队只有 3-5 个后端开发**。5 个人维护 20 个服务，光运维就占一半时间，还有多少精力写业务？我们当时有段时间每个开发至少维护 3 个服务，疲于奔命。

**业务还在快速变化**。拆完微服务后改一个需求要改 3 个服务再联调，开发效率直接砍半。如果业务逻辑还没稳定，先用模块化单体扛着。

**没有 DevOps 基础**。微服务不是写完代码就完了，CI/CD、日志聚合、链路追踪、配置中心、服务注册，这些基础设施缺一不可。我们有个子系统没有接入统一日志平台，出问题排查全靠 `ssh + grep`。

## 按业务域拆 vs 按团队拆

我们一开始按技术层拆：API 网关服务、业务逻辑服务、数据访问服务。**大错特错**。改一个需求三个服务都得动，联调地狱。

后来改按业务域拆：

```
用户域    -> user-service（注册、登录、权限）
商品域    -> product-service（商品管理、分类、搜索）
订单域    -> order-service（下单、支付、退款）
库存域    -> inventory-service（库存扣减、预占、回补）
营销域    -> promotion-service（优惠券、满减、活动）
```

拆分原则：**一个业务域对应一个团队，一个团队最多负责 3 个服务**。超过 3 个就说明要么服务太碎，要么团队要扩充。

```java
// 订单服务调用库存服务 - 通过 FeignClient
@FeignClient(name = "inventory-service",
    fallbackFactory = InventoryFallbackFactory.class)
public interface InventoryClient {

    @PostMapping("/api/inventory/deduct")
    Result<InventoryDTO> deduct(@RequestBody DeductRequest request);
}

// 降级工厂 - 库存服务不可用时的兜底逻辑
@Component
@Slf4j
public class InventoryFallbackFactory
        implements FallbackFactory<InventoryClient> {

    @Override
    public InventoryClient create(Throwable cause) {
        log.error("【库存服务降级】", cause);
        return request -> Result.fail(503, "库存服务暂时不可用，请稍后重试");
    }
}
```

## 拆分过程中的数据一致性

这是微服务拆分最头疼的问题。我们踩过的坑：

### 场景一：分布式事务

下单时需要同时扣减库存和创建订单。原来在单体里一个 `@Transactional` 搞定，拆开后不能跨库事务了。

我们最终采用 **TCC 模式（Try-Confirm-Cancel）**，用 Seata 实现：

```java
// 订单服务 - Try 阶段：创建待确认订单
@TwoPhaseBusinessAction(name = "createOrder",
    commitMethod = "confirm", rollbackMethod = "cancel")
public boolean tryCreateOrder(BusinessActionContext context,
        @BusinessActionContextParameter(paramName = "orderDTO") OrderDTO orderDTO) {
    // 创建 PENDING 状态的订单
    orderDTO.setStatus(OrderStatus.PENDING);
    orderMapper.insert(orderDTO);
    return true;
}

// Confirm 阶段：确认订单
public boolean confirm(BusinessActionContext context) {
    Long orderId = (Long) context.getActionContext("orderId");
    orderMapper.updateStatus(orderId, OrderStatus.CONFIRMED);
    return true;
}

// Cancel 阶段：取消订单
public boolean cancel(BusinessActionContext context) {
    Long orderId = (Long) context.getActionContext("orderId");
    orderMapper.updateStatus(orderId, OrderStatus.CANCELLED);
    return true;
}
```

**踩坑**：Seata 的 AT 模式看着简单，但它的原理是拦截 SQL 自动回滚，对性能影响很大（每条 SQL 多了前后镜像的读写）。我们测试下来，引入 AT 模式后事务吞吐量下降了约 30%。最终只有资金相关的操作用 TCC，其他场景用最终一致性。

### 场景二：最终一致性

库存扣减成功但订单创建失败这种小概率事件，我们用 **本地消息表 + 定时补偿** 实现最终一致性：

```java
@Transactional
public void createOrder(OrderDTO orderDTO) {
    // 1. 创建订单
    orderMapper.insert(orderDTO);

    // 2. 写本地消息表（同一个事务）
    OutboxMessage message = new OutboxMessage();
    message.setTopic("inventory-deduct");
    message.setPayload(JSON.toJSONString(orderDTO.getDeductRequest()));
    message.setStatus(MessageStatus.PENDING);
    outboxMapper.insert(message);
}

// 定时任务 - 每 5 秒扫描未发送的消息
@Scheduled(fixedDelay = 5000)
public void processOutbox() {
    List<OutboxMessage> messages = outboxMapper.findByStatus(
        MessageStatus.PENDING, 100);
    for (OutboxMessage msg : messages) {
        try {
            rocketMQTemplate.sendOneWay(msg.getTopic(), msg.getPayload());
            outboxMapper.updateStatus(msg.getId(), MessageStatus.SENT);
        } catch (Exception e) {
            log.error("【消息发送失败】id={}", msg.getId(), e);
            // 下次定时任务会重试
        }
    }
}
```

### 场景三：跨服务查询

拆分后经常需要跨服务查询数据，比如订单列表要显示商品名称和库存状态。

**错误做法**：循环调用 FeignClient，一个列表请求触发 100 次 RPC。

**正确做法**：

1. 高频查询的数据做冗余，订单表里直接存商品名称和价格快照（下单时的值）
2. 需要实时数据的用批量接口，一次 RPC 拉一批

```java
// 不要这样写
for (Order order : orderList) {
    ProductVO product = productClient.getById(order.getProductId());
    order.setProductName(product.getName());
}

// 要这样写
List<Long> productIds = orderList.stream()
    .map(Order::getProductId).collect(Collectors.toList());
Map<Long, ProductVO> productMap = productClient.batchGetByIds(productIds);
orderList.forEach(order ->
    order.setProductName(productMap.get(order.getProductId()).getName()));
```

## 拆分过程的建议

1. **渐进式拆分**，不要一步到位。先拆最痛的（团队冲突最大、伸缩需求差异最大），验证基础设施成熟后再拆其他的。

2. **先模块化，再服务化**。在单体内先把模块边界划清楚（独立包、独立数据源），确认模块间没有乱七八糟的依赖后，再物理拆分。

3. **API 先行**。拆分前先定义好服务间的 API 契约，用 OpenAPI/Swagger 管理，避免边拆边改接口。

4. **保留回退能力**。拆出来的服务如果发现不稳定，要有能力快速合回去。我们的做法是单体里保留一份代码但不启用，灰度切流。

微服务不是银弹，但也不是洪水猛兽。关键是在对的时间、以对的方式做对的事。
