## 环境管理是微服务的隐形成本

我们的系统有 4 个环境：dev（开发）、test（测试）、staging（预发）、prod（生产）。15 个微服务，每个服务 4 套环境配置，一共 60 份配置要管理。

最离谱的一次事故：staging 环境的配置被误推到了 prod，订单服务连到了测试库，导致测试数据混入真实订单。从那之后我们痛定思痛，严格隔离多环境配置。

## 多环境配置隔离方案

### 方案一：Git 分支隔离（不推荐）

```
config-repo/
  ├── dev/
  │   ├── order-service.yml
  │   └── product-service.yml
  ├── staging/
  ├── prod/
  └── common/
```

问题：配置和代码用同一个 Git 仓库时，分支管理混乱。用独立仓库的话，改一个配置要走 Git 流程，太重了。

### 方案二：Nacos Namespace 隔离（推荐）

```
Namespace: dev    (ID: xxx-dev-xxx)
Namespace: staging (ID: xxx-staging-xxx)
Namespace: prod   (ID: xxx-prod-xxx)
```

不同环境的服务注册到不同的 Namespace，天然隔离。通过环境变量切换：

```yaml
spring:
  cloud:
    nacos:
      config:
        namespace: ${NACOS_NAMESPACE}
      discovery:
        namespace: ${NACOS_NAMESPACE}
```

K8s ConfigMap 按环境区分：

```yaml
# configmap-dev.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: nacos-config
  namespace: dev
data:
  NACOS_NAMESPACE: "dev-namespace-id"
  NACOS_ADDR: "nacos.dev.svc:8848"

# configmap-prod.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: nacos-config
  namespace: prod
data:
  NACOS_NAMESPACE: "prod-namespace-id"
  NACOS_ADDR: "nacos.prod.svc:8848"
```

**踩坑**：K8s 的 `namespace` 和 Nacos 的 `namespace` 是两个概念，不要搞混了。我们在 K8s 的 `dev` namespace 里配错 Nacos namespace，服务注册到了 prod，吓出一身冷汗。

### 方案三：环境变量 + 配置模板

敏感配置（数据库密码、API Key）不放 Nacos，用 K8s Secret：

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: order-service-secret
  namespace: prod
type: Opaque
stringData:
  DB_PASSWORD: "real-password-here"
  REDIS_PASSWORD: "real-password-here"
```

优先级：环境变量 > Nacos 配置 > 本地 application.yml

## 蓝绿部署

蓝绿部署的核心思想：同时维护两套完全相同的生产环境（蓝和绿），切换流量时只需要改负载均衡的指向。

### 实现方案（基于 K8s）

```yaml
# 蓝色版本
apiVersion: apps/v1
kind: Deployment
metadata:
  name: order-service-blue
  labels:
    app: order-service
    version: blue
spec:
  replicas: 3
  selector:
    matchLabels:
      app: order-service
      version: blue
  template:
    metadata:
      labels:
        app: order-service
        version: blue
    spec:
      containers:
        - name: order-service
          image: registry.example.com/order-service:v1.0
          ports:
            - containerPort: 8080

---
# Service 默认指向蓝色
apiVersion: v1
kind: Service
metadata:
  name: order-service
spec:
  selector:
    app: order-service
    version: blue  # 切到 green 只需要改这一行
  ports:
    - port: 80
      targetPort: 8080
```

**切换流程**：

1. 部署 green 版本，等待就绪
2. 验证 green 版本的健康检查通过
3. 修改 Service selector 指向 green
4. 观察一段时间无异常后，删除 blue 版本

**踩坑**：蓝绿部署的资源消耗是正常情况的 2 倍。我们有次做蓝绿切换，K8s 集群资源不够，green 版本启动后 OOM。后来加了 `resource.limits` 和集群自动扩缩容。

## 金丝雀发布

金丝雀发布是把少量流量切到新版本，观察指标无异常后逐步扩大比例。

### 基于 Nacos 元数据实现

```yaml
# 新版本实例注册时带 canary 标签
spring:
  cloud:
    nacos:
      discovery:
        metadata:
          version: v1.1
          canary: "true"
```

```java
/**
 * 自定义负载均衡规则：金丝雀路由
 * 默认 90% 流量走稳定版本，10% 走金丝雀版本
 */
@Component
@Slf4j
public class CanaryLoadBalancer implements ReactorServiceInstanceLoadBalancer {

    private final String serviceId;
    private final ObjectProvider<ServiceInstanceListSupplier> supplierProvider;
    private final AtomicInteger position = new AtomicInteger(0);
    // 金丝雀流量比例：10%
    private static final double CANARY_RATIO = 0.1;

    @Override
    public Mono<Response<ServiceInstance>> choose(Request request) {
        return supplierProvider.getIfAvailable()
            .next()
            .map(instances -> {
                // 分离稳定版本和金丝雀版本
                List<ServiceInstance> canary = instances.stream()
                    .filter(i -> "true".equals(i.getMetadata().get("canary")))
                    .collect(Collectors.toList());
                List<ServiceInstance> stable = instances.stream()
                    .filter(i -> !"true".equals(i.getMetadata().get("canary")))
                    .collect(Collectors.toList());

                // 按比例分配
                boolean useCanary = Math.random() < CANARY_RATIO;
                List<ServiceInstance> target = (useCanary && !canary.isEmpty())
                    ? canary : stable;

                if (target.isEmpty()) {
                    target = instances; // 兜底
                }

                int pos = Math.abs(position.incrementAndGet());
                ServiceInstance instance = target.get(pos % target.size());
                log.debug("【金丝雀路由】{} -> {}",
                    useCanary ? "canary" : "stable", instance.getInstanceId());
                return new DefaultResponse(instance);
            });
    }
}
```

### 基于 Header 的精确路由

更精确的方式是根据请求头决定路由：

```java
/**
 * 从请求头提取金丝雀标识，精确路由到金丝雀实例
 * 用于内部测试人员或特定用户灰度
 */
@Component
public class CanaryRouterFilter implements GlobalFilter, Ordered {

    private static final String CANARY_HEADER = "X-Canary";

    @Override
    public Mono<Void> filter(ServerWebExchange exchange,
                             GatewayFilterChain chain) {
        String canaryHeader = exchange.getRequest()
            .getHeaders().getFirst(CANARY_HEADER);

        if ("true".equals(canaryHeader)) {
            // 在 RequestContext 中设置路由偏好
            exchange.getAttributes().put("CANARY_FLAG", true);
        }
        return chain.filter(exchange);
    }

    @Override
    public int getOrder() {
        return -100;
    }
}
```

## Feature Flag 简化实现

Feature Flag 让你可以在不发版的情况下开关功能。中小厂不需要 LaunchDarkly 这样的商业产品，自己实现一个简单的就够用。

```java
/**
 * 基于 Nacos 配置的 Feature Flag
 * 修改 Nacos 配置后实时生效，无需重启
 */
@Component
@Data
@ConfigurationProperties(prefix = "feature")
public class FeatureFlags {
    /** 是否开启新下单流程 */
    private boolean newOrderFlow = false;
    /** 是否开启商品推荐 */
    private boolean productRecommend = false;
    /** 是否开启积分系统 */
    private boolean pointSystem = false;
}
```

Nacos 中的配置（可以实时修改）：

```yaml
feature:
  new-order-flow: false
  product-recommend: true
  point-system: false
```

使用方式：

```java
@Service
@RequiredArgsConstructor
public class OrderService {
    private final FeatureFlags featureFlags;

    public OrderVO createOrder(CreateOrderRequest request) {
        // 根据 Feature Flag 走不同的逻辑
        if (featureFlags.isNewOrderFlow()) {
            return newOrderFlow(request);
        }
        return oldOrderFlow(request);
    }
}
```

### 带百分比灰度的 Feature Flag

```java
@Data
@ConfigurationProperties(prefix = "feature")
public class FeatureFlags {
    /** 新下单流程的灰度百分比（0-100） */
    private int newOrderFlowPercentage = 0;
}

// 使用
@Service
@RequiredArgsConstructor
public class OrderService {
    private final FeatureFlags featureFlags;

    public OrderVO createOrder(CreateOrderRequest request, Long userId) {
        // 基于用户 ID 的一致性灰度（同一用户始终走相同逻辑）
        boolean useNewFlow = featureFlags.getNewOrderFlowPercentage() > 0
            && (userId.hashCode() % 100) < featureFlags.getNewOrderFlowPercentage();

        return useNewFlow ? newOrderFlow(request) : oldOrderFlow(request);
    }
}
```

**踩坑**：Feature Flag 的百分比灰度一定要基于用户 ID（或其他稳定标识），不能用 `Math.random()`。否则同一个用户刷新页面时功能忽开忽关，体验极差。

## 发布策略选择建议

| 场景 | 推荐策略 | 理由 |
|------|---------|------|
| 日常迭代 | 滚动更新 | K8s 默认策略，简单可靠 |
| 重大版本 | 蓝绿部署 | 可以秒级回滚 |
| 不确定影响 | 金丝雀发布 | 控制爆炸半径 |
| 功能开关 | Feature Flag | 灵活控制，不需要发版 |

中小厂不需要所有策略都上，**滚动更新 + Feature Flag** 能覆盖 90% 的场景。金丝雀和蓝绿部署留到真正需要的时候再引入。
