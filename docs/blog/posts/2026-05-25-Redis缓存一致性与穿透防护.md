## 问题场景

我们的商品详情接口，QPS从日常2000突然飙到8000（大促预热），Redis命中率从99.5%暴跌到60%，大量请求穿透到MySQL，DB连接池直接打满，服务雪崩。

事后复盘发现三个问题叠加：

1. 一批缓存恰好同时过期（缓存雪崩）
2. 有恶意爬虫在遍历不存在的商品ID（缓存穿透）
3. 一个爆款商品的Key被打爆（热点Key问题）

## 分析思路

### 1. 缓存一致性：Cache Aside 的细节

我们用的是最经典的 **Cache Aside Pattern**（旁路缓存模式），但实际执行中有几个容易忽略的细节：

```java
// 读操作
public Product getProduct(Long productId) {
    // 1. 先查缓存
    String cacheKey = "product:" + productId;
    String json = redisTemplate.opsForValue().get(cacheKey);
    if (json != null) {
        if ("NULL".equals(json)) {
            return null; // 缓存的空值，防穿透
        }
        return JSON.parseObject(json, Product.class);
    }

    // 2. 缓存未命中，查数据库
    Product product = productMapper.selectById(productId);

    // 3. 写入缓存（包含空值）
    if (product == null) {
        redisTemplate.opsForValue().set(cacheKey, "NULL", 2, TimeUnit.MINUTES);
    } else {
        // 过期时间加随机值，防止雪崩
        long ttl = 30 + ThreadLocalRandom.current().nextInt(10);
        redisTemplate.opsForValue().set(cacheKey, JSON.toJSONString(product), ttl, TimeUnit.MINUTES);
    }
    return product;
}
```

**为什么不先更新缓存再更新数据库？**

我们最开始就是先更新缓存再更新DB，结果遇到了并发问题：

```
线程A 更新缓存 price=90
线程B 更新缓存 price=80
线程B 更新数据库 price=80
线程A 更新数据库 price=90
// 缓存里是80，数据库里是90，不一致！
```

**为什么不先更新数据库再更新缓存？**

理论上更安全，但也有问题：如果更新DB成功但更新缓存失败（网络抖动），缓存就是脏数据。

**我们的最终方案**：写操作更新DB后，直接**删除缓存**而不是更新缓存。读操作会自动回填。这是 Cache Aside 的标准做法。

```java
// 写操作
@Transactional
public void updateProductPrice(Long productId, BigDecimal newPrice) {
    // 1. 先更新数据库
    productMapper.updatePrice(productId, newPrice);

    // 2. 再删除缓存（不是更新）
    redisTemplate.delete("product:" + productId);
}
```

### 2. 缓存穿透：布隆过滤器

缓存穿透是指查询一个根本不存在的数据，缓存和数据库都查不到，每次请求都打到DB。恶意用户用大量不存在的ID遍历就能把DB打垮。

**方案一：缓存空值**（简单但内存浪费）

上面代码中已经实现了：查不到数据时缓存 `"NULL"` 字符串，短过期时间（2分钟）。

**方案二：布隆过滤器**（推荐用于数据量大的场景）

我们用 **Redisson 的布隆过滤器**，在服务启动时加载所有合法的商品ID：

```java
@Component
public class BloomFilterInitializer {

    @Autowired
    private RedissonClient redissonClient;

    @PostConstruct
    public void init() {
        RBloomFilter<Long> bloomFilter = redissonClient.getBloomFilter("productBloomFilter");
        // 预估1000万个商品，误判率0.01%
        bloomFilter.tryInit(10_000_000L, 0.0001);

        // 分批加载商品ID
        int page = 0;
        int pageSize = 10000;
        while (true) {
            List<Long> ids = productMapper.selectIds(page * pageSize, pageSize);
            if (CollectionUtils.isEmpty(ids)) break;
            ids.forEach(bloomFilter::add);
            page++;
        }
    }
}

// 使用布隆过滤器拦截
public Product getProductWithBloom(Long productId) {
    RBloomFilter<Long> bloomFilter = redissonClient.getBloomFilter("productBloomFilter");

    // 布隆过滤器说不存在，那一定不存在
    if (!bloomFilter.contains(productId)) {
        return null;
    }

    // 布隆过滤器说存在，可能不存在（误判），继续查缓存和DB
    return getProduct(productId);
}
```

**布隆过滤器的坑**：不支持删除。如果商品下架了，布隆过滤器里还是存在。我们的解决方案是下架商品标记为"已下架"状态，不物理删除，配合布隆过滤器使用。

### 3. 缓存雪崩的防御

**过期时间加随机值**是第一步（上面代码已实现），但不够。

我们还加了两层保险：

**保险一：缓存预热**

```java
@Scheduled(cron = "0 0 2 * * ?")  // 凌晨2点执行
public void cacheWarmUp() {
    log.info("【缓存预热开始】");
    // 查询热门商品（最近7天销量TOP 1000）
    List<Product> hotProducts = productMapper.selectHotProducts(1000);
    for (Product product : hotProducts) {
        String key = "product:" + product.getId();
        long ttl = 60 + ThreadLocalRandom.current().nextInt(30);
        redisTemplate.opsForValue().set(key, JSON.toJSONString(product), ttl, TimeUnit.MINUTES);
    }
    log.info("【缓存预热完成】数量: {}", hotProducts.size());
}
```

**保险二：分布式锁防止缓存击穿**

热点Key过期的瞬间，大量并发请求同时查DB。用分布式锁保证只有一个线程去查DB并回填缓存：

```java
public Product getProductWithLock(Long productId) {
    String cacheKey = "product:" + productId;
    String json = redisTemplate.opsForValue().get(cacheKey);
    if (json != null) {
        return "NULL".equals(json) ? null : JSON.parseObject(json, Product.class);
    }

    String lockKey = "lock:product:" + productId;
    RLock lock = redissonClient.getLock(lockKey);
    try {
        lock.lock(5, TimeUnit.SECONDS);

        // 双重检查：拿锁后再查一次缓存
        json = redisTemplate.opsForValue().get(cacheKey);
        if (json != null) {
            return "NULL".equals(json) ? null : JSON.parseObject(json, Product.class);
        }

        // 查数据库并回填
        Product product = productMapper.selectById(productId);
        if (product == null) {
            redisTemplate.opsForValue().set(cacheKey, "NULL", 2, TimeUnit.MINUTES);
        } else {
            long ttl = 30 + ThreadLocalRandom.current().nextInt(10);
            redisTemplate.opsForValue().set(cacheKey, JSON.toJSONString(product), ttl, TimeUnit.MINUTES);
        }
        return product;
    } finally {
        if (lock.isHeldByCurrentThread()) {
            lock.unlock();
        }
    }
}
```

### 4. 热点Key处理

那个爆款商品的Key，QPS高达3000+，单个Redis节点扛不住。

**方案：本地缓存 + Redis + DB 三级缓存**

```java
@Service
public class ProductService {

    // Caffeine 本地缓存，容量1000，过期1分钟
    private final Cache<Long, Product> localCache = Caffeine.newBuilder()
        .maximumSize(1000)
        .expireAfterWrite(1, TimeUnit.MINUTES)
        .build();

    public Product getProduct(Long productId) {
        // L1: 本地缓存
        Product product = localCache.getIfPresent(productId);
        if (product != null) return product;

        // L2: Redis
        // L3: DB（包含上面的布隆过滤器和分布式锁逻辑）
        product = getProductWithLock(productId);

        if (product != null) {
            localCache.put(productId, product);
        }
        return product;
    }
}
```

## 踩坑总结

1. **Cache Aside 的核心是"写时删除缓存"**，不要写时更新缓存，容易出现并发不一致。
2. **缓存空值**是防穿透的最低成本方案，建议所有查询DB返回null的地方都缓存空值。
3. **布隆过滤器适合"数据量大、新增少"的场景**。商品、用户这类数据适合，订单这种频繁新增的不太适合。
4. **过期时间必须加随机值**。`ttl = baseTtl + random(0, offset)`，这个简单的操作能避免90%的雪崩场景。
5. **热点Key的终极方案是本地缓存**。Redis 再快也有网络开销，本地缓存是挡在Redis前面的一道防线。Caffeine 的性能是 Redis 的100倍以上。
6. **监控Redis的慢日志和大Key**。我们每周用 `redis-cli --bigkeys` 扫描一次，超过10KB的Key都要排查。
