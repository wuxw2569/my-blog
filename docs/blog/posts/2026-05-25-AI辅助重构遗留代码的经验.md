## 场景引入

"这段代码谁写的？" "三年前离职的那个。" "有文档吗？" "没有。" "有测试吗？" "也没有。"

每个接手遗留系统的开发者都经历过这段对话。遗留代码的特点是：能跑，没人敢动，没人完全理解。改一行可能崩一片，不改又无法添加新功能。

AI能不能帮我们啃这块硬骨头？这篇文章基于我重构一个5万行遗留Java系统的真实经验。

## 第一步：让AI帮你理解代码

重构之前必须先理解。AI在代码理解方面的能力远超我的预期。

### 1. 模块级理解

```
请分析 src/main/java/com/legacy/order/ 目录下的所有文件，告诉我：

1. 这个模块的整体职责是什么
2. 类与类之间的依赖关系（用文字描述）
3. 核心业务流程是什么（从入口方法开始追踪）
4. 有没有明显的代码异味（God Class、循环依赖等）
5. 有哪些外部依赖（数据库、消息队列、第三方API）
```

AI给出的分析报告，比自己读两天代码的理解还全面。特别是依赖关系的梳理，AI能画出完整的调用链。

### 2. 方法级理解

遇到看不懂的方法，直接丢给AI：

```
请逐行解释以下方法的逻辑，用中文注释标注关键步骤，
并指出可能的边界问题：

[粘贴一个200行的方法]
```

AI会把200行的大方法拆解成"初始化→查询→校验→计算→保存"几个阶段，比自己debug还快。

### 3. 发现隐含的业务规则

遗留代码中最有价值的是"隐含的业务规则"——那些没有文档、藏在if-else里的逻辑：

```
分析以下代码中的所有条件分支，找出隐含的业务规则，
用"当...则..."的格式列出：

[粘贴代码]
```

AI找出的业务规则经常让我惊叹："原来这里这样处理是因为VIP用户有特殊折扣"。

## 第二步：制定重构计划

理解代码后，让AI帮你制定重构计划：

```
基于以下代码分析，制定分阶段重构计划：

当前状态：
- OrderService类有800行，15个public方法
- 直接使用JDBC，没有ORM
- 没有接口抽象，直接依赖具体实现
- 0%测试覆盖率

目标：
- 拆分为合理的类结构
- 引入MyBatis-Plus替换JDBC
- 添加80%测试覆盖率

要求：
1. 每个阶段独立可提交，不能破坏现有功能
2. 按风险从低到高排序
3. 每个阶段估算工作量
```

AI给出的计划通常比自己想的更系统。关键是要让它考虑"不破坏现有功能"这个约束。

## 第三步：逐步重构

### 阶段1：提取接口（低风险）

```
将OrderService中与订单查询相关的5个方法提取为OrderQueryService接口。
要求：
1. 创建OrderQueryService接口
2. 创建OrderQueryServiceImpl实现类
3. 原OrderService改为委托调用
4. 保持所有外部调用点不变
```

### 阶段2：替换JDBC为MyBatis-Plus（中风险）

```
将以下JDBC代码替换为MyBatis-Plus写法：
1. 保留原有SQL逻辑，只换写法
2. 添加Entity映射
3. 保持返回值类型不变

原代码：
[粘贴JDBC代码]
```

**注意**：AI可能在替换过程中丢失一些SQL的特殊处理（如动态SQL、特殊函数），必须逐行对比。

### 阶段3：拆分God Class（高风险）

这是风险最高的步骤。我的策略是：

```
将OrderService按照职责拆分为：
1. OrderCreateService - 创建订单相关
2. OrderQueryService - 查询相关
3. OrderStatusService - 状态变更相关
4. OrderCalcService - 金额计算相关

要求：
- 先创建新的Service类
- 原OrderService改为Facade模式，委托调用新Service
- 每拆出一个方法，添加对应的单元测试
- 拆分完成后运行全量测试验证
```

## 真实案例：一次惊险的重构

### 背景
遗留系统中有一个`calculateDiscount`方法，300行，包含12个if-else分支。功能是根据用户等级、商品类型、活动规则计算折扣。

### 重构过程
1. **AI分析**：识别出12个分支对应6种用户等级 x 2种折扣逻辑
2. **AI生成测试**：基于分析结果生成了24个测试用例
3. **人工补充**：我又补充了8个边界测试
4. **AI重构**：将if-else重构为策略模式

### 惊险时刻
重构后的代码跑测试，24个AI生成的测试全部通过，但我补充的8个边界测试挂了3个。原因是AI在重构时，对"普通用户+限时折扣"的组合处理有误。

**教训**：AI会忽略一些"不起眼"的业务规则。如果你的测试覆盖不够，AI的重构可能悄悄改变业务行为。

## AI可能引入的回归问题

### 1. 丢失边界条件处理

```java
// 原代码有null检查
if (user != null && user.getAddress() != null) {
    // 处理地址
}

// AI重构后可能丢失
processAddress(user.getAddress()); // NPE风险
```

### 2. 改变异常处理语义

```java
// 原代码吞掉异常返回默认值
try {
    return calculate(data);
} catch (Exception e) {
    log.warn("计算失败，使用默认值", e);
    return defaultValue;
}

// AI重构后可能改成抛出异常
return calculate(data); // 改变了调用方预期的行为
```

### 3. 改变事务边界

原代码可能在一个事务中，AI拆分后可能导致事务不一致。这是最危险的回归问题，因为测试不一定能发现。

## 重构检查清单

每次AI重构后，我都会按这个清单检查：

- [ ] 所有原有测试仍然通过
- [ ] 新增测试覆盖了重构的代码
- [ ] 所有public方法签名未变（或已更新所有调用方）
- [ ] 异常处理语义未变
- [ ] 事务边界未变
- [ ] 日志输出的关键信息未丢失
- [ ] 没有引入新的依赖版本冲突

## 总结

AI辅助重构遗留代码的核心价值是**降低认知负担**——帮你理解代码、生成测试、执行重构。但它不能替代你的业务判断。

最佳实践：
1. **先理解再动手**：让AI帮你全面分析，不要直接重构
2. **测试先行**：重构前先补充测试，特别是边界测试
3. **小步前进**：每次只改一小块，验证后再继续
4. **人工Review**：AI重构的代码，逐行Review异常处理和边界条件
5. **保留回退能力**：每个阶段独立提交，随时可以回退

AI不会让遗留代码变成艺术品，但能让"不敢碰"变成"可以改"。
