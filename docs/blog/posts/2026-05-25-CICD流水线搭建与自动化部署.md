### 问题场景

我们团队之前部署流程是这样的：开发打包 -> 把jar包传到服务器 -> SSH上去手动重启。听起来很原始，但这在很多中小厂是常态。直到有一次半夜出了事故，值班同事回滚到上一个版本花了40分钟（翻聊天记录找jar包），领导终于批准搞CI/CD。

### 选型对比：Jenkins vs GitLab CI vs GitHub Actions

| 维度 | Jenkins | GitLab CI | GitHub Actions |
|------|---------|-----------|----------------|
| 部署方式 | 自托管 | GitLab自带 | GitHub托管 |
| 上手难度 | 中等 | 低 | 低 |
| 插件生态 | 最丰富 | 一般 | marketplace活跃 |
| 适合规模 | 任意 | 中小规模 | 开源/小团队 |
| 维护成本 | 高（要自己维护） | 低 | 最低 |

**我们的选择**：公司用的是GitLab私有部署，所以直接用了GitLab CI。理由很简单——零额外基础设施，`.gitlab-ci.yml` 跟代码一起版本管理。

### GitLab CI 流水线实战

```yaml
# .gitlab-ci.yml
stages:
  - build
  - test
  - package
  - deploy

variables:
  MAVEN_OPTS: "-Dmaven.repo.local=$CI_PROJECT_DIR/.m2/repository"
  DOCKER_REGISTRY: "registry.cn-hangzhou.aliyuncs.com/ourteam"

# 缓存Maven依赖，加速构建
cache:
  key: ${CI_COMMIT_REF_SLUG}
  paths:
    - .m2/repository/

# 编译构建
build:
  stage: build
  image: maven:3.9-eclipse-temurin-17
  script:
    - mvn clean compile -B
  only:
    - merge_requests
    - main
    - tags

# 单元测试
test:
  stage: test
  image: maven:3.9-eclipse-temurin-17
  script:
    - mvn test -B
  artifacts:
    reports:
      junit: target/surefire-reports/TEST-*.xml
  only:
    - merge_requests
    - main
    - tags

# 构建Docker镜像并推送
package:
  stage: package
  image: docker:24.0
  services:
    - docker:24.0-dind
  script:
    - docker login -u $CI_REGISTRY_USER -p $CI_REGISTRY_PASSWORD $DOCKER_REGISTRY
    - docker build -t $DOCKER_REGISTRY/app:$CI_COMMIT_SHORT_SHA .
    - docker push $DOCKER_REGISTRY/app:$CI_COMMIT_SHORT_SHA
    # 打latest标签
    - docker tag $DOCKER_REGISTRY/app:$CI_COMMIT_SHORT_SHA $DOCKER_REGISTRY/app:latest
    - docker push $DOCKER_REGISTRY/app:latest
  only:
    - main
    - tags

# 部署到生产环境
deploy_prod:
  stage: deploy
  image: bitnami/kubectl:latest
  script:
    - kubectl set image deployment/app app=$DOCKER_REGISTRY/app:$CI_COMMIT_SHORT_SHA -n production
    - kubectl rollout status deployment/app -n production --timeout=300s
  environment:
    name: production
    url: https://app.ourcompany.com
  when: manual   # 生产环境手动触发
  only:
    - tags
```

### 构建优化：从8分钟到3分钟

**问题**：Maven项目每次CI构建都要重新下载依赖，耗时4-5分钟。

**优化1：Maven缓存**

```yaml
cache:
  key:
    files:
      - pom.xml    # pom.xml变了才刷新缓存
  paths:
    - .m2/repository/
```

**优化2：跳过已稳定的模块**

```bash
# 只构建变更的模块，而不是整个多模块项目
mvn install -pl $(git diff --name-only origin/main | grep '/src/' | cut -d'/' -f1 | sort -u | tr '\n' ',') -am -B
```

`-pl` 指定模块，`-am` 同时构建依赖模块。这个技巧在20个模块的单体项目中效果显著。

**优化3：并行测试**

```bash
mvn test -T 1C -B
# -T 1C 表示每个CPU核心分配一个线程
```

### 制品管理

打出来的jar包和Docker镜像要有地方存，方便回滚和审计。

**Docker镜像**：推到阿里云容器镜像服务，按commit SHA打标签，永不覆盖。

**构建产物归档**：

```yaml
artifacts:
  paths:
    - target/*.jar
  expire_in: 30 days   # 保留30天
```

### 自动回滚

这是我们认为CI/CD最有价值的功能。实现方式：

```bash
#!/bin/bash
# rollback.sh - 回滚到上一个版本

DEPLOYMENT="app"
NAMESPACE="production"

# 获取当前镜像版本
CURRENT=$(kubectl get deployment $DEPLOYMENT -n $NAMESPACE -o jsonpath='{.spec.template.spec.containers[0].image}')
echo "当前版本: $CURRENT"

# 获取上一个revision
PREV_REVISION=$(kubectl rollout history deployment/$DEPLOYMENT -n $NAMESPACE | tail -3 | head -1 | awk '{print $1}')
echo "回滚到revision: $PREV_REVISION"

# 执行回滚
kubectl rollout undo deployment/$DEPLOYMENT --to-revision=$PREV_REVISION -n $NAMESPACE

# 等待回滚完成
if kubectl rollout status deployment/$DEPLOYMENT -n $NAMESPACE --timeout=300s; then
    echo "回滚成功"
else
    echo "回滚超时，需要人工介入"
    exit 1
fi
```

**健康检查驱动的自动回滚**（配合K8s）：

```yaml
# deployment.yaml 关键部分
spec:
  template:
    spec:
      containers:
      - name: app
        readinessProbe:
          httpGet:
            path: /actuator/health/readiness
            port: 8080
          initialDelaySeconds: 30
          periodSeconds: 10
          failureThreshold: 3
        livenessProbe:
          httpGet:
            path: /actuator/health/liveness
            port: 8080
          initialDelaySeconds: 60
          periodSeconds: 10
  strategy:
    rollingUpdate:
      maxSurge: 1
      maxUnavailable: 0    # 保证零停机
    type: RollingUpdate
  minReadySeconds: 30      # Pod就绪后等30秒再继续
```

如果新版本健康检查失败，K8s会自动停止滚动更新，旧版本的Pod不受影响。

### Jenkins场景补充

如果你们用Jenkins，Pipeline的关键配置：

```groovy
pipeline {
    agent any
    
    options {
        timeout(time: 20, unit: 'MINUTES')  // 全局超时
        disableConcurrentBuilds()            // 禁止并发构建
    }
    
    stages {
        stage('构建') {
            steps {
                sh 'mvn clean package -DskipTests -B'
            }
        }
        stage('部署') {
            when { branch 'main' }
            steps {
                sh './deploy.sh'
            }
        }
    }
    
    post {
        failure {
            // 构建失败时通知钉钉群
            sh 'curl -X POST https://oapi.dingtalk.com/robot/send?access_token=xxx -H "Content-Type: application/json" -d \'{"msgtype":"text","text":{"content":"构建失败: ${JOB_NAME} #${BUILD_NUMBER}"}}\''
        }
    }
}
```

### 踩坑总结

1. **CI环境和生产环境的基础镜像要一致**。CI用maven:3.9打包，生产也要用同一版本的JRE，否则可能遇到依赖版本不兼容
2. **敏感信息不要写在CI配置文件里**。用GitLab CI Variables或Vault管理，`.gitlab-ci.yml` 是会提交到仓库的
3. **部署脚本要幂等**。同一个版本部署两次不能出问题。用 `kubectl set image` 而不是自己写脚本去SSH
4. **不要跳过测试直接部署**。"这次改动很小，不测了"——经典flag。把单元测试加到CI的门禁里，不通过就不让打包
5. **回滚方案要在上线前就准备好**。不要等出了事才手忙脚乱地查上一个版本是什么

CI/CD的核心价值不是"自动化"，而是"可重复、可回滚、可审计"。每一次部署都应该能回答：谁部署的？部署了什么？能不能回滚？
