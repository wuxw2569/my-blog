## 场景引入

同一个需求，给AI写一句话和写一段结构化Prompt，输出质量天差地别。我见过同事让AI"写一个用户注册接口"，得到的是最简单的Controller加Service；而我用结构化Prompt，得到的是完整的参数校验、异常处理、密码加密、邮箱验证的完整方案。

Prompt工程不是玄学，是有方法论的。这篇文章分享我在实际开发中验证有效的Prompt技巧和模板。

## 核心技巧

### 1. 角色设定（Role Prompting）

给AI一个角色，它会调整输出的专业度和风格。

```
# 差的Prompt
写一个订单服务

# 好的Prompt
你是一个有10年经验的Java后端工程师，擅长Spring Boot和DDD设计。
请为电商系统设计一个订单服务，要求：
- 遵循DDD的聚合根设计模式
- 考虑并发场景下的库存扣减
- 包含订单状态机管理
```

**实测对比**：角色设定后的输出，代码结构更专业，会主动考虑边界情况，而不是给你最简单的CRUD。

### 2. Few-shot示例

给AI一两个"输入-输出"示例，它会模仿你的风格。

```
请按照以下格式为项目中的工具类编写Javadoc：

示例输入：
public class DateUtils {
    public static String formatDate(Date date, String pattern) { ... }
}

示例输出：
/**
 * 日期工具类，提供常用的日期格式化和解析方法。
 * <p>线程安全，所有方法均为静态方法。</p>
 * 
 * @author ai-helper
 * @since 1.0.0
 */
public class DateUtils {
    /**
     * 将日期对象格式化为指定格式的字符串。
     *
     * @param date 待格式化的日期对象，不能为null
     * @param pattern 格式化模式，如 "yyyy-MM-dd HH:mm:ss"
     * @return 格式化后的日期字符串
     * @throws IllegalArgumentException 当date或pattern为null时抛出
     */
    public static String formatDate(Date date, String pattern) { ... }
}

现在请为以下类编写Javadoc：
[粘贴代码]
```

### 3. Chain of Thought（思维链）

让AI"一步步思考"，复杂任务的准确率显著提升。

```
请分析以下慢查询SQL，按以下步骤思考：

1. 先解释这个SQL的业务意图
2. 分析执行计划，找出瓶颈
3. 检查是否有索引可以优化
4. 给出优化建议（包括改写SQL和添加索引）
5. 评估优化后的预期性能提升

SQL: 
SELECT o.*, u.name, u.phone 
FROM orders o 
LEFT JOIN users u ON o.user_id = u.id 
WHERE o.create_time > '2026-01-01' 
AND o.status IN (1, 2, 3)
ORDER BY o.create_time DESC 
LIMIT 10000, 20;
```

这种方式让AI不会"跳步"，输出更有逻辑性。

### 4. 结构化输出

明确要求输出格式，减少后续整理工作。

```
请审查这段代码，以JSON格式输出结果：

{
  "issues": [
    {
      "file": "文件名",
      "line": 行号,
      "severity": "high|medium|low",
      "category": "security|performance|readability|bug",
      "description": "问题描述",
      "suggestion": "修改建议",
      "code_example": "示例代码"
    }
  ],
  "summary": {
    "total": 数量,
    "high": 数量,
    "medium": 数量,
    "low": 数量,
    "overall_quality": "1-10分评分"
  }
}
```

### 5. 约束条件

明确告诉AI"不要做什么"，和"要做什么"同样重要。

```
请重构这个Service类，要求：

必须：
- 保持所有public方法签名不变（外部有调用）
- 使用Spring的构造器注入
- 保持@Transactional注解在正确的位置

禁止：
- 不要引入新的依赖
- 不要使用反射
- 不要修改数据库表结构
- 不要使用Java 8以上的特性（项目还在用JDK 8）
```

## 实战Prompt模板库

### 模板1：生成API接口

```
角色：Spring Boot后端工程师
任务：生成 [功能名称] 的REST API

技术栈：
- Spring Boot 3.2 + MyBatis-Plus
- 数据库：MySQL 8.0
- 参数校验：@Valid + 自定义注解
- 统一响应：Result<T> 包装类
- 异常处理：@RestControllerAdvice

要求：
1. Controller层：RESTful风格，包含Swagger注解
2. Service层：业务逻辑，事务管理
3. DTO层：请求/响应对象，包含校验注解
4. 参数校验：非空、长度、格式等
5. 异常处理：业务异常和系统异常分别处理

接口详情：
- 路径：POST /api/v1/[资源]
- 请求参数：[列出字段和类型]
- 业务规则：[描述业务逻辑]
- 响应格式：{code, message, data}
```

### 模板2：生成单元测试

```
角色：测试工程师
任务：为以下类编写单元测试

要求：
- 框架：JUnit 5 + Mockito
- 覆盖率目标：行覆盖80%+
- 测试场景：
  1. 正常流程
  2. 边界条件（null、空集合、0值）
  3. 异常场景（业务异常、系统异常）
- Mock策略：只Mock外部依赖（数据库、远程调用）
- 命名规范：方法名_场景_期望结果

请按Given-When-Then格式组织每个测试方法。
```

### 模板3：代码解释

```
请解释以下代码，要求：

1. 总体功能：一句话说明这段代码做什么
2. 逐行解释：对关键行添加注释
3. 设计模式：是否使用了设计模式，为什么
4. 潜在问题：有没有bug、性能问题、安全风险
5. 改进建议：如果有，给出更好的写法

[粘贴代码]
```

### 模板4：SQL优化

```
角色：数据库优化专家
任务：优化以下SQL

环境信息：
- 数据库：MySQL 8.0
- 表数据量：[表名]约[X]万行
- 当前索引：[列出索引]
- 慢查询阈值：200ms

请分析：
1. 当前SQL的执行计划
2. 瓶颈在哪里
3. 给出至少2种优化方案
4. 每种方案的优缺点
5. 推荐方案及理由

[粘贴SQL]
```

## Prompt调优经验

### 1. 迭代式优化

不要期望一次写出完美Prompt。我的工作流是：
1. 写一个基础版本
2. 看AI的输出，找到不满意的地方
3. 在Prompt中添加针对性的约束
4. 重复2-3次，通常能得到满意结果

### 2. Prompt版本管理

好用的Prompt保存下来，我们团队建了一个共享的Prompt库：

```
/prompts
  /java
    api-generator.md
    unit-test.md
    code-review.md
  /sql
    slow-query-optimizer.md
    schema-design.md
  /docs
    javadoc-generator.md
    changelog.md
```

### 3. 不要过度复杂

简单任务不需要复杂Prompt。写一个工具方法，一句"用Java写一个字符串脱敏方法"就够了。过度工程化的Prompt反而会让AI"过度思考"，输出啰嗦的代码。

## 总结

Prompt工程的本质是**清晰地表达你的需求**。你越清楚自己要什么，AI给你的就越接近你要的。

核心原则：
1. **给角色**：让AI进入正确的"人设"
2. **给示例**：一图胜千言，一例胜万句
3. **给约束**：明确边界比明确方向更重要
4. **要结构化输出**：省去后续整理的时间
5. **迭代优化**：好Prompt是调出来的，不是一次写出来的

---

*Prompt技巧在不同AI模型间有差异，本文示例基于GPT-4和Claude测试，其他模型可能需要微调。*
