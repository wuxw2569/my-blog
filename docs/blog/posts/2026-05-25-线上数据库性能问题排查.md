## 问题场景

凌晨2点被告警叫醒：数据库CPU飙到95%，大量接口超时。后端没有DBA，你就是DBA。

此刻你需要一套清晰的排查思路：**先止血，再定位，后优化**。

这篇文章记录的是我们团队在没有专职DBA的情况下，后端开发自己排查数据库性能问题的完整方法论。每个命令、每条路径都是真实踩过的坑。

## 分析思路

### 第一步：止血——快速判断当前状态

登录MySQL后，先看几个关键指标：

```sql
-- 1. 查看当前连接数和状态
SHOW GLOBAL STATUS LIKE 'Threads%';
-- Threads_connected: 当前打开的连接数
-- Threads_running: 正在执行的连接数（这个值大于CPU核数就有问题）

-- 2. 查看是否有锁等待
SELECT * FROM information_schema.INNODB_TRX ORDER BY trx_started;
-- 长时间运行的事务可能是罪魁祸首

-- 3. 查看InnoDB状态摘要
SHOW ENGINE INNODB STATUS\G
-- 重点看 SEMAPHORES、TRANSACTIONS、LATEST DETECTED DEADLOCK 三个部分
```

如果发现有长时间运行的事务（超过60秒），先Kill掉止血：

```sql
-- 找到长事务
SELECT trx_id, trx_state, trx_started, TIMESTAMPDIFF(SECOND, trx_started, NOW()) AS duration,
       trx_mysql_thread_id, trx_query
FROM information_schema.INNODB_TRX
ORDER BY trx_started;

-- Kill掉
KILL <thread_id>;
```

### 第二步：定位——慢查询分析

**1. 开启慢查询日志（如果没开的话）**

```sql
-- 动态开启（重启后失效，记得补到配置文件）
SET GLOBAL slow_query_log = ON;
SET GLOBAL long_query_time = 1;
SET GLOBAL log_queries_not_using_indexes = ON;
```

**2. 用 mysqldumpslow 快速看TOP SQL**

```bash
# 按执行时间排序，取前10条
mysqldumpslow -s t -t 10 /var/log/mysql/slow.log

# 按执行次数排序
mysqldumpslow -s c -t 10 /var/log/mysql/slow.log

# 按返回行数排序
mysqldumpslow -s r -t 10 /var/log/mysql/slow.log
```

**3. 用 pt-query-digest 深度分析**

`mysqldumpslow` 太简单了，`pt-query-digest` 才是真正的神器。

```bash
# 安装
yum install percona-toolkit

# 分析慢查询日志
pt-query-digest /var/log/mysql/slow.log \
  --since '2026-05-24 00:00:00' \
  --until '2026-05-25 00:00:00' \
  --limit 20 \
  --order-by Query_time:avg
```

输出示例：

```
# Profile
# Rank Query ID                     Response time  Calls  R/Call  V/M
# ==== ============================== ============= ====== ======= =====
#    1 0xABC123DEF456                 2345.0000  45.2%   8234  0.2848  0.05 SELECT t_order
#    2 0x789GHI012JKL                 1200.0000  23.1%   5621  0.2135  0.03 SELECT t_user
#    3 0xMNO345PQR678                  890.0000  17.1%    234  3.8034  2.15 UPDATE t_inventory
```

关键字段解读：
- **Rank**：按总响应时间排序
- **Response time**：总响应时间和占比
- **Calls**：执行次数
- **R/Call**：平均每次执行时间
- **V/M**：方差/均值比，越大说明执行时间波动越大（可能偶尔很慢）

往下看每个SQL的详细分析：

```
# Query 1: 2.34 QPS, 0.2848s avg time, 45.2% of total
# Scores: V/M = 0.05
# Time range: 2026-05-24T00:00:01 to 2026-05-24T23:59:58
# Attribute    pct   total     min     max     avg     95%  stddev  median
# ============ === ======= ======= ======= ======= ======= ======= =======
# Count         45    8234
# Exec time     45   2345s    0.1s    2.3s   0.28s   0.52s    0.1s   0.25s
# Lock time     30     12s       0   0.05s   0.001s  0.002s   0.001  0.001s
# Rows sent     80  1.23M       1    2000    150     500     200     100
# Rows examine  90  15.6M      50   50000   1894    5000    3000    1200
```

**重点看 `Rows examine` 和 `Rows sent` 的比值**。如果 examine 远大于 sent，说明做了大量无用扫描。

### 第三步：锁等待排查

CPU高不一定全是慢SQL，锁等待也会导致大量线程排队。

**查看当前锁等待**：

```sql
-- MySQL 8.0+
SELECT
    r.trx_id AS waiting_trx_id,
    r.trx_mysql_thread_id AS waiting_thread,
    r.trx_query AS waiting_query,
    b.trx_id AS blocking_trx_id,
    b.trx_mysql_thread_id AS blocking_thread,
    b.trx_query AS blocking_query
FROM information_schema.INNODB_TRX r
INNER JOIN performance_schema.data_lock_waits w
    ON w.REQUESTING_ENGINE_TRANSACTION_ID = r.trx_id
INNER JOIN information_schema.INNODB_TRX b
    ON b.trx_id = w.BLOCKING_ENGINE_TRANSACTION_ID;
```

**查看具体的锁信息**：

```sql
-- MySQL 8.0 使用 performance_schema.data_locks
SELECT * FROM performance_schema.data_locks;

-- MySQL 5.7 使用 information_schema.INNODB_LOCKS（已废弃）
SELECT * FROM information_schema.INNODB_LOCKS;
```

**一个真实案例**：

我们有个接口执行 `UPDATE t_order SET status = 3 WHERE order_id = 12345`，但这个SQL执行了30秒。排查发现：

```
blocking_query: INSERT INTO t_order (...) VALUES (...)  -- 另一个线程在插入
blocking_thread: 5678
waiting_query: UPDATE t_order SET status = 3 WHERE order_id = 12345
```

原因是 `order_id` 不是主键，走的是二级索引。另一个线程正在插入数据导致了Next-Key Lock，阻塞了这个UPDATE。解决方案：给 `order_id` 加了唯一索引。

### 第四步：CPU/IO 瓶颈定位

**CPU 高的排查**：

```bash
# 1. 看MySQL进程列表
mysqladmin -u root -p processlist

# 或者在MySQL中
SHOW FULL PROCESSLIST;
-- 重点看 State 列：
-- Sending data: 正在读取数据（可能是全表扫描）
-- Sorting result: 正在排序（可能是 filesort）
-- Locked: 被锁住了
-- Waiting for table metadata lock: DDL被阻塞了
```

**IO 高的排查**：

```bash
# 用 iostat 看磁盘IO
iostat -x 1 5

# 关注指标：
# %util: 磁盘利用率，超过80%说明IO是瓶颈
# await: 平均IO等待时间，超过10ms就有问题
# r/s, w/s: 每秒读写次数
```

**InnoDB Buffer Pool 命中率**：

```sql
SHOW GLOBAL STATUS LIKE 'Innodb_buffer_pool_read%';
-- Innodb_buffer_pool_read_requests: 逻辑读次数
-- Innodb_buffer_pool_reads: 物理读次数（从磁盘读）

-- 命中率 = 1 - (Innodb_buffer_pool_reads / Innodb_buffer_pool_read_requests)
-- 低于99%说明Buffer Pool太小或者有大量全表扫描
```

### 第五步：具体SQL优化

定位到问题SQL后，用 `EXPLAIN` 分析执行计划：

```sql
EXPLAIN SELECT * FROM t_order
WHERE user_id = 10086
  AND status IN (1, 2, 3)
  AND create_time >= '2026-01-01'
ORDER BY create_time DESC LIMIT 20;
```

**EXPLAIN 关键字段速查**：

| 字段 | 关注点 |
|------|--------|
| type | ALL（全表扫描）最差，index, range, ref, eq_ref, const 依次变好 |
| key | 是否用了索引，用了哪个 |
| rows | 预估扫描行数，越小越好 |
| Extra | Using filesort（文件排序）、Using temporary（临时表）是需要优化的信号 |
| filtered | 过滤比例，100%最好 |

**常见优化模式**：

```sql
-- 模式1: filesort 消除
-- ORDER BY 字段如果在索引中，可以避免 filesort
ALTER TABLE t_order ADD INDEX idx_user_status_time (user_id, status, create_time);

-- 模式2: 覆盖索引消除回表
-- 把 SELECT 的字段也加到索引中
ALTER TABLE t_order ADD INDEX idx_cover (user_id, status, create_time, order_id, amount);

-- 模式3: 子查询改 JOIN
-- 慢: SELECT * FROM t_order WHERE user_id IN (SELECT id FROM t_user WHERE age > 20);
-- 快: SELECT o.* FROM t_order o INNER JOIN t_user u ON o.user_id = u.id WHERE u.age > 20;

-- 模式4: OR 改 UNION ALL
-- 慢: SELECT * FROM t_order WHERE user_id = 1 OR status = 5;
-- 快: SELECT * FROM t_order WHERE user_id = 1
--     UNION ALL
--     SELECT * FROM t_order WHERE status = 5 AND user_id != 1;
```

## 具体实现：排查脚本

我们沉淀了一个快速排查脚本，部署在每台MySQL服务器上：

```sql
-- file: db_check.sql
-- 快速数据库健康检查脚本

-- 1. 连接状态
SELECT '=== 连接状态 ===' AS section;
SHOW GLOBAL STATUS LIKE 'Threads%';

-- 2. 长事务（超过60秒）
SELECT '=== 长事务 ===' AS section;
SELECT trx_id, trx_state, trx_started,
       TIMESTAMPDIFF(SECOND, trx_started, NOW()) AS duration_seconds,
       trx_mysql_thread_id, trx_query
FROM information_schema.INNODB_TRX
WHERE TIMESTAMPDIFF(SECOND, trx_started, NOW()) > 60
ORDER BY trx_started;

-- 3. 锁等待
SELECT '=== 锁等待 ===' AS section;
SELECT
    r.trx_id AS waiting_trx,
    r.trx_query AS waiting_query,
    b.trx_id AS blocking_trx,
    b.trx_query AS blocking_query,
    TIMESTAMPDIFF(SECOND, r.trx_started, NOW()) AS wait_seconds
FROM information_schema.INNODB_TRX r
INNER JOIN performance_schema.data_lock_waits w
    ON w.REQUESTING_ENGINE_TRANSACTION_ID = r.trx_id
INNER JOIN information_schema.INNODB_TRX b
    ON b.trx_id = w.BLOCKING_ENGINE_TRANSACTION_ID;

-- 4. Buffer Pool 命中率
SELECT '=== Buffer Pool ===' AS section;
SELECT
    (1 - (
        (SELECT VARIABLE_VALUE FROM global_status WHERE VARIABLE_NAME = 'Innodb_buffer_pool_reads')
        /
        (SELECT VARIABLE_VALUE FROM global_status WHERE VARIABLE_NAME = 'Innodb_buffer_pool_read_requests')
    )) * 100 AS hit_rate_pct;

-- 5. 表大小 TOP 10
SELECT '=== 大表 TOP 10 ===' AS section;
SELECT table_schema, table_name,
       ROUND(data_length / 1024 / 1024, 2) AS data_mb,
       ROUND(index_length / 1024 / 1024, 2) AS index_mb,
       table_rows
FROM information_schema.tables
WHERE table_schema NOT IN ('mysql', 'information_schema', 'performance_schema', 'sys')
ORDER BY data_length DESC
LIMIT 10;
```

## 踩坑总结

1. **第一时间看 `SHOW FULL PROCESSLIST`**。90%的问题能在这一步定位：大量 Sending data 说明有慢查询，Locked 说明有锁等待，Waiting for table metadata lock 说明有DDL被阻塞。
2. **不要直接重启MySQL**。重启能临时解决，但问题会反复。先Kill长事务止血，再慢慢排查根因。
3. **pt-query-digest 是排查慢查询的终极武器**。它能帮你从海量慢查询日志中找出真正影响最大的SQL，而不是盯着个别慢SQL看。
4. **锁问题比慢查询更隐蔽**。一个持锁事务就能阻塞大量请求。设置 `innodb_lock_wait_timeout`（默认50秒，建议改成10秒），避免等锁太久。
5. **Buffer Pool 命中率低于99%说明配置有问题**。`innodb_buffer_pool_size` 建议设为物理内存的60%-70%。
6. **定期做健康检查**。我们每周跑一次上面的排查脚本，把大表、长事务、慢SQL的趋势记录下来，防患于未然。
7. **建立监控告警体系**。Prometheus + mysqld_exporter + Grafana 是最低成本的方案。CPU > 80%、连接数 > 80%、慢查询 > 50/min 都要告警。
