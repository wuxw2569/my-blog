### 问题场景

我们团队之前排查线上问题的方式是：SSH到服务器上 -> 找到日志文件 -> grep关键词 -> 肉眼看上下文。服务少的时候还行，当服务拆到十几个、部署了多个实例之后，一个问题可能要登录5台机器分别查日志，效率极低。最终我们搭建了ELK日志平台，把所有日志集中管理和查询。

### 日志级别规范

在搭建平台之前，先规范日志输出。日志乱写，再好的平台也白搭。

**级别使用原则**：

| 级别 | 用途 | 线上是否开启 |
|------|------|-------------|
| ERROR | 影响功能的错误，需要关注和处理 | 必须 |
| WARN | 潜在问题，不影响功能但需要留意 | 必须 |
| INFO | 关键业务节点（下单、支付、发货） | 必须 |
| DEBUG | 开发调试信息（参数详情、中间结果） | 默认关闭 |
| TRACE | 更细粒度的调试信息 | 默认关闭 |

**Spring Boot 配置**：

```yaml
logging:
  level:
    root: INFO
    com.ourcompany: INFO
    # 第三方库只打WARN，减少噪音
    org.springframework: WARN
    org.mybatis: WARN
    # 线上开启DEBUG排查问题时，只开特定包
    # com.ourcompany.order.service: DEBUG
```

**日志输出规范**：

```java
@Slf4j
@Service
public class OrderService {

    public OrderVO createOrder(CreateOrderRequest request) {
        // INFO级别：记录关键业务节点和关键参数
        log.info("【创建订单】用户:{}, 商品:{}, 数量:{}", 
                request.getUserId(), request.getSkuId(), request.getQuantity());
        
        try {
            // ... 业务逻辑
            
            log.info("【订单创建成功】订单号:{}, 金额:{}", 
                    order.getOrderId(), order.getAmount());
            return orderVO;
            
        } catch (InventoryException e) {
            // WARN级别：业务异常，可以恢复
            log.warn("【库存不足】商品:{}, 剩余:{}", request.getSkuId(), e.getRemaining());
            throw new BizException("库存不足");
            
        } catch (Exception e) {
            // ERROR级别：未知异常，需要排查
            // 必须带异常堆栈！第二个参数是Throwable
            log.error("【创建订单异常】用户:{}, 请求:{}", 
                    request.getUserId(), JSON.toJSONString(request), e);
            throw new BizException("系统异常");
        }
    }
}
```

**常见反模式**：
- `log.error("出错了")` —— 没有上下文信息，排查时不知道哪出错了
- `log.error("异常: " + e.getMessage())` —— 丢失堆栈，应该用 `log.error("msg", e)`
- `log.info(JSON.toJSONString(bigObject))` —— 大对象序列化打印影响性能，用 `{}` 占位符
- 循环内打日志 —— 一次批量操作可能打上万行日志

### MDC链路追踪

分布式系统中，一个请求经过多个服务，需要一个traceId把所有日志串起来。

**MDC（Mapped Diagnostic Context）配置**：

```java
@Component
public class TraceFilter implements Filter {

    private static final String TRACE_ID = "traceId";

    @Override
    public void doFilter(ServletRequest request, ServletResponse response, FilterChain chain)
            throws IOException, ServletException {
        String traceId = ((HttpServletRequest) request).getHeader("X-Trace-Id");
        if (StringUtils.isBlank(traceId)) {
            traceId = UUID.randomUUID().toString().replace("-", "");
        }
        MDC.put(TRACE_ID, traceId);
        
        // 把traceId放到响应头，方便前端排查
        ((HttpServletResponse) response).setHeader("X-Trace-Id", traceId);
        
        try {
            chain.doFilter(request, response);
        } finally {
            MDC.remove(TRACE_ID);
        }
    }
}
```

**Feign调用时传递traceId**：

```java
@Component
public class FeignTraceInterceptor implements RequestInterceptor {
    @Override
    public void apply(RequestTemplate template) {
        String traceId = MDC.get("traceId");
        if (traceId != null) {
            template.header("X-Trace-Id", traceId);
        }
    }
}
```

**logback配置中输出traceId**：

```xml
<!-- logback-spring.xml -->
<configuration>
    <appender name="CONSOLE" class="ch.qos.logback.core.ConsoleAppender">
        <encoder>
            <!-- %X{traceId} 输出MDC中的traceId -->
            <pattern>%d{yyyy-MM-dd HH:mm:ss.SSS} [%thread] [%X{traceId}] %-5level %logger{36} - %msg%n</pattern>
        </encoder>
    </appender>
    
    <root level="INFO">
        <appender-ref ref="CONSOLE" />
    </root>
</configuration>
```

日志输出效果：

```
2026-05-25 14:30:15.123 [http-nio-8080-exec-1] [a1b2c3d4e5f6] INFO OrderService - 【创建订单】用户:U001
2026-05-25 14:30:15.234 [http-nio-8080-exec-1] [a1b2c3d4e5f6] INFO InventoryService - 【扣减库存】SKU001
2026-05-25 14:30:15.456 [http-nio-8080-exec-1] [a1b2c3d4e5f6] INFO OrderService - 【订单创建成功】订单号:ORD001
```

同一个请求的所有日志都有相同的traceId，在Kibana中搜索traceId就能看到完整的请求链路。

### ELK Stack 部署

ELK = Elasticsearch + Logstash + Kibana。生产环境建议用Filebeat替代Logstash做日志采集（更轻量）。

**Docker Compose 一键部署**：

```yaml
version: '3.8'
services:
  elasticsearch:
    image: elasticsearch:8.11.3
    environment:
      - discovery.type=single-node
      - xpack.security.enabled=false  # 测试环境先关闭安全认证
      - ES_JAVA_OPTS=-Xms2g -Xmx2g   # 堆内存设为物理内存的50%
    volumes:
      - es-data:/usr/share/elasticsearch/data
    ports:
      - "9200:9200"
    ulimits:
      memlock:
        soft: -1
        hard: -1
    deploy:
      resources:
        limits:
          memory: 4G

  kibana:
    image: kibana:8.11.3
    environment:
      - ELASTICSEARCH_HOSTS=http://elasticsearch:9200
    ports:
      - "5601:5601"
    depends_on:
      - elasticsearch

  filebeat:
    image: elastic/filebeat:8.11.3
    volumes:
      - ./filebeat.yml:/usr/share/filebeat/filebeat.yml:ro
      - /data/logs:/var/log/app:ro  # 挂载应用日志目录
    depends_on:
      - elasticsearch

volumes:
  es-data:
```

### Filebeat 采集配置

```yaml
# filebeat.yml
filebeat.inputs:
  # 采集应用日志
  - type: log
    enabled: true
    paths:
      - /var/log/app/*.log
    # 多行日志合并（Java异常堆栈是多行的）
    multiline.pattern: '^\d{4}-\d{2}-\d{2}'
    multiline.negate: true
    multiline.match: after
    # 添加自定义字段，方便Kibana筛选
    fields:
      app: order-service
      env: production
    fields_under_root: true

  # 采集Nginx访问日志
  - type: log
    enabled: true
    paths:
      - /var/log/nginx/access.log
    fields:
      app: nginx
    fields_under_root: true

# 输出到Elasticsearch
output.elasticsearch:
  hosts: ["http://elasticsearch:9200"]
  # 按日期分索引
  index: "app-logs-%{+yyyy.MM.dd}"

# 或者输出到Logstash做进一步处理
# output.logstash:
#   hosts: ["logstash:5044"]

# 处理器：解析日志格式
processors:
  - dissect:
      tokenizer: "%{timestamp} [%{thread}] [%{traceId}] %{level} %{logger} - %{message}"
      field: "message"
      target_prefix: "log"
  - drop_fields:
      fields: ["agent", "ecs", "log.offset"]
```

**踩坑点**：Java异常堆栈是多行的，如果不配 `multiline`，每一行会被当成独立的一条日志，在Kibana里只看到异常的第一行，后面堆栈全丢了。

### Kibana 查询技巧

**1. 基础查询**

在Kibana的Discover页面，使用KQL语法：

```
# 搜索ERROR日志
log.level: "ERROR"

# 搜索特定服务的日志
app: "order-service" AND log.level: "ERROR"

# 按traceId搜索完整请求链路
log.traceId: "a1b2c3d4e5f6"

# 搜索特定用户的日志（日志中包含用户ID）
message: *U00123*

# 排除健康检查的日志
NOT message: *actuator*
```

**2. 时间范围**

Kibana右上角可以选时间范围。排查问题时建议先选最近1小时，再根据需要扩大范围。

**3. 常用可视化**

创建Dashboard，添加以下图表：

- **错误趋势折线图**：按时间统计ERROR日志数量，发现异常峰值
- **接口耗时TOP10柱状图**：找出最慢的接口
- **错误日志表格**：展示最近的ERROR详情，包含traceId方便跳转
- **日志量按服务分布饼图**：快速定位哪个服务日志量异常

**4. 高级查询**

```
# 搜索耗时超过1秒的请求（假设日志中有耗时信息）
message: *耗时* AND message: /[1-9]\d{3,}ms/

# 搜索NullPointer异常
message: *NullPointerException*

# 按traceId找到完整链路后，按时间排序查看调用顺序
log.traceId: "xxx" | sort by @timestamp asc
```

### 索引管理

日志量大了之后，Elasticsearch磁盘占用会快速增长。需要配置索引生命周期管理（ILM）：

```json
// 通过Kibana的Dev Tools创建ILM策略
PUT _ilm/policy/app-logs-policy
{
  "policy": {
    "phases": {
      "hot": {
        "actions": {
          "rollover": {
            "max_size": "10gb",
            "max_age": "1d"
          }
        }
      },
      "warm": {
        "min_age": "3d",
        "actions": {
          "shrink": { "number_of_shards": 1 },
          "forcemerge": { "max_num_segments": 1 }
        }
      },
      "delete": {
        "min_age": "30d",
        "actions": {
          "delete": {}
        }
      }
    }
  }
}
```

效果：
- 热数据保留3天，快速查询
- 3天后转为暖数据，压缩存储
- 30天后自动删除

### 生产环境注意事项

**Elasticsearch集群**：

生产环境至少3个节点做集群：

```yaml
# elasticsearch.yml - 节点1
cluster.name: our-elk
node.name: es-node-1
network.host: 0.0.0.0
discovery.seed_hosts: ["es-node-1", "es-node-2", "es-node-3"]
cluster.initial_master_nodes: ["es-node-1", "es-node-2", "es-node-3"]

# 内存锁定
bootstrap.memory_lock: true
```

**Kibana安全**：

```yaml
# 开启安全认证
xpack.security.enabled: true
xpack.security.enrollment.enabled: true
```

用Nginx给Kibana做反向代理+基础认证：

```nginx
server {
    listen 80;
    server_name kibana.ourcompany.com;
    
    auth_basic "Kibana Login";
    auth_basic_user_file /etc/nginx/.htpasswd;
    
    location / {
        proxy_pass http://localhost:5601;
    }
}
```

### 踩坑总结

1. **日志格式要统一**。所有服务用相同的日志格式，Filebeat解析规则才能共用。建议把日志配置做成公共Maven依赖
2. **traceId一定要全链路传递**。从网关到每个微服务，Feign调用、MQ消息都要传递，否则在Kibana里查不到完整链路
3. **Elasticsearch磁盘要监控**。磁盘使用率超过85%后ES会变成只读模式，所有写入失败。建议设置告警
4. **Filebeat要配置multiline**。Java异常堆栈不合并的话，在Kibana里基本没法用
5. **不要把所有日志都发到ELK**。DEBUG级别的日志、访问日志中的健康检查请求，这些高频低价值日志要在采集端过滤掉
6. **索引定期清理**。不设置ILM策略的话，磁盘迟早被撑满。生产环境日志保留15-30天足够
7. **Elasticsearch的JVM堆内存不要超过32GB**。超过32GB会失去压缩指针优化，内存利用率反而下降

ELK日志平台的搭建不复杂，复杂的是日志规范的制定和执行。好的日志习惯 + 集中化的日志平台，能让线上问题排查时间从小时级降到分钟级。
