## 场景引入

"新功能开发完了，但单测还没写。"这句话在deadline前出现的频率有多高，每个开发者心里都有数。单测重要，但写单测确实枯燥——特别是给工具类写边界测试，给Service写Mock测试。

AI能不能帮我们把这个苦差事自动化？我实测了三个月，结论是：**能，但有明确的边界**。

## AI写单测的效率

### 效率最高的场景

#### 1. 工具类/纯函数

这是AI的强项。给它一个工具方法，它能生成完整的边界测试：

输入Prompt：
```
为以下方法编写JUnit 5单元测试，覆盖所有边界条件：

public class StringUtils {
    /**
     * 字符串脱敏，保留前3位和后4位，中间用*替代
     */
    public static String desensitize(String input) {
        if (input == null || input.length() <= 7) {
            return input;
        }
        return input.substring(0, 3) 
            + "*".repeat(input.length() - 7) 
            + input.substring(input.length() - 4);
    }
}
```

AI输出（质量很高，直接可用）：
```java
@Test
void testDesensitize_normalPhone() {
    assertEquals("138****5678", StringUtils.desensitize("13812345678"));
}

@Test
void testDesensitize_exactlySevenChars() {
    assertEquals("1234567", StringUtils.desensitize("1234567"));
}

@Test
void testDesensitize_sixChars() {
    assertEquals("123456", StringUtils.desensitize("123456"));
}

@Test
void testDesensitize_nullInput() {
    assertNull(StringUtils.desensitize(null));
}

@Test
void testDesensitize_emptyString() {
    assertEquals("", StringUtils.desensitize(""));
}

@Test
void testDesensitize_longInput() {
    String result = StringUtils.desensitize("123456789012345");
    assertEquals("123*********45", result);
}
```

**效率对比**：人工写这类测试约15分钟，AI约2分钟（含Review时间）。效率提升约7倍。

#### 2. 数据转换类

DTO转换、对象映射这类代码，AI写测试又快又准：

```java
// 给AI这段代码
public class UserConverter {
    public static UserDTO toDTO(User user) {
        UserDTO dto = new UserDTO();
        dto.setId(user.getId());
        dto.setName(user.getName());
        dto.setPhone(desensitize(user.getPhone()));
        dto.setAge(Period.between(user.getBirthday(), LocalDate.now()).getYears());
        return dto;
    }
}
```

AI会自动生成测试null字段、日期边界（今天生日）、正常转换等场景。

### 效率一般的场景

#### 1. Service层业务逻辑

给Service写单测需要大量Mock，AI能生成基本框架，但经常需要调整：

```java
// AI生成的Service测试 - 框架正确，但Mock细节需要调整
@Test
void testCreateOrder_success() {
    // Given
    when(userRepository.findById(1L)).thenReturn(Optional.of(testUser));
    when(productRepository.findById(1L)).thenReturn(Optional.of(testProduct));
    // AI不知道库存检查的业务逻辑，可能Mock不当
    
    // When
    Order order = orderService.createOrder(request);
    
    // Then
    assertNotNull(order);
    assertEquals(OrderStatus.CREATED, order.getStatus());
    // AI可能不会验证库存扣减是否被调用
}
```

**问题**：AI不了解你的业务规则，Mock的设置经常不完整。需要人工补充业务断言。

#### 2. 异常场景测试

AI倾向于写"正常流程"的测试，对异常场景的覆盖不够：

```java
// AI通常不会主动测试的场景
@Test
void testCreateOrder_concurrentStockDeduction() {
    // 并发场景下的库存扣减 - AI不会想到这个
}

@Test
void testCreateOrder_productDeletedDuringCheckout() {
    // 下单过程中商品被删除 - AI不会想到这个
}
```

### 效率很差的场景

#### 1. 集成测试

需要真实数据库、消息队列、外部服务的测试，AI几乎无法生成可运行的代码：

```java
// AI生成的集成测试 - 看起来合理但通常跑不通
@SpringBootTest
@Testcontainers
class OrderIntegrationTest {
    @Container
    static MySQLContainer<?> mysql = new MySQLContainer<>("mysql:8.0");
    // AI可能不会正确配置Testcontainers
    // 可能不会处理Flyway迁移
    // 可能不会正确注入数据源
}
```

#### 2. 涉及复杂业务状态机的测试

订单状态流转、审批流程这类有大量状态组合的测试，AI生成的测试用例要么覆盖不全，要么状态转换逻辑错误。

## Mock策略指南

### AI擅长的Mock

简单的依赖注入Mock：
```java
@Mock
private UserRepository userRepository;

// AI能正确设置的Mock
when(userRepository.findById(anyLong())).thenReturn(Optional.of(testUser));
```

### AI不擅长的Mock

复杂的链式调用和行为验证：
```java
// AI经常遗漏的行为验证
verify(messageQueue, times(1)).send(eq("order.created"), argThat(event -> 
    event.getOrderId().equals(orderId) && 
    event.getTimestamp() != null
));
```

**建议**：让AI生成Mock框架，人工补充Mock细节和行为验证。

## 效率统计

基于我们团队3个月的使用数据：

| 场景 | 人工耗时 | AI+人工 | 效率提升 |
|-----|---------|---------|---------|
| 工具类测试（10个方法） | 2小时 | 20分钟 | 6x |
| Controller测试（5个接口） | 1.5小时 | 30分钟 | 3x |
| Service测试（3个方法） | 2小时 | 1小时 | 2x |
| 集成测试（1个场景） | 3小时 | 2.5小时 | 1.2x |

## 最佳实践

1. **工具类交给AI**：边界测试、参数校验测试，AI又快又准
2. **Service测试人机配合**：AI生成框架，人工补充业务断言和Mock细节
3. **集成测试人工为主**：AI只能帮忙生成样板代码
4. **始终Review**：AI可能生成编译通过但逻辑错误的测试（如断言永远为true）
5. **维护测试**：业务变更时，让AI同步更新测试用例

## 总结

AI写单测的最大价值是**消除重复劳动**——那些你知道该写但懒得写的边界测试、那些格式化的Mock设置。但它不能替代你对业务逻辑的理解，不能帮你设计测试策略，更不能帮你发现测试中的逻辑错误。

把AI当作"测试代码的代码生成器"，它能让你从80%的测试覆盖率轻松提升到90%，但最后那10%的业务关键路径，还是得靠你自己。
