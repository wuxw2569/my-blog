## 前言

很多 Java 后端开发在写 HTTP 接口调用时，对 TCP 连接的生命周期几乎没有感知。Spring RestTemplate 一包，URL 一填，请求就发出去了。直到线上出现"偶发 Connection Reset"、"连接被对端关闭"之类的报错，才会开始关注底层连接到底是怎么管理的。

这篇文章不讲 TCP 三次握手四次挥手的理论（面试八股已经够多了），直接从**什么场景该用长连接、什么场景该用短连接**出发，结合实际项目中的坑来说明。

## 问题场景

我们有一个中台服务 A，需要同时调用三种下游：

1. **内部用户服务 B**：同机房，调用频率极高（每秒上千次），接口响应快（5ms 以内）
2. **第三方短信服务 C**：外部公网，调用频率低（每天几万次），偶尔慢（1~3 秒）
3. **内部大数据服务 D**：调用频率低但单次传输数据量大（响应体 50MB+）

三种场景的网络特性完全不同，用同一套连接策略肯定不行。

## 分析思路

先理清长连接和短连接的本质区别：

**短连接**：每次 HTTP 请求都要经历 TCP 三次握手 → 数据传输 → 四次挥手。开销大，但连接生命周期短，不会被"脏"数据影响。

**长连接（Keep-Alive）**：TCP 连接建立后不立即关闭，后续请求复用同一个连接。省去了握手和挥手的开销，但引入了连接管理的复杂性。

选择的依据其实就是三个因素：**调用频率、响应延迟、连接稳定性**。

| 场景 | 调用频率 | 响应延迟 | 建议 |
|------|----------|----------|------|
| 高频内部调用 | 极高 | 低 | 长连接 + 连接池 |
| 低频外部调用 | 低 | 不确定 | 长连接 + 连接池（池子小一点） |
| 大数据传输 | 中 | 高 | 长连接 + 大超时 |
| 一次性 webhook | 极低 | 不关心 | 短连接即可 |

## 具体实现

### 场景一：高频内部调用 — 长连接 + 连接池

对内部用户服务 B，我们用了 OkHttp 配合连接池，核心代码如下：

```java
@Bean
public OkHttpClient internalClient() {
    ConnectionPool pool = new ConnectionPool(
            50,                // 空闲连接数
            60, TimeUnit.SECONDS  // 空闲超时
    );

    return new OkHttpClient.Builder()
            .connectionPool(pool)
            .connectTimeout(1, TimeUnit.SECONDS)
            .readTimeout(3, TimeUnit.SECONDS)
            // 关键：Keep-Alive 由 HTTP Header 控制
            .addNetworkInterceptor(chain -> {
                Request request = chain.request().newBuilder()
                        .header("Connection", "keep-alive")
                        .build();
                return chain.proceed(request);
            })
            .build();
}
```

长连接在这里的优势很明显：省去了每次 3 次握手的 1 个 RTT（同机房约 0.5ms），高频调用下累积效果显著。

### 场景二：第三方外部调用 — 长连接但更保守

对短信服务 C，虽然调用频率低，但我们也用长连接，原因是建立 TLS 握手的成本很高（外网环境通常 100~300ms）。但配置要更保守：

```java
@Bean
public OkHttpClient externalClient() {
    ConnectionPool pool = new ConnectionPool(
            10,                     // 少量空闲连接
            30, TimeUnit.SECONDS    // 空闲超时短一些
    );

    return new OkHttpClient.Builder()
            .connectionPool(pool)
            .connectTimeout(5, TimeUnit.SECONDS)
            .readTimeout(15, TimeUnit.SECONDS)
            // 外部服务可能不稳定，允许失败重试
            .retryOnConnectionFailure(true)
            .build();
}
```

### 场景三：大数据传输 — 长连接 + 大超时

对大数据服务 D，核心问题是 socketTimeout。普通的 10 秒超时根本不够：

```java
@Bean
public RestTemplate bigDataClient() {
    PoolingHttpClientConnectionManager cm =
            new PoolingHttpClientConnectionManager();
    cm.setMaxTotal(10);
    cm.setDefaultMaxPerRoute(10);

    // 核心：大数据传输超时要足够长
    RequestConfig config = RequestConfig.custom()
            .setConnectTimeout(3000)
            // 5 分钟读取超时，因为单次响应可能 50MB
            .setSocketTimeout(300_000)
            .build();

    CloseableHttpClient httpClient = HttpClients.custom()
            .setConnectionManager(cm)
            .setDefaultRequestConfig(config)
            // 开启空闲连接检测，避免大数据传输后连接状态不可靠
            .evictExpiredConnections()
            .evictIdleConnections(10, TimeUnit.SECONDS)
            .build();

    return new RestTemplate(new HttpComponentsClientHttpRequestFactory(httpClient));
}
```

## Keep-Alive 配置的坑

长连接的核心是 HTTP 的 `Keep-Alive` 机制，但这里面有很多隐蔽的坑。

### 坑一：服务端悄悄关闭连接

HTTP 1.1 默认开启 Keep-Alive，但服务端（比如 Nginx）通常会配置 `keepalive_timeout`。如果客户端不知道服务端的超时时间，就可能在连接已被服务端关闭后还去复用它，导致 `Connection Reset`。

```nginx
# Nginx 默认 keepalive_timeout 是 75 秒
# 如果客户端 80 秒没请求，Nginx 已经关闭了连接
# 客户端再发请求就会收到 connection reset
keepalive_timeout 65s;
```

**解决方案**：客户端的空闲连接超时要**小于**服务端的 keepalive_timeout。比如服务端是 65 秒，客户端就设 60 秒。

### 坑二：Connection 头被代理层改写

有些 Nginx 版本或负载均衡设备会改写 `Connection: keep-alive` 为 `Connection: close`。导致你以为是长连接，实际上每次都被改成了短连接。

排查方法：

```bash
curl -v -o /dev/null https://your-service.com/api/test 2>&1 | grep -i "connection"
```

### 坑三：长连接排队效应

连接池里的长连接数量有限（比如 100 个），高并发时所有请求排队等连接。如果其中一个下游服务变慢，就会拖慢整个连接池（慢请求占着连接不释放），最终导致所有下游都超时。

这就是经典的**连接池头阻塞**问题。

**解决方案**：按下游服务分组，独立连接池。

```java
// 不要：一个 RestTemplate 调所有服务
// 要：每个下游一个独立的 RestTemplate
@Bean("userRestTemplate")
public RestTemplate userRestTemplate() { ... }

@Bean("paymentRestTemplate")
public RestTemplate paymentRestTemplate() { ... }

@Bean("smsRestTemplate")
public RestTemplate smsRestTemplate() { ... }
```

## 踩坑总结

1. **长连接不是银弹**。连接池管理引入了额外的复杂性：连接泄漏、连接状态不一致、排队效应等。如果调用频率极低（一天几次），短连接反而更简单可靠。

2. **客户端超时 < 服务端超时**。这是长连接最基本的原则。如果服务端 keepalive_timeout 是 65 秒，客户端空闲回收就设 60 秒。

3. **按下游分连接池**。一个连接池打天下是初学者最容易犯的错。下游 A 慢了，不应该影响下游 B 的调用。

4. **开启空闲连接驱逐**。所有客户端库都支持这个功能，但很多人不知道要开。

5. **生产环境必须监控连接池指标**。available/leased/pending 三个数字是最基本的，出了问题第一眼看这三个就能快速定位。

最后说一个反直觉的结论：在我们的中台项目中，把所有调用都改成正确配置的长连接后，整体 QPS 提升了约 15%，主要收益来自减少了 TCP 握手和 TLS 握手的开销。但对于第三方调用，收益就没那么明显了，因为瓶颈往往在网络延迟而不是连接建立。
