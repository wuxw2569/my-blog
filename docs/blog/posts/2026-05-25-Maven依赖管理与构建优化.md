### 问题场景

接手一个20多个模块的Java项目，本地编译一次要6分钟，CI上更慢。依赖冲突时不时冒出来，最离谱的一次是两个版本的Jackson同时存在，本地测试没问题、线上JSON序列化随机报错。这篇文章记录了我们治理Maven项目的过程。

### 依赖冲突解决

**排查依赖树**

遇到NoSuchMethodError、ClassNotFoundException这类问题，第一反应是查依赖冲突：

```bash
# 查看完整依赖树
mvn dependency:tree

# 搜索特定依赖的来源
mvn dependency:tree -Dincludes=com.fasterxml.jackson.core:jackson-databind

# 输出到文件方便分析
mvn dependency:tree > dep-tree.txt
```

**常见的冲突场景**：

```
[INFO] +- com.ourcompany:order-service:jar:1.0.0
[INFO] |  \- com.fasterxml.jackson.core:jackson-databind:jar:2.15.2
[INFO] \- com.ourcompany:common-lib:jar:1.2.0
[INFO]    \- com.fasterxml.jackson.core:jackson-databind:jar:2.13.5
```

Maven默认用"最近路径优先"，不同模块引入的不同版本会打架。

**解决方式：dependencyManagement统一管控版本**

在父pom中声明所有第三方依赖的版本：

```xml
<dependencyManagement>
    <dependencies>
        <!-- Jackson BOM，一次性管理所有Jackson组件版本 -->
        <dependency>
            <groupId>com.fasterxml.jackson</groupId>
            <artifactId>jackson-bom</artifactId>
            <version>2.15.3</version>
            <type>pom</type>
            <scope>import</scope>
        </dependency>
        
        <!-- 其他公共依赖 -->
        <dependency>
            <groupId>com.google.guava</groupId>
            <artifactId>guava</artifactId>
            <version>32.1.3-jre</version>
        </dependency>
        
        <dependency>
            <groupId>org.projectlombok</groupId>
            <artifactId>lombok</artifactId>
            <version>1.18.30</version>
            <scope>provided</scope>
        </dependency>
    </dependencies>
</dependencyManagement>
```

子模块中不用再写version：

```xml
<dependencies>
    <dependency>
        <groupId>com.fasterxml.jackson.core</groupId>
        <artifactId>jackson-databind</artifactId>
        <!-- 版本由父pom的dependencyManagement统一管理 -->
    </dependency>
</dependencies>
```

**排除冲突依赖**：

```xml
<dependency>
    <groupId>com.thirdparty:some-sdk</groupId>
    <artifactId>some-sdk</artifactId>
    <version>2.0.0</version>
    <exclusions>
        <!-- 排除它自带的旧版Jackson -->
        <exclusion>
            <groupId>com.fasterxml.jackson.core</groupId>
            <artifactId>jackson-databind</artifactId>
        </exclusion>
    </exclusions>
</dependency>
```

### 私服配置

公司用的是Nexus作为Maven私服。配置很简单但容易忽略。

**settings.xml 配置**：

```xml
<!-- ~/.m2/settings.xml -->
<settings>
    <servers>
        <server>
            <id>nexus-releases</id>
            <username>deployment</username>
            <password>${env.NEXUS_PASSWORD}</password>
        </server>
        <server>
            <id>nexus-snapshots</id>
            <username>deployment</username>
            <password>${env.NEXUS_PASSWORD}</password>
        </server>
    </servers>

    <mirrors>
        <!-- 所有请求走私服，私服上没有的再去中央仓库拉 -->
        <mirror>
            <id>nexus</id>
            <mirrorOf>central</mirrorOf>
            <url>http://nexus.ourcompany.com/repository/maven-public/</url>
        </mirror>
    </mirrors>

    <profiles>
        <profile>
            <id>nexus</id>
            <repositories>
                <repository>
                    <id>central</id>
                    <url>http://nexus.ourcompany.com/repository/maven-public/</url>
                    <releases><enabled>true</enabled></releases>
                    <snapshots><enabled>true</enabled></snapshots>
                </repository>
            </repositories>
        </profile>
    </profiles>

    <activeProfiles>
        <activeProfile>nexus</activeProfile>
    </activeProfiles>
</settings>
```

**踩坑点**：`<server>` 的 `id` 必须和 `pom.xml` 中 `<repository>` 或 `<distributionManagement>` 的 `id` 完全一致，否则认证信息不会生效，报401。

**pom.xml 发布配置**：

```xml
<distributionManagement>
    <repository>
        <id>nexus-releases</id>
        <url>http://nexus.ourcompany.com/repository/maven-releases/</url>
    </repository>
    <snapshotRepository>
        <id>nexus-snapshots</id>
        <url>http://nexus.ourcompany.com/repository/maven-snapshots/</url>
    </snapshotRepository>
</distributionManagement>
```

### 多模块项目结构

一个典型的中型项目结构：

```
our-project/
├── pom.xml                          # 父POM，管理版本和插件
├── our-common/                      # 公共工具类
│   └── pom.xml
├── our-model/                       # 实体类、DTO
│   └── pom.xml
├── our-dao/                         # 数据访问层
│   └── pom.xml
├── our-service/                     # 业务逻辑层
│   └── pom.xml
├── our-api/                         # 对外API接口定义（供其他服务调用）
│   └── pom.xml
└── our-web/                         # Web启动模块（最终打包的模块）
    └── pom.xml
```

**父POM关键配置**：

```xml
<groupId>com.ourcompany</groupId>
<artifactId>our-project</artifactId>
<version>1.0.0-SNAPSHOT</version>
<packaging>pom</packaging>

<modules>
    <module>our-common</module>
    <module>our-model</module>
    <module>our-dao</module>
    <module>our-service</module>
    <module>our-api</module>
    <module>our-web</module>
</modules>

<!-- 统一编码 -->
<properties>
    <project.build.sourceEncoding>UTF-8</project.build.sourceEncoding>
    <maven.compiler.source>17</maven.compiler.source>
    <maven.compiler.target>17</maven.compiler.target>
    <spring-boot.version>3.2.0</spring-boot.version>
</properties>
```

### 构建速度优化

**优化前**：全量构建6分钟。**优化后**：增量构建1分钟以内。

**1. 跳过测试**

```bash
# 开发阶段跳过测试
mvn clean package -DskipTests

# 跳过测试编译（更快，连测试代码都不编译）
mvn clean package -Dmaven.test.skip=true
```

**2. 并行构建**

```bash
# 自动检测CPU核心数并行构建
mvn clean package -T 1C

# 指定线程数
mvn clean package -T 4
```

**3. 只构建变更模块**

```bash
# 只构建service模块及其依赖
mvn clean package -pl our-service -am
```

**4. 离线模式**（依赖已经下载好的情况下）

```bash
mvn clean package -o
```

**5. 使用Maven Daemon（mvnd）**

```bash
# 安装mvnd（比mvn快30%-50%）
# https://github.com/apache/maven-mvnd

# 使用方式和mvn完全一样
mvnd clean package -DskipTests
```

mvnd的原理是保持一个后台守护进程，避免每次启动JVM的开销。在我们项目的实测中，首次构建快了20%，后续增量构建快了40%。

### 版本锁定策略

**Spring Boot项目的BOM管理**：

```xml
<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-dependencies</artifactId>
            <version>${spring-boot.version}</version>
            <type>pom</type>
            <scope>import</scope>
        </dependency>
    </dependencies>
</dependencyManagement>
```

用了Spring Boot的BOM之后，Spring生态内所有组件的版本都由BOM管控，不需要你手动指定。

**SNAPSHOT版本管理规范**：
- 开发分支用SNAPSHOT：`1.1.0-SNAPSHOT`
- 发版时改为正式版本：`1.1.0`
- 发版后立刻升到下一个SNAPSHOT：`1.2.0-SNAPSHOT`

```bash
# Maven版本插件，一键升级版本号
mvn versions:set -DnewVersion=1.1.0

# 发版后升到下一个开发版本
mvn versions:set -DnewVersion=1.2.0-SNAPSHOT
```

### 踩坑总结

1. **dependencyManagement 和 dependencies 的区别**：前者只声明版本不引入依赖，后者直接引入。父POM用dependencyManagement，子模块用dependencies
2. **不要在子模块中覆盖父POM的版本管理**。如果某个子模块确实需要不同版本，先跟团队沟通，避免出现同一项目两个Jackson版本的尴尬
3. **SNAPSHOT版本不要在生产部署中使用**。CI打包时加 `-nsu` (no snapshot updates) 避免SNAPSHOT被更新导致构建不可重复
4. **`mvn clean` 不等于清除一切**。有时候需要 `rm -rf ~/.m2/repository/com/ourcompany` 才能清除本地缓存的旧版本
5. **注意maven-compiler-plugin版本**。默认版本可能不支持Java 17的新语法特性，建议显式指定：
```xml
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-compiler-plugin</artifactId>
    <version>3.11.0</version>
    <configuration>
        <release>17</release>
    </configuration>
</plugin>
```

Maven不是最好的构建工具，但它足够成熟、生态足够完善。把依赖管理搞清楚、把构建速度优化好，比折腾切换Gradle更有实际价值。
