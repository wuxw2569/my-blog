## 问题场景

我们有一个订单查询接口，线上RT从最初的200ms逐渐飙升到2s+。业务表 `t_order` 数据量到了800万级别，DBA不在、DBA就是你自己——这是很多中小厂的真实写照。

执行 `SHOW PROCESSLIST` 发现大量查询处于 Sending data 状态，慢查询日志里全是这个接口的SQL。更要命的是，之前有人加了"优化索引"，结果反而让另一条查询变慢了。

## 分析思路

### 1. 先看执行计划，不要凭感觉加索引

拿到慢SQL的第一件事，用 `EXPLAIN` 看执行计划。我们当时的主要查询SQL如下：

```sql
SELECT order_id, user_id, status, amount, create_time
FROM t_order
WHERE user_id = 10086
  AND status IN (1, 2, 3)
  AND create_time >= '2026-01-01'
ORDER BY create_time DESC
LIMIT 20;
```

执行 `EXPLAIN` 后发现：

```
+----+-------------+--------+------+---------------+------+---------+------+--------+-----------------------------+
| id | select_type | table  | type | possible_keys | key  | key_len | ref  | rows   | Extra                       |
+----+-------------+--------+------+---------------+------+---------+------+--------+-----------------------------+
|  1 | SIMPLE      | t_order| ALL  | idx_user      | NULL | NULL    | NULL | 789123 | Using where; Using filesort |
+----+-------------+--------+------+---------------+------+---------+------+--------+-----------------------------+
```

`type=ALL` 全表扫描，`key=NULL` 索引没用上，`Using filesort` 还做了文件排序。虽然有 `idx_user` 索引，但因为后面跟着 `IN` 条件和范围查询，优化器认为走索引还不如全表扫描。

### 2. 联合索引的顺序是最关键的决策

这里有一个经典原则：**等值查询的列放在前面，范围查询的列放在后面**。

当前查询的条件分析：
- `user_id = 10086` —— 等值匹配
- `status IN (1, 2, 3)` —— 等值匹配（IN 在这里是多个等值）
- `create_time >= '2026-01-01'` —— 范围匹配
- `ORDER BY create_time DESC` —— 排序字段

联合索引应该这样设计：

```sql
ALTER TABLE t_order ADD INDEX idx_user_status_time (user_id, status, create_time);
```

**为什么是这个顺序？**

B+树是按索引列从左到右依次比较的。`user_id` 等值查询先定位到子树，`status` IN 条件继续缩小范围，`create_time` 范围扫描时数据已经是按 `create_time` 排序的——这样连 `ORDER BY` 都可以利用索引有序性，避免 filesort。

如果把 `create_time` 放在 `status` 前面：

```sql
-- 错误顺序
ALTER TABLE t_order ADD INDEX idx_user_time_status (user_id, create_time, status);
```

那 `create_time` 的范围查询会导致后面的 `status` 列无法走索引（最左前缀原则），而且 `IN` 条件也会受到限制。

### 3. 覆盖索引避免回表

加上联合索引后，`EXPLAIN` 显示 `type=range`，`key=idx_user_status_time`，但 `Extra` 里有 `Using index condition`。这意味着虽然走了索引，但还需要**回表**去取 `order_id, amount` 等字段。

如果这个查询非常高频，可以考虑覆盖索引：

```sql
-- 将查询需要的字段都加到索引中（末尾追加）
ALTER TABLE t_order DROP INDEX idx_user_status_time;
ALTER TABLE t_order ADD INDEX idx_user_status_time_cover (user_id, status, create_time, order_id, amount);
```

这样 `EXPLAIN` 的 `Extra` 会显示 `Using index`，表示完全在索引中完成查询，零回表。代价是索引更大，写入性能略有下降。

**实际取舍**：我们最终只把 `order_id` 加了进去（作为隐含的主键包含），`amount` 没加。因为订单表写入量大，索引不宜过大，而且20条记录的回表开销完全可接受。

## 具体实现

### 索引失效的常见场景

在优化过程中，我们还发现了几个索引失效的坑，总结如下：

**1. 函数操作导致索引失效**

```sql
-- 错误：对索引列使用函数
SELECT * FROM t_order WHERE DATE(create_time) = '2026-01-15';

-- 正确：改写为范围查询
SELECT * FROM t_order
WHERE create_time >= '2026-01-15 00:00:00'
  AND create_time < '2026-01-16 00:00:00';
```

**2. 隐式类型转换**

`user_id` 是 `VARCHAR(32)` 类型，但代码里传了 `Long` 类型：

```java
// 错误：MyBatis参数类型不匹配
@Select("SELECT * FROM t_order WHERE user_id = #{userId}")
Order selectByUserId(@Param("userId") Long userId);
```

MySQL 会对 `user_id` 列做 `CAST` 转换，相当于加了函数，索引直接失效。改为传 `String` 类型即可。

**3. LIKE 左模糊**

```sql
-- 索引失效
SELECT * FROM t_order WHERE order_no LIKE '%20260115%';

-- 可以走索引
SELECT * FROM t_order WHERE order_no LIKE '20260115%';
```

**4. OR 条件中包含无索引列**

```sql
-- 如果 remark 列没有索引，整个查询退化为全表扫描
SELECT * FROM t_order WHERE user_id = 10086 OR remark LIKE '%VIP%';

-- 改用 UNION ALL
SELECT * FROM t_order WHERE user_id = 10086
UNION ALL
SELECT * FROM t_order WHERE remark LIKE '%VIP%' AND user_id != 10086;
```

### 慢查询日志的配置

在 `my.cnf` 中开启慢查询日志：

```ini
slow_query_log = 1
slow_query_log_file = /var/log/mysql/slow.log
long_query_time = 1
log_queries_not_using_indexes = 1
min_examined_row_limit = 100
```

我们用 `pt-query-digest` 分析慢查询日志，快速定位 TOP SQL：

```bash
pt-query-digest /var/log/mysql/slow.log --since '2026-05-20' --limit 20
```

### 线上加索引的注意事项

大表加索引不能直接 `ALTER TABLE`，会锁表。我们用的是 `pt-online-schema-change`：

```bash
pt-online-schema-change \
  --alter "ADD INDEX idx_user_status_time (user_id, status, create_time)" \
  --host=10.0.1.100 \
  --port=3306 \
  --user=root \
  --password=xxx \
  --database=order_db \
  --table=t_order \
  --execute
```

这个工具会创建新表、拷贝数据、交换表名，全程不锁表。但要注意：**磁盘空间需要至少预留原表1倍的空间**。我们有次磁盘不够直接失败了，还好是在从库先试的。

## 踩坑总结

1. **联合索引的列顺序**决定了索引是否能被有效使用。等值在前、范围在后，排序字段紧跟等值字段。
2. **覆盖索引不是万能的**，要权衡查询收益和写入成本。写多读少的表不适合加太多覆盖列。
3. **EXPLAIN 不看等于盲人摸象**。每次加索引前后都要跑 EXPLAIN，确认 `type`、`key`、`Extra` 的变化。
4. **线上加索引用工具**，不要直接 `ALTER TABLE`。`pt-online-schema-change` 或 GitHub 的 `gh-ost` 都是成熟方案。
5. **索引不是越多越好**。我们有过一张表15个索引，写入性能一塌糊涂。一般单表索引控制在5-6个以内。
6. **定期清理无效索引**。用 `pt-duplicate-key-checker` 检查重复索引，用 `sys.schema_unused_indexes` 找未使用的索引。
