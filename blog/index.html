Prompt工程实战：开发者写好提示词的方法与模板

场景引入 ​

同一个需求，给AI写一句话和写一段结构化Prompt，输出质量天差地别。我见过同事让AI"写一个用户注册接口"，得到的是最简单的Controller加Service；而我用结构化Prompt，得到的是完整的参数校验、异常处理、密码加密、邮箱验证的完整方案。

Prompt工程不是玄学，是有方法论的。这篇文章分享我在实际开发中验证有效的Prompt技巧和模板。

核心技巧 ​

1. 角色设定（Role Prompting） ​

给AI一个角色，它会调整输出的专业度和风格。

# 差的Prompt
写一个订单服务

# 好的Prompt
你是一个有10年经验的Java后端工程师，擅长Spring Boot和DDD设计。
请为电商系统设计一个订单服务，要求：
- 遵循DDD的聚合根设计模式
- 考虑并发场景下的库存扣减
- 包含订单状态机管理

实测对比：角色设定后的输出，代码结构更专业，会主动考虑边界情况，而不是给你最简单的CRUD。

2. Few-shot示例 ​

给AI一两个"输入-输出"示例，它会模仿你的风格。

请按照以下格式为项目中的工具类编写Javadoc：

示例输入：

3. Chain of Thought（思维链） ​

让AI"一步步思考"，复杂任务的准确率显著提升。

请分析以下慢查询SQL，按以下步骤思考：

1. 先解释这个SQL的业务意图
2. 分析执行计划，找出瓶颈
3. 检查是否有索引可以优化
4. 给出优化建议（包括改写SQL和添加索引）
5. 评估优化后的预期性能提升

SQL: 
SELECT o.*, u.name, u.phone 
FROM orders o 
LEFT JOIN users u ON o.user_id = u.id 
WHERE o.create_time > '2026-01-01' 
AND o.status IN (1, 2, 3)
ORDER BY o.create_time DESC 
LIMIT 10000, 20;

这种方式让AI不会"跳步"，输出更有逻辑性。

明确告诉AI"不要做什么"，和"要做什么"同样重要。

Prompt调优经验 ​

1. 迭代式优化 ​

不要期望一次写出完美Prompt。我的工作流是：

  1. 写一个基础版本
  2. 看AI的输出，找到不满意的地方
  3. 在Prompt中添加针对性的约束
  4. 重复2-3次，通常能得到满意结果

2. Prompt版本管理 ​

好用的Prompt保存下来，我们团队建了一个共享的Prompt库：

/prompts
  /java
    api-generator.md
    unit-test.md
    code-review.md
  /sql
    slow-query-optimizer.md
    schema-design.md
  /docs
    javadoc-generator.md
    changelog.md

3. 不要过度复杂 ​

简单任务不需要复杂Prompt。写一个工具方法，一句"用Java写一个字符串脱敏方法"就够了。过度工程化的Prompt反而会让AI"过度思考"，输出啰嗦的代码。

总结 ​

Prompt工程的本质是清晰地表达你的需求。你越清楚自己要什么，AI给你的就越接近你要的。

核心原则：

  1. 给角色：让AI进入正确的"人设"
  2. 给示例：一图胜千言，一例胜万句
  3. 给约束：明确边界比明确方向更重要
  4. 要结构化输出：省去后续整理的时间
  5. 迭代优化：好Prompt是调出来的，不是一次写出来的

Spring AI与向量数据库的集成实践

问题场景 ​

在做 RAG 知识库时，我们选型遇到了一个实际问题：用哪个向量数据库？

团队里有人用过 Redis，有人听过 Milvus，还有人建议用 Qdrant。各有各的道理，但没人能说清楚在我们的场景下到底该选哪个。

我们的场景是：知识库规模约 10 万条文档片段，需要支持实时更新，QPS 预估 50-100，P99 延迟要求 200ms 以内。

这篇文章记录了我们从选型到落地的完整过程，以及踩过的坑。

分析思路 ​

Spring AI 的 VectorStore 抽象层屏蔽了底层实现差异，理论上切换向量库只需要换依赖和配置。但实际上每个向量库的能力差异不小，特别是在过滤查询、索引管理、运维成本方面。

选型评估维度：

  • 性能：检索延迟、并发能力
  • 运维：部署复杂度、监控、备份
  • 功能：过滤查询、混合搜索、索引类型
  • 成本：存储、内存、许可证

具体实现 ​

Embedding 生成 ​

不管用哪个向量库，第一步都是把文本转成向量。DashScope 提供了 text-embedding-v2 模型：

java
@Service
@Slf4j
@RequiredArgsConstructor
public class EmbeddingService {

    private final EmbeddingModel embeddingModel;

    /**
     * 将文本列表转为向量
     * DashScope text-embedding-v2 支持批量请求
     */
    public List<float[]> embed(List<String> texts) {
        log.info("【Embedding】生成向量, 文本数: {}", texts.size());

        EmbeddingResponse response = embeddingModel.embed(
                new EmbeddingRequest(texts, EmbeddingOptions.builder()
                        .model("text-embedding-v2")
                        .build()));

        List<float[]> vectors = response.getResults().stream()
                .map(r -> {
                    double[] doubles = r.getOutput();
                    float[] floats = new float[doubles.length];
                    for (int i = 0; i < doubles.length; i++) {
                        floats[i] = (float) doubles[i];
                    }
                    return floats;
                })
                .toList();

        log.info("【Embedding】向量维度: {}", vectors.isEmpty() ? 0 : vectors[0].length);
        return vectors;
    }

    /**
     * 单条文本的 Embedding
     */
    public float[] embedOne(String text) {
        return embed(List.of(text)).get(0);
    }
}

踩坑点：DashScope 的 embedding 接口有批量限制，单次最多 25 条文本。如果要处理大量文档，需要自己做分批请求。

java
@Service
@Slf4j
@RequiredArgsConstructor
public class MilvusSearchService {

    private final VectorStore vectorStore;

    /**
     * 基础相似度搜索
     */
    public List<Document> search(String query, int topK) {
        SearchRequest request = SearchRequest.query(query)
                .withTopK(topK)
                .withSimilarityThreshold(0.6);
        return vectorStore.similaritySearch(request);
    }

    /**
     * 带元数据过滤的搜索
     * Milvus 支持标量过滤 + 向量搜索的混合查询
     */
    public List<Document> searchWithFilter(String query, String category, int topK) {
        SearchRequest request = SearchRequest.query(query)
                .withTopK(topK)
                .withSimilarityThreshold(0.5)
                // Milvus 过滤表达式语法
                .withFilterExpression("category == '" + category + "'");
        return vectorStore.similaritySearch(request);
    }

    /**
     * 范围过滤：只搜索最近 7 天更新的文档
     */
    public List<Document> searchRecent(String query, int topK) {
        long sevenDaysAgo = Instant.now().minus(7, ChronoUnit.DAYS).getEpochSecond();
        SearchRequest request = SearchRequest.query(query)
                .withTopK(topK)
                .withFilterExpression("updateTimestamp > " + sevenDaysAgo);
        return vectorStore.similaritySearch(request);
    }
}

向量库管理工具 ​

java
@RestController
@RequestMapping("/api/vector")
@RequiredArgsConstructor
@Slf4j
public class VectorStoreAdminController {

    private final VectorStore vectorStore;

    /**
     * 批量添加文档（带进度跟踪）
     */
    @PostMapping("/batch-add")
    public Map<String, Object> batchAdd(@RequestBody List<DocumentDto> docs) {
        log.info("【批量入库】文档数: {}", docs.size());

        // 转换为 Document 对象
        List<Document> documents = docs.stream()
                .map(d -> new Document(d.content(), d.metadata()))
                .toList();

        // 分批入库，每批 100 条
        int batchSize = 100;
        int successCount = 0;
        for (int i = 0; i < documents.size(); i += batchSize) {
            List<Document> batch = documents.subList(i,
                    Math.min(i + batchSize, documents.size()));
            try {
                vectorStore.add(batch);
                successCount += batch.size();
                log.debug("【批量入库】进度: {}/{}", successCount, documents.size());
            } catch (Exception e) {
                log.error("【批量入库失败】批次: {}, 错误: {}", i / batchSize, e.getMessage());
            }
        }

        return Map.of("total", docs.size(), "success", successCount);
    }

    /**
     * 按元数据删除文档
     * 注意：不是所有 VectorStore 都支持按条件删除

RAG 检索增强生成技术入门 ​

前言 ​

如果你只记住一个 AI 应用开发的技术名词，那应该是 RAG。

RAG（Retrieval-Augmented Generation，检索增强生成）是目前企业落地 AI 应用最主流的架构模式。几乎所有"让 AI 读你的文档来回答问题"的应用，背后都是 RAG。

为什么 RAG 如此重要？先从一个问题说起。

为什么需要 RAG？ ​

LLM 有三个致命缺陷：

  1. 知识有截止日期：模型的训练数据有时间边界，不知道最新的信息
  2. 不了解你的私有数据：模型没读过你公司的内部文档、产品手册、业务数据
  3. 会"幻觉"：当模型不确定时，它可能会编造一个看起来很合理但实际错误的回答

你可能会想：那我直接把所有文档塞到 prompt 里不就行了？

问题在于：

  • 文档可能有几十万字，超出上下文窗口
  • 即使塞得进去，模型对长文本中间部分的关注度很低
  • 每次调用都塞大量文本，token 成本会爆炸

RAG 的核心思想就是：先帮模型"查资料"，再让它"回答问题"。 就像开卷考试一样，先翻书找到相关章节，再回答问题。

RAG 的基本流程 ​

一个标准的 RAG 系统分为两个阶段：离线索引阶段 和 在线查询阶段。

阶段一：离线索引（数据准备） ​

原始文档（PDF/Word/网页/数据库）
  ↓
[文档解析] 提取纯文本
  ↓
[文本切分] 把长文档分成小块（chunk）
  ↓
[Embedding] 每个文本块变成向量
  ↓
[向量存储] 向量 + 原文一起存入向量数据库

这一步是离线完成的，相当于给你的知识库建"索引"。

阶段二：在线查询（用户提问时） ​

用户提问："我们的退货政策是什么？"
  ↓
[Embedding] 把问题变成向量
  ↓
[向量检索] 在向量数据库中找到最相关的几个文本块
  ↓
[构建 Prompt] 把检索到的文本块 + 用户问题 组装成 prompt
  ↓
[LLM 生成] 让 LLM 基于检索到的内容生成回答
  ↓
返回给用户："根据公司政策，7天内无理由退货..."

Spring AI实现RAG知识库检索增强

问题场景 ​

客服系统接入大模型后，有个明显的问题：模型不知道我们公司的产品信息、售后政策、常见问题。纯靠模型的通用知识，回答经常"一本正经地胡说八道"。

我们需要一个方案，把公司的文档知识喂给模型。最直觉的做法是把文档塞到 Prompt 里，但公司文档库有几百篇，全塞进去 Token 爆炸，而且检索效率极低。

RAG（Retrieval Augmented Generation，检索增强生成）就是解决这个问题的标准方案：先从知识库里检索出跟问题相关的文档片段，再把这些片段作为上下文交给模型回答。

分析思路 ​

RAG 的核心流程分三步：

  1. 离线索引：加载文档 -> 文本分割 -> 生成 Embedding -> 存入向量库
  2. 在线检索：用户问题 -> 生成问题 Embedding -> 向量相似度搜索 -> 取回相关片段
  3. 生成回答：把检索结果 + 用户问题组装成 Prompt -> 调用模型生成回答

Spring AI 对这三个步骤都有封装，但用起来还是有不少需要自己处理的地方。

@Service
@Slf4j
@RequiredArgsConstructor
public class DocumentIndexService {

    private final VectorStore vectorStore;

    /**
     * 加载并索引文档目录下的所有文件
     * 支持 PDF、Word、TXT、Markdown 等格式
     */
    public void indexDocuments(String directoryPath) {
        log.info("【文档索引开始】目录: {}", directoryPath);
        File dir = new File(directoryPath);
        if (!dir.exists() || !dir.isDirectory()) {
            throw new IllegalArgumentException("目录不存在: " + directoryPath);
        }

        // 使用 Tika 解析各种文档格式
        List<Document> documents = new ArrayList<>();
        for (File file : Objects.requireNonNull(dir.listFiles())) {
            try {
                // TikaDocumentReader 自动识别文件类型
                TikaDocumentReader reader = new TikaDocumentReader(new FileSystemResource(file));
                List<Document> docs = reader.get();
                // 给每个文档打上元数据标签，方便后续过滤
                docs.forEach(doc -> {
                    doc.getMetadata().put("source", file.getName());
                    doc.getMetadata().put("indexTime", Instant.now().toString());
                });
                documents.addAll(docs);
                log.info("【文档加载】文件: {}, 片段数: {}", file.getName(), docs.size());
            } catch (Exception e) {
                log.warn("【文档加载失败】文件: {}, 错误: {}", file.getName(), e.getMessage());
            }
        }

        // 文本分割
        TokenTextSplitter splitter = new TokenTextSplitter(
                800,   // 每段最大 token 数
                200,   // 重叠 token 数（保证上下文连贯）
                5,     // 最小每段 token 数
                10000, // 最大 token 数
                true   // 保留分隔符
        );
        List<Document> chunks = splitter.apply(documents);
        log.info("【文本分割完成】原始文档: {} 篇, 分割后: {} 个片段", documents.size(), chunks.size());

        // 写入向量库（自动生成 Embedding）
        vectorStore.add(chunks);
        log.info("【文档索引完成】已写入向量库");
    }
}

踩坑点：TokenTextSplitter 的 chunk size 设置很关键。太大了检索结果不够精确，太小了上下文不完整。我们经验是 500-1000 token 比较合适，看文档类型。结构化文档（如 FAQ）可以小一点，叙述性文档（如产品手册）要大一点。

第三步：检索服务 ​

java
@Service
@Slf4j
@RequiredArgsConstructor
public class KnowledgeSearchService {

    private final VectorStore vectorStore;

    /**
     * 检索与问题相关的知识片段
     * @param query 用户问题
     * @param topK 返回的最大片段数
     * @param threshold 相似度阈值（0-1），低于此值的结果不返回
     */
    public List<Document> search(String query, int topK, double threshold) {
        log.info("【知识检索】查询: {}, topK: {}, 阈值: {}", query, topK, threshold);

        SearchRequest request = SearchRequest.query(query)
                .withTopK(topK)
                .withSimilarityThreshold(threshold);

        List<Document> results = vectorStore.similaritySearch(request);

        log.info("【检索完成】命中 {} 个片段", results.size());
        results.forEach(doc -> {
            log.debug("【检索结果】来源: {}, 相似度: {}, 内容摘要: {}",
                    doc.getMetadata().get("source"),
                    doc.getScore(),
                    doc.getText().substring(0, Math.min(100, doc.getText().length())));
        });

        return results;
    }

    /**
     * 带过滤条件的检索
     * 比如只检索特定来源的文档
     */
    public List<Document> searchWithFilter(String query, String sourceFilter, int topK) {
        SearchRequest request = SearchRequest.query(query)
                .withTopK(topK)
                .withSimilarityThreshold(0.7)
                .withFilterExpression("source == '" + sourceFilter + "'");

        return vectorStore.similaritySearch(request);
    }
}

问题场景 ​

我们团队（80人规模，Java后端为主）在2024年初开始全面容器化迁移。最初的Dockerfile是开发同学随手写的，一个Spring Boot应用镜像竟然打到了1.2GB，启动时间超过60秒，线上还频繁出现OOM。这篇文章记录了我们从"能跑就行"到"生产可用"的完整过程。

多阶段构建：镜像从1.2GB瘦身到280MB ​

第一版Dockerfile很朴素——把本地打好的jar直接COPY进去：

dockerfile
# 反面教材：直接用JDK基础镜像 + 拷贝fat jar
FROM openjdk:17
COPY target/app.jar /app.jar
ENTRYPOINT ["java", "-jar", "/app.jar"]

问题很明显：openjdk:17 基础镜像约 500MB，加上jar包和各种缓存层，镜像体积巨大。

改进方案：多阶段构建

dockerfile
# ============ 阶段1：构建 ============
FROM maven:3.9-eclipse-temurin-17 AS builder
WORKDIR /build
# 先拷贝pom，利用Docker层缓存加速依赖下载
COPY pom.xml .
RUN mvn dependency:go-offline -B
COPY src ./src
RUN mvn package -DskipTests -B

# ============ 阶段2：运行 ============
FROM eclipse-temurin:17-jre-alpine
WORKDIR /app
# 只拷贝构建产物
COPY --from=builder /build/target/app.jar app.jar
# 创建非root用户运行
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
USER appuser
EXPOSE 8080
ENTRYPOINT ["java", "-jar", "app.jar"]

踩坑点：

  • eclipse-temurin:17-jre-alpine 比完整JDK镜像小很多，生产环境只需要JRE
  • 拷贝 pom.xml 单独执行 dependency:go-offline 是利用Docker层缓存的关键——只要 pom.xml 不变，依赖下载这一层就不会重新构建
  • 用非root用户运行是安全审计的硬性要求

JVM 容器化内存配置：别再被OOM搞得半夜起来重启 ​

这是我们踩坑最深的地方。Java应用跑在容器里，JVM默认会根据物理机内存分配堆大小，而不是容器限制。一个被限制了512MB的容器，JVM可能尝试分配4GB堆内存，直接被内核OOM Kill。

Java 17 + 容器感知

JDK 10+ 默认开启了 -XX:+UseContainerSupport，但光靠这个不够。我们的配置：

dockerfile
ENTRYPOINT ["java", \
  "-XX:+UseContainerSupport", \
  "-XX:MaxRAMPercentage=70.0", \
  "-XX:InitialRAMPercentage=50.0", \
  "-XX:+HeapDumpOnOutOfMemoryError", \
  "-XX:HeapDumpPath=/app/logs/heapdump.hprof", \
  "-XX:+UseG1GC", \
  "-XX:MaxGCPauseMillis=200", \
  "-Djava.security.egd=file:/dev/./urandom", \
  "-jar", "app.jar"]

关键参数说明：

  • MaxRAMPercentage=70.0：堆内存最多占容器内存的70%，剩下30%给Metaspace、线程栈、NIO DirectBuffer等
  • HeapDumpOnOutOfMemoryError：OOM时自动dump，线上排查必备
  • -Djava.security.egd：加速随机数生成，否则在容器中/dev/random可能阻塞导致启动慢

Fine-tuning 微调 vs Prompt 工程，怎么选？

Fine-tuning 微调 vs Prompt 工程，怎么选？ ​

前言 ​

这是一个被问了无数遍的问题：我想让 AI 模型更好地完成我的任务，是应该写更好的 Prompt，还是应该做微调（Fine-tuning）？

简短的回答：90% 的情况下，先把 Prompt 工程做到极致再说。 微调是大招，不到万不得已不要用。

这篇文章解释为什么。

什么是 Prompt 工程？ ​

Prompt 工程就是通过设计更好的输入提示（Prompt），让模型更好地完成任务。

它不改变模型本身，只是改变你和模型"沟通"的方式。

Prompt 工程的常见技巧：

1. 明确角色设定
   "你是一个资深的 Java 后端工程师，擅长性能优化..."

2. 给出示例（Few-shot）
   "输入: 用户投诉快递太慢
    输出: {"category": "物流", "sentiment": "负面", "urgency": "中"}"

3. 思维链（Chain-of-Thought）
   "请一步一步思考：
    1. 首先分析问题的关键信息
    2. 然后列出可能的解决方案
    3. 最后给出推荐方案"

4. 格式约束
   "请严格按照以下 JSON 格式输出，不要包含其他内容：
    {"name": "...", "age": ..., "city": "..."}"

5. 分步指令
   "第一步：阅读以下文档
    第二步：提取关键信息
    第三步：生成摘要"

Prompt 工程的成本为零（不需要训练），见效快（改了 prompt 立刻就能看到效果），是性价比最高的优化手段。

什么是 Fine-tuning（微调）？ ​

微调是用你自己的数据对模型进行额外训练，让模型"学会"你想要的行为模式。

打个比方：

  • Prompt 工程 = 在考试前给学生一份"答题指南"
  • 微调 = 让学生做大量的模拟题，让他形成"肌肉记忆"

微调会改变模型的权重参数，所以它确实能让模型在特定任务上有更好的表现。但代价也很高。

前言 ​

如果你是一个后端开发、前端开发或者全栈工程师，最近一两年你一定被各种"AI"、"大模型"、"LLM"的新闻刷屏了。但说实话，很多科普文章要么太学术、要么太营销，看完之后你可能还是不太清楚这东西到底是什么、能干什么、不能干什么。

这篇文章就站在一个"有编程基础但AI知识薄弱的开发者"的角度，把 LLM 最核心的概念讲清楚。

LLM 到底是什么？ ​

LLM（Large Language Model），大语言模型。 你可以把它理解成一个"超级自动补全器"。

就像你手机输入法会根据你已打的字猜测下一个字一样，LLM 也是在做类似的事情——根据前面的文本，预测下一个最可能出现的词。只不过它的规模大了几个数量级，训练数据覆盖了互联网上大量的文本，所以它"补全"出来的内容看起来像是一个知识渊博的人在回答问题。

用一个更具体的类比：

  • 传统程序：你写好规则，它按规则执行 → 规则是确定的
  • LLM：你给它一段话，它基于概率生成下一段话 → 输出是有随机性的

所以 LLM 本质上是一个概率模型，不是数据库，也不是搜索引擎。它不"知道"答案，而是"推测"出一个合理的回答。

Token：LLM 世界的"货币单位" ​

你调用 LLM API 的时候，会看到 token 这个概念。Token 是 LLM 处理文本的最小单位。

Token 不等于汉字。

  • 英文中，一个单词可能被拆成 1-3 个 token（如 "unbelievable" → "un" + "believ" + "able"）
  • 中文中，一个汉字通常是 1-2 个 token
  • 大致经验值：1 个 token 约等于 0.75 个英文单词，或者 1.5 个中文字符

Token 为什么重要？因为：

  1. 计费单位：API 调用按 token 计费，输入和输出分别算
  2. 上下文窗口的度量：模型一次能处理多少 token 是有限制的
  3. 响应速度：token 数越多，生成越慢

# 大约输出: 文本长度: 16, Token数: 6

文件上传下载的网络层优化

前言 ​

做过企业级系统的同学应该都有过这种经历：业务方说"我要上传一个 2GB 的 Excel 报表"，你用 MultipartFile 一接，结果 OOM 了。或者："我要下载一份审计日志，大概 5GB"，你用 IOUtils.copy() 一刷，结果下载到 80% 网络抖了一下，前功尽弃。

这两个场景分别对应了文件传输的两个核心问题：大文件上传和大文件下载。这篇文章分享我们在实际项目中总结的网络层优化方案。

问题场景 ​

我们的文档管理系统需要支持：

  1. 员工上传 50MB~2GB 的文件（设计图纸、视频素材等）
  2. 管理员导出 100MB~5GB 的审计日志
  3. 支持上传中断后断点续传
  4. 下载速度要快，用 CDN 加速

初始方案的痛点：

  • 直接用 MultipartFile，超过 200MB 就 OOM
  • 单次上传，网络断了要重传
  • 下载不支持断点，断了从头来
  • 文件不走 CDN，全部经过应用服务器

具体实现 ​

一、大文件分片上传 ​

核心思路：前端把大文件切成固定大小的分片（比如 5MB 一片），逐片上传到后端，全部上传完成后后端合并。

后端接口设计：

分片上传核心实现：

java
@Service
@Slf4j
public class FileService {

    // 分片临时目录
    private static final String CHUNK_TEMP_DIR = "/data/upload/chunks/";

    /**
     * 初始化上传：生成 uploadId，创建分片目录
     */
    public UploadInitVO initUpload(UploadInitDTO dto) {
        // 用文件 MD5 作为幂等键，相同文件秒传
        String existingFile = checkFileExists(dto.getMd5());
        if (existingFile != null) {
            log.info("【秒传】文件已存在, md5={}", dto.getMd5());
            UploadInitVO vo = new UploadInitVO();
            vo.setUploadId(null);  // null 表示秒传
            vo.setFileUrl(existingFile);
            vo.setCompleted(true);
            return vo;
        }

        // 生成 uploadId（UUID 或雪花 ID）
        String uploadId = UUID.randomUUID().toString().replace("-", "");
        String chunkDir = CHUNK_TEMP_DIR + uploadId;
        new File(chunkDir).mkdirs();

        // 查询已上传的分片（断点续传）
        List<Integer> uploadedChunks = getUploadedChunks(chunkDir);

        UploadInitVO vo = new UploadInitVO();
        vo.setUploadId(uploadId);
        vo.setUploadedChunks(uploadedChunks);
        vo.setCompleted(false);
        return vo;
    }

    /**
     * 上传分片：直接写磁盘，不进 JVM 堆
     */
    public void uploadChunk(String uploadId, int chunkIndex, MultipartFile chunk) {
        String chunkPath = CHUNK_TEMP_DIR + uploadId + "/" + chunkIndex;
        try {
            // 用 transferTo 直接写磁盘，避免占用堆内存
            chunk.transferTo(new File(chunkPath));
            log.info("【分片上传】uploadId={}, chunkIndex={}, size={}",
                    uploadId, chunkIndex, chunk.getSize());
        } catch (IOException e) {
            log.error("【分片上传失败】uploadId={}, chunkIndex={}", uploadId, chunkIndex, e);
            throw new BizException("分片上传失败");
        }
    }

    /**
     * 合并分片：顺序读取分片文件，流式写入目标文件
     */
    public String mergeChunks(String uploadId) {
        String chunkDir = CHUNK_TEMP_DIR + uploadId;
        File dir = new File(chunkDir);
        File[] chunks = dir.listFiles();
        if (chunks == null || chunks.length == 0) {
            throw new BizException("没有找到分片文件");
        }

        // 按分片序号排序
        Arrays.sort(chunks, Comparator.comparingInt(
                f -> Integer.parseInt(f.getName())));

        // 合并后的文件路径
        String targetPath = "/data/upload/files/" + uploadId + getFileExtension(chunks[0]);

        try (FileOutputStream fos = new FileOutputStream(targetPath)) {
            byte[] buffer = new byte[8192];
            for (File chunk : chunks) {
                try (FileInputStream fis = new FileInputStream(chunk)) {
                    int len;
                    while ((len = fis.read(buffer)) != -1) {
                        fos.write(buffer, 0, len);
                    }
                }
            }
        } catch (IOException e) {
            log.error("【分片合并失败】uploadId={}", uploadId, e);
            throw new BizException("分片合并失败");
        }

        // 清理分片临时文件
        for (File chunk : chunks) {
            chunk.delete();
        }
        dir.delete();

        return targetPath;
    }
}

二、流式下载 + 断点续传 ​

大文件下载的核心是流式传输：不要把整个文件读进内存，而是一边读一边写。

同时支持 HTTP Range 请求，实现断点续传。

java
@GetMapping("/download/{fileId}")
public void download(@PathVariable String fileId,
                     HttpServletRequest request,
                     HttpServletResponse response) throws IOException {
    File file = getRealFile(fileId);
    long fileLength = file.length();

    // 解析 Range 请求头
    String range = request.getHeader("Range");
    long start = 0;
    long end = fileLength - 1;

    if (range != null) {
        // Range: bytes=1024-2048 或 Range: bytes=1024-
        range = range.replace("bytes=", "");
        String[] parts = range.split("-");
        start = Long.parseLong(parts[0]);
        if (parts.length > 1 && !parts[1].isEmpty()) {
            end = Long.parseLong(parts[1]);
        }

        // 设置 206 Partial Content
        response.setStatus(HttpServletResponse.SC_PARTIAL_CONTENT);
    }

    long contentLength = end - start + 1;

    // 设置响应头
    response.setContentType("application/octet-stream");
    response.setHeader("Content-Disposition",
            "attachment; filename=\"" + URLEncoder.encode(file.getName(), "UTF-8") + "\"");
    response.setHeader("Accept-Ranges", "bytes");
    response.setHeader("Content-Length", String.valueOf(contentLength));

    if (range != null) {
        response.setHeader("Content-Range",
                "bytes " + start + "-" + end + "/" + fileLength);
    }

    // 流式写入
    try (RandomAccessFile raf = new RandomAccessFile(file, "r");
         OutputStream out = response.getOutputStream()) {
        raf.seek(start);
        byte[] buffer = new byte[8192];
        long remaining = contentLength;
        int len;

        while (remaining > 0) {
            len = (int) Math.min(buffer.length, remaining);
            int read = raf.read(buffer, 0, len);
            if (read == -1) break;
            out.write(buffer, 0, read);
            remaining -= read;
        }
        out.flush();
    }
}

场景引入 ​

某创业公司的后端开发小王，用AI助手快速完成了用户认证模块。代码能跑，测试通过，上线三个月后被安全研究员发现：密码存储用的是MD5，JWT密钥硬编码在代码里，Token没有过期机制。

AI代码的六大安全风险 ​

1. SQL注入：最常见的漏洞 ​

AI生成的数据库操作代码，特别是"老派"写法，经常存在SQL注入：

企业合规风险 ​

代码版权问题 ​

AI生成的代码是否受版权保护？这是法律灰色地带：

  • 美国版权局倾向于认为"纯AI生成"的内容不受版权保护
  • 企业使用AI代码可能面临来自训练数据中开源代码的GPL等许可证问题
  • 建议：AI生成的代码必须经过人工修改和审查，确保有足够的"人类创造性贡献"

敏感代码泄露 ​

使用云端AI服务时，你的代码会发送到第三方服务器：

  • Copilot、Cursor等都使用云端API
  • 代码中的业务逻辑、算法、架构可能被"学习"
  • 竞争对手可能通过类似Prompt获得相似代码

建议：

  • 核心算法和商业机密代码不要使用云端AI
  • 考虑私有化部署方案
  • 使用企业版AI工具（通常有数据隔离保证）

代码审查清单 ​

必查项（Blocking） ​

  • [ ] 是否存在SQL拼接（必须使用参数化查询）

AI的局限 ​

1. 复杂JOIN查询 ​

涉及3张以上表的复杂查询，AI经常出错：

需求：查询每个用户的最近一笔订单的商品名称和数量

AI可能生成：

sql
-- AI的方案：子查询方式，性能可能不好
SELECT u.name, o.id, p.name, oi.quantity
FROM users u
JOIN orders o ON u.id = o.user_id
JOIN order_items oi ON o.id = oi.order_id
JOIN products p ON oi.product_id = p.id
WHERE o.create_time = (
    SELECT MAX(create_time) FROM orders WHERE user_id = u.id
);

问题：

  • 子查询在大数据量下性能差
  • 一个用户可能有多笔同一时间创建的订单
  • 没有考虑分页

更好的写法需要用窗口函数，但AI不一定能想到：

sql
WITH ranked_orders AS (
    SELECT *, ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY create_time DESC) AS rn
    FROM orders
)
SELECT u.name, ro.id, p.name, oi.quantity
FROM ranked_orders ro
JOIN users u ON u.id = ro.user_id
JOIN order_items oi ON ro.id = oi.order_id
JOIN products p ON oi.product_id = p.id
WHERE ro.rn = 1;

3. 数据迁移SQL ​

涉及数据迁移的SQL，AI很容易忽略数据一致性：

sql

-- 问题：没有考虑并发写入、没有备份、没有分批执行

安全的迁移方式：

sql
-- 1. 先备份

SELECT * FROM orders WHERE status = 3 AND create_time < '2025-01-01';

-- 2. 分批执行
UPDATE orders SET status = 5 
WHERE status = 3 AND create_time < '2025-01-01' 
LIMIT 1000;

-- 3. 验证数据
SELECT COUNT(*) FROM orders WHERE status = 5 AND create_time < '2025-01-01';

AI文档的常见问题 ​

"这段代码谁写的？" "三年前离职的那个。" "有文档吗？" "没有。" "有测试吗？" "也没有。"

每个接手遗留系统的开发者都经历过这段对话。遗留代码的特点是：能跑，没人敢动，没人完全理解。改一行可能崩一片，不改又无法添加新功能。

AI能不能帮我们啃这块硬骨头？这篇文章基于我重构一个5万行遗留Java系统的真实经验。

第一步：让AI帮你理解代码 ​

重构之前必须先理解。AI在代码理解方面的能力远超我的预期。

1. 模块级理解 ​

请分析 src/main/java/com/legacy/order/ 目录下的所有文件，告诉我：

1. 这个模块的整体职责是什么
2. 类与类之间的依赖关系（用文字描述）
3. 核心业务流程是什么（从入口方法开始追踪）
4. 有没有明显的代码异味（God Class、循环依赖等）
5. 有哪些外部依赖（数据库、消息队列、第三方API）

AI给出的分析报告，比自己读两天代码的理解还全面。特别是依赖关系的梳理，AI能画出完整的调用链。

2. 方法级理解 ​

遇到看不懂的方法，直接丢给AI：

并指出可能的边界问题：

遗留代码中最有价值的是"隐含的业务规则"——那些没有文档、藏在if-else里的逻辑：

AI给出的计划通常比自己想的更系统。关键是要让它考虑"不破坏现有功能"这个约束。

真实案例：一次惊险的重构 ​

背景 ​

遗留系统中有一个calculateDiscount方法，300行，包含12个if-else分支。功能是根据用户等级、商品类型、活动规则计算折扣。

重构过程 ​

  1. AI分析：识别出12个分支对应6种用户等级 x 2种折扣逻辑
  2. AI生成测试：基于分析结果生成了24个测试用例
  3. 人工补充：我又补充了8个边界测试
  4. AI重构：将if-else重构为策略模式

惊险时刻 ​

重构后的代码跑测试，24个AI生成的测试全部通过，但我补充的8个边界测试挂了3个。原因是AI在重构时，对"普通用户+限时折扣"的组合处理有误。

教训：AI会忽略一些"不起眼"的业务规则。如果你的测试覆盖不够，AI的重构可能悄悄改变业务行为。

AI可能引入的回归问题 ​

1. 丢失边界条件处理 ​

java
// 原代码有null检查
if (user != null && user.getAddress() != null) {
    // 处理地址
}

// AI重构后可能丢失
processAddress(user.getAddress()); // NPE风险

2. 改变异常处理语义 ​

java
// 原代码吞掉异常返回默认值
try {
    return calculate(data);
} catch (Exception e) {
    log.warn("计算失败，使用默认值", e);
    return defaultValue;
}

// AI重构后可能改成抛出异常
return calculate(data); // 改变了调用方预期的行为

3. 改变事务边界 ​

原代码可能在一个事务中，AI拆分后可能导致事务不一致。这是最危险的回归问题，因为测试不一定能发现。

重构检查清单 ​

每次AI重构后，我都会按这个清单检查：

  • [ ] 所有原有测试仍然通过
  • [ ] 新增测试覆盖了重构的代码
  • [ ] 所有public方法签名未变（或已更新所有调用方）
  • [ ] 异常处理语义未变
  • [ ] 事务边界未变
  • [ ] 日志输出的关键信息未丢失
  • [ ] 没有引入新的依赖版本冲突

总结 ​

AI辅助重构遗留代码的核心价值是降低认知负担——帮你理解代码、生成测试、执行重构。但它不能替代你的业务判断。

最佳实践：

  1. 先理解再动手：让AI帮你全面分析，不要直接重构
  2. 测试先行：重构前先补充测试，特别是边界测试
  3. 小步前进：每次只改一小块，验证后再继续
  4. 人工Review：AI重构的代码，逐行Review异常处理和边界条件
  5. 保留回退能力：每个阶段独立提交，随时可以回退

AI不会让遗留代码变成艺术品，但能让"不敢碰"变成"可以改"。

Embedding 向量嵌入技术通俗理解 ​

前言 ​

如果你了解 RAG（检索增强生成），或者听说过"向量数据库"，那你一定遇到过 "Embedding" 这个词。它是 AI 应用开发中最基础也最核心的技术之一。

这篇文章用最通俗的方式，帮你理解 Embedding 到底在干什么。

什么是 Embedding？ ​

一句话：Embedding 就是把文本变成一组数字（向量），让计算机可以计算文本之间的"相似度"。

举个生活中的例子：

假设你要在图书馆找一本关于"前端框架 React 最佳实践"的书。如果图书馆只有按字母排序的目录，你得一本一本翻。但如果每本书都被映射到了一个"知识地图"上的某个位置，而且"越相似的书在地图上离得越近"，那你只需要去"React/前端"那片区域找就行了。

Embedding 就是这张"知识地图"的坐标。

一段文本经过 Embedding 模型处理后，会变成一个由数字组成的向量。比如：

"今天天气真好" → [0.12, -0.34, 0.56, 0.78, ..., -0.91]  (1536个数字)
"今天阳光明媚" → [0.11, -0.33, 0.55, 0.80, ..., -0.89]  (很接近的向量)
"Python并发编程" → [-0.45, 0.67, 0.12, -0.23, ..., 0.34]  (差异很大的向量)

注意看：语义相似的文本，向量的数值模式也相似；语义不同的文本，向量差异很大。

为什么要变成向量？ ​

你可能会问：用关键词搜索不就行了？为什么要这么麻烦转成向量？

关键词搜索有一个根本问题：它只认字面，不认意思。

2026年的今天，AI编程助手已经从"锦上添花"变成了开发者的"刚需"。作为一个在中小厂摸爬滚打多年的开发者，我深度使用Cursor和GitHub Copilot超过一年。这篇文章不是官方文档的翻译，而是基于真实项目中的使用体验，帮你判断哪个工具更适合你的团队。

工具背景 ​

GitHub Copilot：微软/GitHub出品，依托GitHub海量开源代码训练，与VS Code、JetBrains等IDE深度集成。目前定价$19/月（个人版）。

Cursor：基于VS Code fork的AI原生编辑器，集成了GPT-4、Claude等多个模型，主打Agent能力。Pro版$20/月，提供500次快速请求。

核心能力对比 ​

1. 代码补全 ​

Copilot的Tab补全是其核心卖点。在写Java业务代码时，它能根据方法名和上下文准确预测接下来几行：

java

// Copilot 自动补全
    User user = userRepository.findById(userId)
        .orElseThrow(() -> new BusinessException("用户不存在"));
    return UserConverter.toDTO(user);
}

Cursor的补全能力稍弱，但它的Tab补全会结合项目中的其他文件，对项目特定的工具类、枚举使用更准确。

实测结论：写标准CRUD代码，Copilot准确率约75%，Cursor约65%。但Cursor在理解项目上下文方面更强。

这是两者差距最大的地方。

Copilot Chat：集成在侧边栏，响应速度快，但上下文窗口有限。在解释一段500行的遗留代码时，经常丢失上下文。

Cursor能同时理解两个文件的关系，给出有业务逻辑深度的分析。Copilot Chat在同一问题下，经常只分析其中一个文件。

这是Cursor的绝对优势。Cursor的Composer/Agent模式可以：

  • 同时修改多个文件
  • 自动执行终端命令
  • 根据错误信息自动修复

线上数据库性能问题排查

问题场景 ​

凌晨2点被告警叫醒：数据库CPU飙到95%，大量接口超时。后端没有DBA，你就是DBA。

此刻你需要一套清晰的排查思路：先止血，再定位，后优化。

这篇文章记录的是我们团队在没有专职DBA的情况下，后端开发自己排查数据库性能问题的完整方法论。每个命令、每条路径都是真实踩过的坑。

分析思路 ​

第一步：止血——快速判断当前状态 ​

登录MySQL后，先看几个关键指标：

sql
-- 1. 查看当前连接数和状态
SHOW GLOBAL STATUS LIKE 'Threads%';
-- Threads_connected: 当前打开的连接数
-- Threads_running: 正在执行的连接数（这个值大于CPU核数就有问题）

-- 2. 查看是否有锁等待
SELECT * FROM information_schema.INNODB_TRX ORDER BY trx_started;
-- 长时间运行的事务可能是罪魁祸首

-- 3. 查看InnoDB状态摘要
SHOW ENGINE INNODB STATUS\G
-- 重点看 SEMAPHORES、TRANSACTIONS、LATEST DETECTED DEADLOCK 三个部分

如果发现有长时间运行的事务（超过60秒），先Kill掉止血：

sql
-- 找到长事务
SELECT trx_id, trx_state, trx_started, TIMESTAMPDIFF(SECOND, trx_started, NOW()) AS duration,
       trx_mysql_thread_id, trx_query
FROM information_schema.INNODB_TRX
ORDER BY trx_started;

-- Kill掉
KILL <thread_id>;

第二步：定位——慢查询分析 ​

1. 开启慢查询日志（如果没开的话）

sql
-- 动态开启（重启后失效，记得补到配置文件）
SET GLOBAL slow_query_log = ON;
SET GLOBAL long_query_time = 1;
SET GLOBAL log_queries_not_using_indexes = ON;

2. 用 mysqldumpslow 快速看TOP SQL

bash
# 按执行时间排序，取前10条
mysqldumpslow -s t -t 10 /var/log/mysql/slow.log

# 按执行次数排序
mysqldumpslow -s c -t 10 /var/log/mysql/slow.log

# 按返回行数排序
mysqldumpslow -s r -t 10 /var/log/mysql/slow.log

3. 用 pt-query-digest 深度分析

mysqldumpslow 太简单了，pt-query-digest 才是真正的神器。

bash
# 安装
yum install percona-toolkit

# 分析慢查询日志
pt-query-digest /var/log/mysql/slow.log \
  --since '2026-05-24 00:00:00' \
  --until '2026-05-25 00:00:00' \
  --limit 20 \
  --order-by Query_time:avg

输出示例：

# Profile
# Rank Query ID                     Response time  Calls  R/Call  V/M
# ==== ============================== ============= ====== ======= =====
#    1 0xABC123DEF456                 2345.0000  45.2%   8234  0.2848  0.05 SELECT t_order
#    2 0x789GHI012JKL                 1200.0000  23.1%   5621  0.2135  0.03 SELECT t_user
#    3 0xMNO345PQR678                  890.0000  17.1%    234  3.8034  2.15 UPDATE t_inventory

关键字段解读：

  • Rank：按总响应时间排序
  • Response time：总响应时间和占比
  • Calls：执行次数
  • R/Call：平均每次执行时间
  • V/M：方差/均值比，越大说明执行时间波动越大（可能偶尔很慢）

往下看每个SQL的详细分析：

# Query 1: 2.34 QPS, 0.2848s avg time, 45.2% of total
# Scores: V/M = 0.05
# Time range: 2026-05-24T00:00:01 to 2026-05-24T23:59:58
# Attribute    pct   total     min     max     avg     95%  stddev  median
# ============ === ======= ======= ======= ======= ======= ======= =======
# Count         45    8234
# Exec time     45   2345s    0.1s    2.3s   0.28s   0.52s    0.1s   0.25s
# Lock time     30     12s       0   0.05s   0.001s  0.002s   0.001  0.001s
# Rows sent     80  1.23M       1    2000    150     500     200     100
# Rows examine  90  15.6M      50   50000   1894    5000    3000    1200

重点看 Rows examine 和 Rows sent 的比值。如果 examine 远大于 sent，说明做了大量无用扫描。

第三步：锁等待排查 ​

CPU高不一定全是慢SQL，锁等待也会导致大量线程排队。

查看当前锁等待：

sql

前言 ​

之前做一个电商后台管理系统，产品经理提了个需求：订单状态变化时，运营人员的页面要实时更新。最初方案是前端轮询（每 3 秒请求一次），结果 200 个运营同时在线时，每秒多了 60 多个无效请求，服务端压力不小。

核心需求：

  1. 订单状态变更时，推送给负责该订单的运营
  2. 支持按角色（客服、仓储、财务）分频道推送
  3. 支持集群部署（4 台机器），消息不能丢
  4. 客户端断线后自动重连

一、基础 WebSocket + STOMP 配置 ​

STOMP 是 WebSocket 上的文本协议，优点是 Spring 原生支持，支持点对点和广播两种模式，还能配合 SockJS 做降级（不支持 WebSocket 的旧浏览器可以回退到长轮询）。

java
@Configuration
@EnableWebSocketMessageBroker
public class WebSocketConfig implements WebSocketMessageBrokerConfigurer {

    @Override
    public void configureMessageBroker(MessageBrokerRegistry config) {
        // 客户端订阅的目的地前缀（从服务端接收消息）
        config.enableSimpleBroker("/topic", "/queue")
                // 心跳间隔：客户端和服务端每 10 秒互发心跳
                .setHeartbeatValue(new long[]{10000, 10000});
        // 客户端发送消息的目的地前缀（发送到服务端）
        config.setApplicationDestinationPrefixes("/app");
        // 点对点消息前缀
        config.setUserDestinationPrefix("/user");
    }

    @Override
    public void registerStompEndpoints(StompEndpointRegistry registry) {
        registry.addEndpoint("/ws")
                // 允许的跨域来源
                .setAllowedOriginPatterns("*")
                // 开启 SockJS 降级

class WebSocketClient {
    constructor() {
        this.reconnectAttempts = 0;
        this.maxReconnectAttempts = 10;
    }

    connect(token) {
        const socket = new SockJS('/ws');
        this.stompClient = Stomp.over(socket);
        // 关闭 STOMP 的 debug 日志
        this.stompClient.debug = null;

        this.stompClient.connect(
            { Authorization: 'Bearer ' + token },
            (frame) => {
                console.log('WebSocket 已连接');
                this.reconnectAttempts = 0;

                // 订阅订单状态变更
                this.stompClient.subscribe('/topic/orders', (message) => {
                    const order = JSON.parse(message.body);
                    this.onOrderUpdate(order);
                });

                // 订阅个人通知
                this.stompClient.subscribe('/user/queue/notify', (message) => {
                    const notify = JSON.parse(message.body);
                    this.onNotify(notify);
                });
            },
            (error) => {
                console.error('WebSocket 连接失败', error);
                this.reconnect();
            }
        );
    }

    reconnect() {
        if (this.reconnectAttempts >= this.maxReconnectAttempts) {
            console.error('重连次数超限，放弃重连');
            return;
        }
        // 指数退避重连
        const delay = Math.min(1000 * Math.pow(2, this.reconnectAttempts), 30000);
        this.reconnectAttempts++;
        console.log(`第${this.reconnectAttempts}次重连，等待${delay}ms`);
        setTimeout(() => this.connect(), delay);
    }
}

java
@Service
@Slf4j
public class ClusterNotifyService {

    @Autowired
    private SimpMessagingTemplate messagingTemplate;

    @Autowired
    private StringRedisTemplate redisTemplate;

    private static final String CHANNEL = "ws:broadcast";

    @PostConstruct
    public void subscribe() {
        // 启动 Redis 消息监听
        redisTemplate.getConnectionFactory()
                .getConnection()
                .subscribe(new MessageListener() {
                    @Override
                    public void onMessage(byte[] message, byte[] pattern) {
                        try {
                            WsMessage wsMsg = JSON.parseObject(
                                    new String(message), WsMessage.class);
                            // 转发给本机的 WebSocket 客户端
                            messagingTemplate.convertAndSend(
                                    wsMsg.getDestination(), wsMsg.getPayload());
                        } catch (Exception e) {
                            log.error("【WS集群广播】消息处理失败", e);
                        }
                    }
                }, CHANNEL.getBytes());
    }

    /**
     * 广播消息到所有集群节点
     */
    public void broadcastCluster(String destination, Object payload) {
        WsMessage wsMsg = new WsMessage(destination, JSON.toJSONString(payload));
        redisTemplate.convertAndSend(CHANNEL, JSON.toJSONString(wsMsg));
    }
}

@Component
@Slf4j
public class WebSocketSessionManager implements ApplicationListener<SessionConnectedEvent>,
        ApplicationListener<SessionDisconnectEvent> {

    // 用户ID -> SessionId 映射
    private final ConcurrentHashMap<String, String> userSessions = new ConcurrentHashMap<>();

    @Override
    public void onApplicationEvent(SessionConnectedEvent event) {
        String userId = extractUserId(event);
        String sessionId = event.getMessage().getHeaders()
                .get("simpSessionId", String.class);
        userSessions.put(userId, sessionId);
        log.info("【WS连接】用户={}, sessionId={}", userId, sessionId);
    }

    @Override
    public void onApplicationEvent(SessionDisconnectEvent event) {
        String sessionId = event.getSessionId();
        userSessions.entrySet().removeIf(entry ->
                entry.getValue().equals(sessionId));
        log.info("【WS断开】sessionId={}", sessionId);
    }

    public boolean isOnline(String userId) {
        return userSessions.containsKey(userId);
    }
}

我们的用户服务有 4 台机器，用 Nginx 做负载均衡。每次发布（逐台滚动重启）时，总有少量用户报 502 Bad Gateway。运维说"发布时有个别请求失败是正常的"，但产品不接受。

场景二：某个接口偶尔超时 30 秒 ​

一个导入功能的接口偶尔需要 20~30 秒处理大量数据。正常情况下 Nginx 默认 60 秒超时没问题，但实际测试发现经常 30 秒就断了。

场景三：ip_hash 导致流量不均 ​

用了 ip_hash 做会话保持，但客户大多在同一个 NAT 后面，导致某台机器被打爆。

分析思路 ​

Nginx 负载均衡的核心配置就两个部分：upstream（定义后端服务器组）和 proxy_pass（转发请求）。先把最常用的配置梳理清楚，再解决上述问题。

TCP长连接与短连接的选择

前言 ​

很多 Java 后端开发在写 HTTP 接口调用时，对 TCP 连接的生命周期几乎没有感知。Spring RestTemplate 一包，URL 一填，请求就发出去了。直到线上出现"偶发 Connection Reset"、"连接被对端关闭"之类的报错，才会开始关注底层连接到底是怎么管理的。

这篇文章不讲 TCP 三次握手四次挥手的理论（面试八股已经够多了），直接从什么场景该用长连接、什么场景该用短连接出发，结合实际项目中的坑来说明。

问题场景 ​

我们有一个中台服务 A，需要同时调用三种下游：

  1. 内部用户服务 B：同机房，调用频率极高（每秒上千次），接口响应快（5ms 以内）
  2. 第三方短信服务 C：外部公网，调用频率低（每天几万次），偶尔慢（1~3 秒）
  3. 内部大数据服务 D：调用频率低但单次传输数据量大（响应体 50MB+）

三种场景的网络特性完全不同，用同一套连接策略肯定不行。

分析思路 ​

先理清长连接和短连接的本质区别：

短连接：每次 HTTP 请求都要经历 TCP 三次握手 → 数据传输 → 四次挥手。开销大，但连接生命周期短，不会被"脏"数据影响。

长连接（Keep-Alive）：TCP 连接建立后不立即关闭，后续请求复用同一个连接。省去了握手和挥手的开销，但引入了连接管理的复杂性。

选择的依据其实就是三个因素：调用频率、响应延迟、连接稳定性。

问题场景 ​

作为Java后端开发，日常有很多重复性工作：批量修改数据、分析线上日志、给测试人员准备数据、从Excel导入配置。这些事情用Java写太重，手动操作又容易出错。Python是这类场景的最佳选择——几行代码就能搞定。

数据库批量操作 ​

场景：批量更新用户状态

产品说要把3个月没登录的用户标记为"休眠"，涉及10万条数据。手动在数据库里写SQL？万一改错了回不回来。

python
# batch_update_users.py
import pymysql
from datetime import datetime, timedelta

# 数据库连接配置
DB_CONFIG = {
    'host': '10.0.1.100',
    'port': 3306,
    'user': 'readonly_user',  # 先用readonly验证数量
    'password': 'xxx',
    'database': 'user_db',
    'charset': 'utf8mb4'
}

def count_inactive_users():
    """统计不活跃用户数量，先确认范围"""
    conn = pymysql.connect(**DB_CONFIG)
    try:
        with conn.cursor() as cursor:
            cutoff = datetime.now() - timedelta(days=90)
            sql = """
                SELECT COUNT(*) FROM t_user 
                WHERE last_login_time < %s AND status = 'ACTIVE'
            """
            cursor.execute(sql, (cutoff,))
            count = cursor.fetchone()[0]
            print(f"待更新用户数: {count}")
            return count
    finally:
        conn.close()

def batch_update(batch_size=1000):
    """分批更新，每批1000条，避免长事务"""
    config = DB_CONFIG.copy()
    config['user'] = 'write_user'  # 更新用写账号
    
    conn = pymysql.connect(**config)
    cutoff = datetime.now() - timedelta(days=90)
    
    try:
        total_updated = 0
        while True:
            with conn.cursor() as cursor:
                # 每次只更新batch_size条
                sql = """
                    UPDATE t_user 
                    SET status = 'DORMANT', updated_at = NOW() 
                    WHERE last_login_time < %s AND status = 'ACTIVE'
                    LIMIT %s
                """
                affected = cursor.execute(sql, (cutoff, batch_size))
                conn.commit()
                
                total_updated += affected
                print(f"已更新: {total_updated}, 本批: {affected}")
                
                if affected == 0:
                    break
        
        print(f"全部完成，共更新 {total_updated} 条")
    except Exception as e:
        conn.rollback()
        print(f"执行失败: {e}")
    finally:
        conn.close()

if __name__ == '__main__':
    count = count_inactive_users()
    if count > 0:
        confirm = input(f"确认更新 {count} 条数据？(y/n): ")
        if confirm == 'y':
            batch_update()

关键设计：

  • 先用只读账号统计数量，人工确认后再用写账号执行
  • 分批执行（LIMIT），避免一次更新10万行锁表
  • 每批单独commit，出错只回滚当前批次

日志分析 ​

场景：统计每个接口的QPS和错误率

python
# log_analyzer.py
import re
from collections import defaultdict
from datetime import datetime

# 日志格式: 2026-05-25 14:30:15.123 [http-nio-8080-exec-1] INFO OrderController - GET /api/orders/123 200 45ms
LOG_PATTERN = re.compile(
    r'(\d{4}-\d{2}-\d{2} \d{2}:\d{2}:\d{2}).*'
    r'(GET|POST|PUT|DELETE)\s+(/\S+)\s+(\d{3})\s+(\d+)ms'
)

def analyze_log(file_path):
    """分析日志文件，统计接口性能"""
    stats = defaultdict(lambda: {
        'count': 0, 'errors': 0, 'total_ms': 0, 'max_ms': 0
    })
    
    with open(file_path, 'r', encoding='utf-8') as f:
        for line in f:
            match = LOG_PATTERN.search(line)
            if match:
                time_str, method, path, status, cost_ms = match.groups()
                cost_ms = int(cost_ms)
                status = int(status)
                
                # 归一化路径（把ID参数替换为占位符）
                normalized_path = re.sub(r'/\d+', '/{id}', path)
                key = f"{method} {normalized_path}"
                
                stats[key]['count'] += 1
                stats[key]['total_ms'] += cost_ms
                if status >= 400:
                    stats[key]['errors'] += 1
                if cost_ms > stats[key]['max_ms']:
                    stats[key]['max_ms'] = cost_ms
    
    # 按调用次数排序输出
    print(f"{'接口':<40} {'调用次数':>8} {'错误数':>6} {'平均耗时':>8} {'最大耗时':>8} {'错误率':>8}")
    print("-" * 100)
    
    for path, data in sorted(stats.items(), key=lambda x: -x[1]['count']):
        avg_ms = data['total_ms'] / data['count'] if data['count'] > 0 else 0
        error_rate = data['errors'] / data['count'] * 100 if data['count'] > 0 else 0
        print(f"{path:<40} {data['count']:>8} {data['errors']:>6} {avg_ms:>7.0f}ms {data['max_ms']:>6}ms {error_rate:>7.2f}%")

if __name__ == '__main__':
    analyze_log('/data/logs/app/app.log')

线上网络问题排查工具箱

前言 ​

线上网络问题是最让人头疼的一类故障。CPU 高了可以看火焰图，内存溢出了可以 dump 分析，但网络问题——"接口偶尔超时"、"连接突然断开"、"部分用户反馈打不开页面"——这些描述本身就模糊，排查起来经常像无头苍蝇。

这篇文章整理了我在过去两年中反复使用的网络排查工具和方法，附带三个真实案例。工具不在多，在于用得熟练。

工具一：curl — 最基础的排查利器 ​

curl 不只是"发个请求"那么简单，它的 -v（verbose）和 -w（write-out）选项能提供大量诊断信息。

查看完整的请求/响应过程 ​

bash
curl -v -o /dev/null https://api.example.com/health 2>&1

# 关注这些关键信息：
# * TCP_NODELAY set           → TCP 连接开始
# * Connected to ...          → TCP 连接建立成功
# * ALPN, server accepted to use h2  → 协议协商（HTTP/2）
# * SSL connection using TLSv1.3     → TLS 握手信息
# < HTTP/2 200                → 响应状态码
# < content-type: ...         → 响应头

测量请求各阶段耗时 ​

bash
curl -o /dev/null -s -w "
DNS解析:       %{time_namelookup}s
TCP连接:       %{time_connect}s
TLS握手:       %{time_appconnect}s
首字节时间:     %{time_starttransfer}s
总耗时:        %{time_total}s
下载大小:      %{size_download} bytes
HTTP状态码:    %{http_code}
" https://api.example.com/health

# 典型输出：
# DNS解析:       0.012s
# TCP连接:       0.015s
# TLS握手:       0.089s
# 首字节时间:     0.156s
# 总耗时:        0.167s

如果 DNS 解析时间长，说明 DNS 有问题。如果 TCP 连接时间长，说明网络不通畅或目标端口有问题。如果 TLS 握手时间长，说明证书配置或 TLS 版本有优化空间。

模拟特定请求 ​

bash

工具二：tcpdump — 抓包之王 ​

tcpdump 是 Linux 上最强大的抓包工具。很多网络问题只有看了抓包才能确认根因。

常用抓包命令 ​

bash
# 抓取某个端口的流量
tcpdump -i eth0 port 8080 -nn -c 100

# 抓取特定主机之间的流量
tcpdump -i eth0 host 10.0.1.10 and port 8080 -nn

# 抓取 TCP SYN 包（排查连接建立问题）
tcpdump -i eth0 'tcp[tcpflags] & (tcp-syn) != 0' -nn

# 抓取 TCP RST 包（排查连接被重置问题）
tcpdump -i eth0 'tcp[tcpflags] & (tcp-rst) != 0' -nn

# 抓取 HTTP 请求（明文 HTTP）
tcpdump -i eth0 port 80 -A -s 0 | grep -E "^(GET|POST|PUT|DELETE)"

# 保存到文件，用 Wireshark 分析
tcpdump -i eth0 port 8080 -w /tmp/capture.pcap -c 1000

实战：排查连接超时 ​

真实案例一：偶发接口超时（DNS 问题） ​

现象：用户服务调用地址服务，P99 从 10ms 偶尔飙到 500ms。

排查过程：

bash
# 1. 用 arthas 追踪接口耗时
monitor -c 10 com.example.service.AddressService resolve

# 发现：大部分请求 5ms，偶尔 300~500ms

# 2. 用 arthas 追踪 Socket 连接
watch java.net.Socket connect '{params, #cost}' -n 50

# 发现：慢请求的 Socket.connect 耗时 300ms+

# 3. 怀疑是 DNS 问题，用 tcpdump 验证
tcpdump -i eth0 port 53 -nn -tttt

# 发现：某些 DNS 查询确实要 300ms+，且都是对内网域名的查询

# 4. 检查 Java DNS 缓存配置
# 发现 -Dsun.net.inetaddr.ttl 没有设置（默认 -1，永久缓存）
# 意味着 DNS 结果一旦过期，重新查询就可能触发慢解析

解决：设置 JVM 参数 -Dsun.net.inetaddr.ttl=30，同时优化内网 DNS 服务器配置。

真实案例二：连接被重置（RST） ​

现象：调用第三方支付接口，偶发 Connection reset by peer。

排查过程：

bash
# 1. tcpdump 抓 RST 包
tcpdump -i eth0 host pay-api.third-party.com and 'tcp[tcpflags] & (tcp-rst) != 0' -nn

# 发现：RST 来自本机，不是远端

# 2. 检查连接池配置
# 发现 OkHttp 连接池空闲超时 5 分钟，但第三方 keepalive_timeout 是 30 秒
# 第三方在 30 秒后关闭连接，OkHttp 不知道，继续使用已关闭的连接

# 3. Wireshark 分析确认
# 用 tcpdump 抓包后用 Wireshark 打开
# 确认：本机在连接被服务端关闭 40 秒后发了一个请求，服务端返回 RST

解决：将 OkHttp 的空闲连接回收时间设为 25 秒（小于第三方的 30 秒），同时开启 retryOnConnectionFailure(true)。

真实案例三：部分用户 HTTPS 握手失败 ​

现象：部分用户反馈网站打不开，大部分用户正常。

排查过程：

bash
# 1. 用不同方式测试
curl -v https://www.example.com
# 正常

curl -v --tls-max 1.2 https://www.example.com
# 正常

openssl s_client -connect www.example.com:443 -tls1
# TLS 1.0 握手失败（符合预期，我们禁用了 TLS 1.0）

# 2. 让报错的用户提供 User-Agent
# 发现是 Android 4.4 的 WebView，只支持 TLS 1.0

# 3. 但同一台手机的 Chrome 可以正常访问
# 原因：Chrome 内置了 TLS 1.2 支持，但 WebView 用的是系统库

# 4. 检查 Nginx 证书链配置
openssl s_client -connect www.example.com:443 -showcerts
# 发现中间证书缺失，部分旧客户端无法自动下载中间证书

解决：修复证书链（使用 fullchain.pem 而不是 domain.crt），同时评估是否支持 TLS 1.0（安全原因，最终决定不支持，引导用户升级浏览器）。

排查流程总结 ​

遇到线上网络问题，按照这个流程来：

1. 确认问题范围
   ├── 是所有用户还是部分用户？
   ├── 是所有接口还是特定接口？
   └── 是持续的还是偶发的？

2. 快速诊断
   ├── curl -v 测试目标接口
   ├── curl -w 测量各阶段耗时
   └── 确定卡在哪个阶段（DNS / TCP / TLS / HTTP）

3. 深入排查
   ├── DNS 问题 → dig + 检查 /etc/resolv.conf
   ├── TCP 问题 → tcpdump + Wireshark
   ├── TLS 问题 → openssl s_client + Wireshark
   └── 应用层问题 → arthas + 日志

4. 根因确认
   └── 能复现的问题才是能解决的问题

DNS解析问题排查与优化

前言 ​

有一次线上问题排查了两天，最后发现根因居然是 DNS 解析慢。事情是这样的：我们的一个微服务调用另一个内部服务的域名（比如 user-service.internal），大部分请求 5ms 就能返回，但 P99 偶尔飙到 500ms。CPU、内存、GC 全部正常，网络也不丢包，日志也没异常。

最终用 arthas 抓到慢请求全部卡在 Socket.connect 上。进一步排查发现，是 DNS 解析偶尔要 300~400ms。

这篇文章就从这个案例出发，系统地讲讲 Java 后端开发需要了解的 DNS 知识。

问题场景 ​

我们的架构是典型的微服务，服务间通过域名（非 IP）互调。内网有自建 DNS 服务器。问题表现：

  • 同一个接口，99% 的请求正常，1% 的响应时间突然飙到 500ms+
  • 问题随机出现，不跟随后端节点
  • 用 IP 直连替代域名后，问题消失

这就把问题锁定在了 DNS 解析上。

分析思路 ​

DNS 解析涉及多个环节，任何一个环节出问题都会导致解析慢或失败：

  1. JVM DNS 缓存：Java 自带 DNS 缓存机制
  2. 操作系统 DNS 缓存：Linux 的 nscd 或 systemd-resolved
  3. 内网 DNS 服务器：公司自建 DNS 的性能和可用性
  4. DNS 查询链路：客户端 → 本地缓存 → 递归 DNS → 权威 DNS

具体实现 ​

第一步：确认是 DNS 问题 ​

bash
# 用 dig 测量 DNS 解析时间
dig user-service.internal +stats

# 关注这一行：Query time: 342 msec
# 如果偶尔出现 300ms+ 的查询，说明 DNS 服务器有问题

# 或者用 time 包裹 nslookup
time nslookup user-service.internal

在 Java 代码中确认：

java
// 用 arthas 追踪 DNS 解析耗时
// watch java.net.InetAddress getAllByName0 '{params, returnObj, throwExp}' -n 20 -x 2

第二步：检查 Java DNS 缓存配置 ​

Java 的 DNS 缓存是很多问题的根源。默认行为是永久缓存成功解析的结果，这意味着：

  • 如果 DNS 记录变更了（比如服务迁移到新 IP），Java 应用不会感知
  • 如果第一次解析失败，也会被缓存（默认 10 秒）

查看和修改缓存 TTL：

java
// 查看默认 TTL
System.out.println("成功解析缓存 TTL: " + sun.net.InetAddressCachePolicy.get());
System.out.println("失败解析缓存 TTL: " + sun.net.InetAddressCachePolicy.getNegative());

// 通过 JVM 参数调整（推荐）
// -Dsun.net.inetaddr.ttl=30         成功解析缓存 30 秒
// -Dsun.net.inetaddr.negative.ttl=5 失败解析缓存 5 秒

也可以在代码中动态设置：

java
// 在应用启动时调用
// 缓存成功结果 30 秒（而不是默认的永久缓存）
sun.net.InetAddressCachePolicy.set(30);
// 缓存失败结果 5 秒
sun.net.InetAddressCachePolicy.setNegative(5);

踩坑点：在 Java 9+ 中，sun.net.InetAddressCachePolicy 的内部实现有变化。更推荐的做法是通过 SecurityManager 或自定义 InetAddress 的 NameService。在 Spring Boot 项目中，最简单的方式是 JVM 启动参数：

bash
java -Dsun.net.inetaddr.ttl=30 \
     -Dsun.net.inetaddr.negative.ttl=5 \
     -jar app.jar

第三步：内网 DNS 配置优化 ​

排查发现我们内网 DNS 服务器（bind9）的一个 zone 文件配置有问题：部分域名的递归查询超时设置太长。

第四步：应用层 DNS 解析优化 ​

对于服务间调用，我们最终采用了本地 DNS 缓存 + 异步刷新的方案：

java
@Component
@Slf4j
public class DnsCacheManager {

    // 本地缓存：域名 -> IP 列表
    private final LoadingCache<String, List<String>> dnsCache;

    public DnsCacheManager() {
        dnsCache = CacheBuilder.newBuilder()
                // 缓存 60 秒后自动刷新
                .expireAfterWrite(60, TimeUnit.SECONDS)
                // 异步刷新，刷新期间返回旧值
                .refreshAfterWrite(30, TimeUnit.SECONDS)
                .build(new CacheLoader<>() {
                    @Override
                    public List<String> load(String hostname) throws UnknownHostException {
                        InetAddress[] addresses = InetAddress.getAllByName(hostname);
                        List<String> ips = Arrays.stream(addresses)
                                .map(InetAddress::getHostAddress)
                                .collect(Collectors.toList());
                        log.info("【DNS解析】{} -> {}", hostname, ips);
                        return ips;
                    }
                });
    }

    public String resolve(String hostname) {
        try {
            List<String> ips = dnsCache.get(hostname);
            // 简单轮询，生产环境可以用更复杂的策略
            return ips.get(ThreadLocalRandom.current().nextInt(ips.size()));
        } catch (ExecutionException e) {
            log.error("【DNS解析失败】{}", hostname, e);
            return hostname;
        }
    }
}

除了证书过期这种低级错误，HTTPS 相关的问题其实还不少：

  • TLS 握手慢：外网用户首次访问延迟高，TCP 连接建好后 TLS 握手还要 1~2 个 RTT
  • 证书链不完整：部分浏览器或旧版 Android 报"不安全"，抓包发现是中间证书缺失
  • TLS 版本不兼容：升级到 TLS 1.3 后，某些老客户端连不上

具体实现 ​

一、Let's Encrypt 证书自动续期 ​

我们用 certbot 管理 Let's Encrypt 证书。正确的自动续期配置如下：

bash
# 安装 certbot
apt install certbot python3-certbot-nginx

# 首次申请证书
certbot --nginx -d api.example.com -d www.example.com

# 测试续期
certbot renew --dry-run

# 自动续期 cron job（每天凌晨 3 点检查）
# /etc/cron.d/certbot-renew
0 3 * * * root certbot renew --quiet --deploy-hook "systemctl reload nginx"

踩坑教训：--deploy-hook 很关键。证书续期后必须 reload Nginx 才能生效。我们之前只配了 renew，没配 deploy-hook，导致新证书生成了但 Nginx 还在用旧证书。

更安全的做法是写个检查脚本：

bash
#!/bin/bash
# /opt/scripts/check-cert.sh
DOMAIN="api.example.com"
EXPIRE_DATE=$(echo | openssl s_client -servername $DOMAIN -connect $DOMAIN:443 2>/dev/null \
  | openssl x509 -noout -enddate | cut -d= -f2)
EXPIRE_TS=$(date -d "$EXPIRE_DATE" +%s)
NOW_TS=$(date +%s)
DAYS_LEFT=$(( ($EXPIRE_TS - $NOW_TS) / 86400 ))

if [ $DAYS_LEFT -lt 7 ]; then
    echo "【证书告警】${DOMAIN} 证书将在 ${DAYS_LEFT} 天后过期" | \
      mail -s "HTTPS证书即将过期" ops@example.com
fi

if [ $DAYS_LEFT -lt 0 ]; then
    echo "【证书已过期】${DOMAIN}" | \
      mail -s "HTTPS证书已过期！" ops@example.com
fi

二、TLS 版本选择：1.2 vs 1.3 ​

我们的选择：同时支持 TLS 1.2 和 TLS 1.3。

原因很实际：虽然 TLS 1.3 更快更安全，但还有一部分老客户端（Android 4.x、旧版 Java 8 等）不支持。直接只开 TLS 1.3 会导致这些用户无法访问。

Nginx 配置：

nginx
server {
    listen 443 ssl http2;
    server_name api.example.com;

    # 证书
    ssl_certificate /etc/letsencrypt/live/api.example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/api.example.com/privkey.pem;

    # TLS 版本：同时支持 1.2 和 1.3
    ssl_protocols TLSv1.2 TLSv1.3;

    # TLS 1.2 密码套件（去除弱密码）
    ssl_ciphers 'ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384';
    ssl_prefer_server_ciphers on;

    # TLS 1.3 的 0-RTT（需谨慎开启）
    # ssl_early_data on;  # 有重放攻击风险，支付类接口不要开

    # 启用 OCSP Stapling
    ssl_stapling on;
    ssl_stapling_verify on;
    ssl_trusted_certificate /etc/letsencrypt/live/api.example.com/chain.pem;
    resolver 8.8.8.8 8.8.4.4 valid=300s;
    resolver_timeout 5s;
}

    accountService.deduct(request.getUserId(), request.getAmount());
}

问题显而易见：如果步骤2成功但步骤3失败，库存扣了、订单创建了，但余额没扣。数据不一致。

必须强一致性的场景：支付扣款 + 订单状态变更。这种钱相关的，不能用最终一致性。

可以用最终一致性的场景：订单创建后扣库存、发短信通知、推消息到ES。这些允许短暂不一致，最终能补偿就行。

结论：我们80%的场景不需要分布式事务，只有20%的核心链路需要。

跨域问题的完整解决方案

前言 ​

"为什么我的接口 Postman 能调通，浏览器调就报跨域错误？"——这大概是每个前端联调时都会问的问题。后端开发通常的做法是在 Controller 上加个 @CrossOrigin，或者搜一段 Nginx 配置粘上去。能用就行，但不一定理解为什么。

这篇文章从 CORS 的原理讲起，把跨域问题一次性讲清楚。

问题场景 ​

我们的系统架构：前端 Vue 应用跑在 localhost:3000，后端 Spring Boot API 跑在 localhost:8080，生产环境通过 Nginx 反向代理，前端和 API 在同一个域名下（www.example.com）。

开发环境跨域是必然的。问题出在：

  1. 开发环境用 @CrossOrigin("*") 能通，但生产环境出现 Cookie 丢失
  2. 部分 POST 请求（带 JSON Body）浏览器报 CORS 错误，但 GET 请求正常
  3. 文件上传接口偶尔报跨域错误

分析思路 ​

要理解跨域，先要理解同源策略。两个 URL 的协议、域名、端口三者全部一致才算同源。任何一个不同，浏览器就会限制跨域请求。

但注意：跨域限制是浏览器的行为。Postman、curl、后端服务间调用不受此限制。

CORS（Cross-Origin Resource Sharing）是浏览器解决跨域的标准方案。核心机制：

  1. 简单请求：GET/HEAD/POST，Content-Type 为 text/plain、multipart/form-data、application/x-www-form-urlencoded，直接发送。
  2. 非简单请求：其他情况（如 POST + application/json、PUT、DELETE），浏览器先发一个 OPTIONS 预检请求（Preflight），询问服务器是否允许。

这就是为什么 GET 请求没问题，POST + JSON 就报错的原因。

具体实现 ​

方案一：Spring Boot 全局 CORS 配置 ​

推荐方式：通过 WebMvcConfigurer 全局配置

java
@Configuration
public class CorsConfig implements WebMvcConfigurer {

    @Override
    public void addCorsMappings(CorsRegistry registry) {
        registry.addMapping("/**")
                // 生产环境不要用 *，要指定具体域名
                .allowedOriginPatterns(
                        "http://localhost:3000",
                        "http://localhost:5173",  // Vite 默认端口
                        "https://www.example.com"
                )
                .allowedMethods("GET", "POST", "PUT", "DELETE", "OPTIONS")
                .allowedHeaders("*")
                // 允许携带 Cookie（前后端分离时必须开）
                .allowCredentials(true)
                // 预检请求缓存时间（秒），减少 OPTIONS 请求
                .maxAge(3600);
    }
}

注意 allowCredentials(true) 和 allowedOrigins("*") 不能同时使用。这是 CORS 规范的限制。如果你同时配了这两个，Spring Boot 会直接报错。解决方法是用 allowedOriginPatterns 替代 allowedOrigins（Spring Boot 2.4+ 支持）。

java
// 错误写法（Spring Boot 会抛异常）
.allowCredentials(true)
.allowedOrigins("*")

// 正确写法
.allowCredentials(true)
.allowedOriginPatterns("*")  // 用 allowedOriginPatterns 替代

方案二：注解方式（针对单个 Controller） ​

java
@RestController
@RequestMapping("/api/user")
// Controller 级别的 CORS 配置
@CrossOrigin(
        origins = "https://www.example.com",
        methods = {RequestMethod.GET, RequestMethod.POST},
        allowCredentials = "true",
        maxAge = 3600
)
public class UserController {

    @GetMapping("/info")
    public Result<UserInfo> getUserInfo() { ... }
}

踩坑：@CrossOrigin 和全局 addCorsMappings 同时使用时，会产生重复的 CORS 头。浏览器遇到重复的 Access-Control-Allow-Origin 会直接拒绝请求。所以二选一，不要混用。

方案三：Nginx 代理跨域（推荐生产环境） ​

生产环境最干净的方案：前端和 API 都通过 Nginx 代理到同一个域名下，从根本上消除跨域。

nginx
server {
    listen 443 ssl;
    server_name www.example.com;

    # 前端静态资源
    location / {
        root /var/www/dist;
        try_files $uri $uri/ /index.html;
    }

    # API 接口代理
    location /api/ {
        proxy_pass http://127.0.0.1:8080;

        # 不需要手动加 CORS 头，因为同源了
        # 如果因为历史原因还是需要加：
        add_header Access-Control-Allow-Origin $http_origin always;
        add_header Access-Control-Allow-Credentials true always;
        add_header Access-Control-Allow-Methods 'GET, POST, PUT, DELETE, OPTIONS' always;
        add_header Access-Control-Allow-Headers 'Content-Type, Authorization, X-Requested-With' always;

        # 处理 OPTIONS 预检请求
        if ($request_method = 'OPTIONS') {
            add_header Access-Control-Allow-Origin $http_origin always;
            add_header Access-Control-Allow-Credentials true always;
            add_header Access-Control-Allow-Methods 'GET, POST, PUT, DELETE, OPTIONS' always;
            add_header Access-Control-Allow-Headers 'Content-Type, Authorization, X-Requested-With' always;
            add_header Access-Control-Max-Age 3600;
            add_header Content-Length 0;
            return 204;
        }
    }
}

Nginx 踩坑：add_header 在 if 块中不会继承外层的 add_header。所以 if 块中必须重复写所有的 add_header。这是 Nginx 的已知行为，不是 bug。

方案四：Spring Cloud Gateway 统一处理 ​

如果用了 Spring Cloud Gateway，可以在网关层统一处理 CORS：

java
@Configuration
public class GatewayCorsConfig {

    @Bean
    public CorsWebFilter corsWebFilter() {
        CorsConfiguration config = new CorsConfiguration();
        config.addAllowedOriginPattern("*");
        config.addAllowedMethod("*");
        config.addAllowedHeader("*");
        config.setAllowCredentials(true);
        config.setMaxAge(3600L);

        UrlBasedCorsConfigurationSource source = new UrlBasedCorsConfigurationSource();
        source.registerCorsConfiguration("/**", config);

        return new CorsWebFilter(source);
    }
}

注意：如果下游服务也配了 CORS，网关又配了一层，就会出现重复头的问题。建议只在网关层配 CORS，下游服务去掉 CORS 配置。

预检请求优化 ​

预检请求（OPTIONS）虽然不带业务数据，但每次非简单请求都会多一次网络往返。优化方法：

  1. 设置 maxAge：告诉浏览器在指定时间内（如 1 小时）不需要再发预检请求
java
.maxAge(3600)  // 缓存 1 小时

  1. 减少不必要的 Content-Type：前端请求尽量用 application/json，不要用自定义的 Content-Type（如 application/x-my-app-v1+json），否则每次都会触发预检。

  2. 优化 Nginx 对 OPTIONS 的响应：让 OPTIONS 请求在 Nginx 层直接返回，不要透传到后端

nginx
if ($request_method = 'OPTIONS') {
    add_header Access-Control-Max-Age 3600;
    add_header Content-Length 0;
    return 204;
}

数据库连接池HikariCP调优

问题场景 ​

线上服务隔三差五报 HikariPool-1 - Connection is not available, request timed out after 30000ms。一开始以为是并发量大，加了 maximumPoolSize 从20调到50，结果更惨——MySQL 报 Too many connections，直接把数据库搞挂了。

连接池这个东西，配置不对就是定时炸弹。

分析思路 ​

1. 先搞清楚连接池的工作原理 ​

HikariCP 的核心机制很简单：维护一个固定大小的连接池，请求来了取连接、用完还回去。关键在于：

  • maximumPoolSize 不是越大越好，它受 MySQL max_connections 限制
  • 连接池大小的合理公式：connections = (core_count * 2) + effective_spindle_count
  • 对于8核SSD服务器，(8 * 2) + 1 = 17，所以20个连接已经够了

那为什么会不够用？ 因为连接被"借"走后没还回来——连接泄漏。

2. HikariCP 核心参数详解 ​

我们的生产配置：

yaml
spring:
  datasource:
    hikari:
      # 最小空闲连接数（默认等于maximumPoolSize）
      minimum-idle: 10
      # 最大连接数
      maximum-pool-size: 20
      # 连接超时时间（从池中获取连接的最大等待时间）
      connection-timeout: 30000
      # 空闲连接最大存活时间（超过此时间的空闲连接会被回收）
      idle-timeout: 600000
      # 连接最大存活时间（强制回收，防止MySQL端超时断开）
      max-lifetime: 1800000
      # 连接泄漏检测阈值（借出超过此时间未归还的连接会被记录日志）
      leak-detection-threshold: 60000
      # 连接测试查询
      connection-test-query: SELECT 1
      # 连接校验超时
      validation-timeout: 5000

每个参数的含义和踩坑点：

maximum-pool-size（最大连接数）

java
// 错误认知：越大越好
// 实际上：每个连接占MySQL的一个线程 + 内存
// 20个应用实例 * 50连接 = 1000连接，MySQL默认max_connections=151
// 不调整MySQL配置，应用层加再多连接也没用

MySQL 端也要配合调整：

sql
-- 查看当前最大连接数
SHOW VARIABLES LIKE 'max_connections';
-- 调整为500（根据实际服务器配置）
SET GLOBAL max_connections = 500;

max-lifetime（最大存活时间）

这个值必须比 MySQL 的 wait_timeout 小，否则MySQL端已经断开了连接，应用还在用，就会报 Communications link failure。

sql
SHOW VARIABLES LIKE 'wait_timeout';  -- 默认28800秒（8小时）

所以我们设 max-lifetime=1800000（30分钟），远小于8小时。

leak-detection-threshold（泄漏检测阈值）

这个参数是排查连接泄漏的神器。设为60秒后，如果一个连接被借出超过60秒还没归还，HikariCP 会打印一条告警日志并附带借出时的堆栈：

WARNING: Connection leak detection triggered for com.mysql.cj.jdbc.ConnectionImpl@1a2b3c4d, stack trace follows
java.lang.Exception: Apparent connection leak detected
    at com.zaxxer.hikari.HikariDataSource.getConnection(HikariDataSource.java:128)
    at com.example.service.OrderService.queryOrder(OrderService.java:45)
    ...

看到这个堆栈就知道是哪个方法借了连接没还。

3. 连接泄漏的排查 ​

常见泄漏原因：

原因一：手动获取连接没关闭

java
// 错误写法
public void badMethod() {
    Connection conn = dataSource.getConnection();
    PreparedStatement ps = conn.prepareStatement("SELECT ...");
    ResultSet rs = ps.executeQuery();
    // 如果中间抛异常，conn永远不会归还到连接池
    processResult(rs);
    conn.close();
}

// 正确写法：try-with-resources
public void goodMethod() {
    try (Connection conn = dataSource.getConnection();
         PreparedStatement ps = conn.prepareStatement("SELECT ...");
         ResultSet rs = ps.executeQuery()) {
        processResult(rs);
    }
}

原因二：事务未正确关闭

java
// 错误写法：编程式事务异常时未回滚
public void badTransaction() {
    TransactionStatus status = txManager.getTransaction(new DefaultTransactionDefinition());
    try {
        doSomething();
        // 如果这里抛异常，事务没回滚，连接也没释放
        txManager.commit(status);
    } catch (Exception e) {
        // 忘记回滚
        log.error("操作失败", e);
    }
}

// 正确写法
public void goodTransaction() {
    TransactionStatus status = txManager.getTransaction(new DefaultTransactionDefinition());
    try {
        doSomething();
        txManager.commit(status);
    } catch (Exception e) {
        txManager.rollback(status);
        throw e;
    }
}

原因三：MyBatis 嵌套查询的连接持有

java
// 危险：在遍历结果时发起新的数据库查询（N+1问题也会导致连接长时间占用）
@Select("SELECT * FROM t_order WHERE user_id = #{userId}")
List<Order> selectByUserId(Long userId);

// OrderMapper 中
@Select("SELECT * FROM t_order_detail WHERE order_id = #{orderId}")
List<OrderDetail> selectDetails(Long orderId);

如果在 Service 层这样用：

java
List<Order> orders = orderMapper.selectByUserId(userId);
for (Order order : orders) {
    // 每次循环都获取新连接执行查询，外层连接还在持有
    order.setDetails(orderMapper.selectDetails(order.getOrderId()));
}

外层查询的连接一直不释放，循环越多连接占用越久。优化方案：用 @One 注解做 JOIN 查询，或者用 IN 查询一次性取详情。

IDEA高效开发技巧与调试实战

问题场景 ​

用了几年IDEA，发现很多同事还在用最基础的功能——打断点、F8单步、看变量值。IDEA的调试和效率功能远不止这些。这篇文章把我日常用得最多的技巧整理出来，都是在真实开发场景中验证过的。

远程调试：在线上环境Debug ​

线上出了问题，日志看不出原因时，远程调试是最高效的排查手段（但只能在测试环境用，生产环境慎用）。

Step 1：启动应用时开启远程调试

bash
# Java 9+ 使用的方式
java -agentlib:jdwp=transport=dt_socket,server=y,suspend=n,address=5005 -jar app.jar

# Java 8 的方式（旧写法）
java -Xdebug -Xrunjdwp:transport=dt_socket,server=y,suspend=n,address=5005 -jar app.jar

关键参数：

  • suspend=n：不暂停启动，等调试器连接后再继续。如果设为 y，应用会卡在启动等你连
  • address=5005：监听端口，确保防火墙允许

Docker容器中开启远程调试：

dockerfile
ENV JAVA_TOOL_OPTIONS="-agentlib:jdwp=transport=dt_socket,server=y,suspend=n,address=*:5005"
EXPOSE 5005

Step 2：IDEA配置远程连接

Run -> Edit Configurations -> + -> Remote JVM Debug：

  • Host: 测试服务器IP
  • Port: 5005
  • Use module classpath: 选择你的项目模块

连接成功后，跟本地调试一样打断点。

踩坑点：远程调试时本地代码要和远程部署的代码版本一致，否则断点位置会错乱。打完断点记得及时移除，否则会影响线上性能。

条件断点与高级断点 ​

条件断点

不是所有断点都要在每次命中时暂停。右键断点，输入条件表达式：

java
// 只在订单金额大于1万时暂停
order.getAmount() > 10000

// 只在特定用户ID时暂停
"user_12345".equals(userId)

// 只在第N次命中时暂停（Hit Count）
// 在断点属性中设置 Hit Count = 100，第100次命中时暂停

日志断点（不暂停的断点）

有时候你不想暂停程序执行，只想看某个变量的值。右键断点：

  1. 取消 "Suspend" 勾选
  2. 勾选 "Evaluate and log"
  3. 输入表达式："当前用户: " + user.getId() + ", 订单数: " + orders.size()

这样值会输出到IDEA的Console窗口，程序不会暂停。

方法断点

在接口方法名左边的槽位点击，会出现一个菱形断点。当实现类被调用时会暂停。适合调试Spring的AOP和代理对象。

字段断点（Field Watchpoint）

在类的字段声明处打断点，当这个字段被读取或修改时暂停。适合排查"这个值什么时候被改了"的问题。

热部署 ​

Spring Boot DevTools

最简单的热部署方案，修改代码后自动重启：

xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-devtools</artifactId>
    <optional>true</optional>
</dependency>

  1. Settings -> Build -> Compiler -> Build project automatically 勾选
  2. Settings -> Advanced Settings -> Allow auto-make to start even if developed application is currently running 勾选

JRebel（付费方案）

真正的热加载，不需要重启。修改方法体后实时生效，速度比DevTools快得多。团队有预算的话推荐购买。
