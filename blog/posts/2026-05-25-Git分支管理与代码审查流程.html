问题场景 ​

我司后端团队从最初的"所有人往master上推"，到后来引入Git Flow搞得分支满天飞，再到最终落地适合我们规模的分支策略，前后经历了一年多的混乱期。这篇文章记录了我们从混乱到有序的过程。

Git Flow vs Trunk Based：选哪个？ ​

Git Flow 的问题

我们最初按网上教程搞Git Flow：master、develop、feature/、release/、hotfix/*。80人的团队，20个开发同时在做需求，分支数量爆炸：

feature/user-order-redesign
feature/payment-v3
feature/coupon-engine
release/v2.3.0
hotfix/20240315-null-pointer
... 还有几十个

问题在于：

  • 分支生命周期长（平均2-3周），合并时冲突像炸弹
  • develop分支经常处于"半成品"状态，测试环境不稳定
  • release分支需要额外的cherry-pick操作，容易遗漏

Trunk Based 的落地

我们最终采用了简化版的Trunk Based：

main          ← 生产分支，始终可部署
  ├── feat/xxx  ← 短生命周期feature分支（1-3天）
  └── release/v2.3  ← 仅在发版时创建，发布后合并回main

核心规则：

  1. feature分支存活不超过3天，超过就拆分需求
  2. 所有改动必须通过PR合入main，禁止直接push
  3. main分支始终可部署，通过CI保证
  4. 使用feature flag控制未完成功能的可见性

PR/MR 规范：让代码审查真正有效 ​

我们用GitLab，所以叫MR，但思路通用。

MR 模板

在项目根目录创建 .gitlab/merge_request_templates/Default.md：

markdown
## 改动说明
<!-- 用1-3句话说明这个MR做了什么，为什么要做 -->

## 改动类型
- [ ] 新功能
- [ ] Bug修复
- [ ] 重构
- [ ] 配置变更
- [ ] 文档更新

## 影响范围
- [ ] 涉及数据库变更
- [ ] 涉及接口变更
- [ ] 涉及配置变更
- [ ] 涉及第三方服务调用

## 自测情况
<!-- 自测了哪些场景，附截图或日志 -->

## 关联Issue
Closes #xxx

审查要点清单（贴在团队wiki，不是形式主义而是真用的）：

  • SQL有没有走索引？WHERE条件字段有没有索引？
  • 异常处理是不是catch了Exception然后吞掉了？
  • 敏感信息有没有打印到日志（手机号、身份证、密码）？
  • 线程池/连接池参数是不是写死的？
  • 有没有N+1查询问题？

Commit Message 规范 ​

我们没有强制conventional commits，而是用一种简单实用的格式：

<类型>: <简要描述>

<可选的详细说明>

类型只有几种：

  • feat: 新功能
  • fix: 修复
  • refactor: 重构（不改功能）
  • config: 配置变更
  • sql: 数据库变更
  • docs: 文档

通过git hook强制执行：

bash
#!/bin/bash
# .git/hooks/commit-msg

commit_msg=$(cat "$1")
pattern="^(feat|fix|refactor|config|sql|docs): .{2,}"

if ! echo "$commit_msg" | grep -qE "$pattern"; then
    echo "Commit message不符合规范！"
    echo "格式: <类型>: <描述>"
    echo "类型: feat|fix|refactor|config|sql|docs"
    echo "示例: fix: 修复订单查询时的NPE问题"
    exit 1
fi

冲突解决策略 ​

长生命周期分支合并时的冲突，是我们最头疼的问题。几种常见场景和应对：

场景1：配置文件冲突

application.yml 频繁被不同分支修改，合并必冲突。

解决方案：拆分配置文件

yaml
# application.yml - 只放公共配置
spring:
  profiles:
    active: ${SPRING_ACTIVE_PROFILE:dev}

# application-db.yml - 数据库相关
# application-redis.yml - Redis相关
# application-mq.yml - 消息队列相关

场景2：MyBatis XML冲突

两个人同时改了同一个Mapper XML，合并时XML结构错乱，CI直接报错。

解决方案：

  • 一个模块的Mapper XML不要超过500行，该拆就拆
  • 不同功能的SQL放到不同的XML文件中

场景3：冲突解决流程

bash
# 1. 先把main最新代码rebase到自己的分支
git fetch origin
git rebase origin/main

# 2. 解决冲突（如果有）
# 编辑冲突文件，然后：
git add .
git rebase --continue

# 3. 强制推送自己的分支（因为rebase改写了历史）
git push --force-with-lease origin feat/your-branch

注意：--force-with-lease 比 --force 安全，它会检查远程分支有没有被别人更新过，避免覆盖别人的提交。

保护分支配置 ​

在GitLab中对main分支设置保护：

yaml
# GitLab 设置 → Repository → Protected Branches
Branch: main
Allowed to merge: Maintainers
Allowed to push: No one    # 禁止直接push，必须走MR
Allowed to force push: No  # 禁止强推

踩坑总结 ​

  1. 不要一上来就搞复杂的分支策略。50人以下的团队，Trunk Based + 短生命周期分支就够了，Git Flow适合版本发布节奏慢的项目
  2. 代码审查的目的是发现问题，不是挑刺。审查者应该先理解需求背景再看代码，不要上来就纠结命名风格
  3. feature分支超过3天没合并，大概率会出问题。要么需求太大需要拆分，要么分支管理出了问题
  4. CI必须强依赖。没有CI保证main分支始终可部署，Trunk Based就是空中楼阁
  5. 合并冲突预防大于解决。拆分文件、减少热点文件、频繁同步main，比学会复杂冲突解决技巧更有用
  6. 别忘了.gitignore。IDE配置文件、本地构建产物提交上去只会制造无意义的冲突

分支策略没有银弹，关键是找到适合自己团队规模和发布节奏的方式，然后严格执行。规则不需要多，但定下来的就要守住。
