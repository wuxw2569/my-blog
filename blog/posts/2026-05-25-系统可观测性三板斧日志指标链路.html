可观测性不是大厂的专利 ​

在中小厂，很多团队的"监控"就是出了问题之后收到用户投诉，然后登录服务器翻日志。我也经历过这个阶段，直到有一次线上故障排查了 6 个小时才发现是数据库连接池泄漏——如果有可观测性基础设施，10 分钟就能定位。

可观测性三根支柱：日志（Logging）、指标（Metrics）、链路（Tracing）。下面分别讲我们在生产环境的实际做法。

一、日志体系：ELK Stack ​

架构 ​

应用日志 -> Filebeat -> Kafka -> Logstash -> Elasticsearch -> Kibana

为什么加一层 Kafka？因为我们日均日志量 50GB，Logstash 的消费速度跟不上 Filebeat 的推送速度。Kafka 做缓冲，削峰填谷。

应用日志格式标准化 ​

xml
<!-- logback-spring.xml -->
<configuration>
    <springProperty scope="context" name="APP_NAME"
                    source="spring.application.name"/>

    <appender name="KAFKA" class="com.github.danielwegener.logback.kafka.KafkaAppender">
        <topic>app-logs</topic>
        <keyingStrategy
            class="com.github.danielwegener.logback.kafka.keying.RoundRobinKeyingStrategy"/>
        <deliveryStrategy
            class="com.github.danielwegener.logback.kafka.delivery.AsynchronousDeliveryStrategy"/>
        <producerConfig>
            bootstrap.servers=${KAFKA_ADDR:kafka:9092}
        </producerConfig>
        <encoder class="net.logstash.logback.encoder.LogstashEncoder">
            <!-- 关键：把 TraceId 写入日志 JSON，用于关联链路 -->
            <customFields>{"app":"${APP_NAME}","env":"${ENV:dev}"}</customFields>
        </encoder>
    </appender>

    <!-- 控制台输出（开发环境用） -->
    <appender name="CONSOLE" class="ch.qos.logback.core.ConsoleAppender">
        <encoder>
            <pattern>%d{HH:mm:ss.SSS} [%tid] [%thread] %-5level %logger{36} - %msg%n</pattern>
        </encoder>
    </appender>

    <root level="INFO">
        <springProfile name="dev,test">
            <appender-ref ref="CONSOLE"/>
        </springProfile>
        <springProfile name="staging,prod">
            <appender-ref ref="KAFKA"/>
        </springProfile>
    </root>
</configuration>

Logstash 管道配置 ​

ruby
# logstash.conf
input {
  kafka {
    bootstrap_servers => "kafka:9092"
    topics => ["app-logs"]
    group_id => "logstash-consumer"
    codec => json
  }
}

filter {
  # 从日志中提取 TraceId，方便关联链路追踪
  if [tid] {
    mutate {
      add_field => { "trace_id" => "%{tid}" }
    }
  }

  # 从日志消息中提取接口路径和耗时
  if [message] =~ /【接口入参】/ {
    grok {
      match => {
        "message" => "【接口入参】%{DATA:module} \| %{DATA:method} \| args=%{GREEDYDATA:request_args}"
      }
    }
  }
}

output {
  elasticsearch {
    hosts => ["elasticsearch:9200"]
    index => "app-logs-%{+YYYY.MM.dd}"
  }
}

日志等级策略 ​

java
// 日志级别使用原则
log.debug("【调试】查询参数: {}", params);        // 开发环境用
log.info("【业务】订单创建成功, orderId={}", id);   // 正常业务流程
log.warn("【告警】库存不足, productId={}, remain={}", pid, count); // 需要关注但不紧急
log.error("【异常】支付回调处理失败, payNo={}", payNo, e);  // 需要立即处理

踩坑：生产环境不要开 DEBUG 级别。我们有一次为了排查问题临时开了 DEBUG，结果日志量暴涨 10 倍，Kafka 积压，ELK 整个链路都挂了。正确的做法是通过 Nacos 动态调整某个 Logger 的级别：

java
// 动态调整指定包的日志级别，不影响全局
@NacosConfigListener(dataId = "log-level-config", groupId = "COMMON_GROUP")
public void onLogLevelChange(String config) {
    JSONObject levels = JSON.parseObject(config);
    levels.forEach((loggerName, level) -> {
        Logger logger = (Logger) LoggerFactory.getLogger(loggerName);
        logger.setLevel(Level.valueOf(level));
        log.info("【日志级别调整】{} -> {}", loggerName, level);
    });
}

二、指标监控：Prometheus + Grafana ​

应用暴露指标端点 ​

yaml
management:
  endpoints:
    web:
      exposure:
        include: health,prometheus
  metrics:
    tags:
      application: ${spring.application.name}
    distribution:
      percentiles-histogram:
        http.server.requests: true
      percentiles:
        http.server.requests: 0.5,0.9,0.95,0.99

访问 http://localhost:8080/actuator/prometheus 就能看到所有指标。

自定义业务指标 ​

java
@Component
@RequiredArgsConstructor
public class BusinessMetrics {

    private final MeterRegistry meterRegistry;
    private Counter orderCreatedCounter;
    private Counter orderFailedCounter;
    private Timer orderProcessTimer;

    @PostConstruct
    public void init() {
        // 订单创建计数器
        orderCreatedCounter = Counter.builder("business.order.created")
            .description("订单创建数量")
            .tag("type", "counter")
            .register(meterRegistry);

        // 订单处理耗时
        orderProcessTimer = Timer.builder("business.order.process.time")
            .description("订单处理耗时")
            .publishPercentiles(0.5, 0.95, 0.99)
            .register(meterRegistry);
    }

    public void recordOrderCreated() {
        orderCreatedCounter.increment();
    }

    public void recordOrderProcessTime(long millis) {
        orderProcessTimer.record(millis, TimeUnit.MILLISECONDS);
    }
}

Grafana 核心看板 ​

我们维护的几个关键看板：

JVM 看板：堆内存使用、GC 频率和耗时、线程数、CPU 使用率

接口看板：QPS、P50/P95/P99 延迟、错误率、状态码分布

业务看板：订单量、支付成功率、异常订单数

Grafana 告警规则示例：

yaml
# 当接口 P99 延迟超过 2 秒时告警
- alert: HighLatencyP99
  expr: histogram_quantile(0.99,
    sum(rate(http_server_requests_seconds_bucket[5m])) by (le)) > 2
  for: 5m
  labels:
    severity: warning
  annotations:
    summary: "接口 P99 延迟超过 2 秒"
    description: "当前 P99 延迟: {{ $value }}s"

三、三者的统一关联 ​

日志、指标、链路追踪各自有价值，但组合起来才是真正的可观测性。

TraceId 串联日志和链路 ​

这是最核心的关联手段。一次请求从网关到后端服务，所有日志共享同一个 TraceId，在 Kibana 里搜索 TraceId 就能看到完整的请求链路。

指标异常 -> 日志定位 -> 链路分析 ​

排查问题的典型流程：

  1. Grafana 监控告警：订单服务 P99 延迟从 200ms 升到 2s
  2. 打开 Grafana 看板，发现是 /api/order/create 接口变慢
  3. 到 Kibana 搜索这个接口的慢日志：method:"/api/order/create" AND cost:>1000
  4. 找到一条耗时 3s 的请求，拿到 TraceId
  5. 用 TraceId 到 SkyWalking 查看链路详情，发现是库存服务的 SQL 查询耗时 2.5s
  6. 定位到具体的慢 SQL，优化索引解决

统一日志规范 ​

所有服务的日志遵循统一格式：

【模块/动作】关键参数

示例：

【订单创建开始】userId=123, productCount=3
【库存校验通过】productId=456, remain=100
【支付请求发出】payNo=PAY20260525001, amount=299.00
【支付回调处理】payNo=PAY20260525001, status=SUCCESS
【订单创建完成】orderId=789, cost=523ms

这样在 Kibana 里搜索 【订单创建完成】 就能看到所有订单创建的结果和耗时。

踩坑总结 ​

  1. 不要把所有日志都送 ES。ES 存储成本高，DEBUG 和 TRACE 级别的日志只在开发环境输出，生产环境只保留 INFO 及以上。我们有一段时间生产环境开着 DEBUG，ES 磁盘一周就满了。

  2. Prometheus 的数据保留策略。默认保留 15 天就够了。高基数标签（如 userId）会撑爆 Prometheus 内存，千万注意不要在标签里放用户 ID、订单号这种高基数字段。

  3. 链路追踪的采样率。全量采集在日均千万请求的系统里不可行，存储成本太高。建议：错误请求和慢请求 100% 采样，正常请求 5-10% 采样。

  4. 告警不要太多。一开始我们配了几十条告警规则，每天收到上百条告警，很快就麻木了。后来精简到 5 条关键告警（服务不可用、错误率>5%、P99>2s、CPU>80%、磁盘>85%），每条都是真问题。

  5. 成本控制。中小厂的基础设施预算有限，ELK 用热温冷架构：最近 3 天的数据放热节点（SSD），3-7 天放温节点（HDD），7 天以上的自动删除。Prometheus 的数据保留 15 天，长期趋势用 Grafana 的录制规则生成低精度数据保留 90 天。

可观测性建设不是一蹴而就的，建议优先级：日志标准化 > 接口监控 > 链路追踪 > 业务指标。先解决"出问题能不能快速定位"，再解决"能不能提前发现问题"。
