问题场景 ​

我们的客服系统上线后，业务方提了新需求：简单问答用便宜的模型（qwen-turbo），复杂分析用高质量模型（qwen-max），还有一些特殊场景比如代码生成需要用专门的代码模型。更麻烦的是，有时候 DashScope 的 API 会抖动，需要能自动降级到备选方案。

这就不是单模型能搞定的了，需要一套多模型接入 + 动态切换的方案。

分析思路 ​

多模型管理核心要解决三个问题：

  1. 模型注册：怎么同时配置多个模型实例
  2. 路由策略：什么场景用什么模型，这个决策逻辑放哪里
  3. 降级方案：主模型调不通时怎么切换

Spring AI 的 ChatModel 接口天然支持多实例，我们可以注册多个 Bean，然后在 Service 层做路由。路由策略可以用策略模式实现，降级用 Resilience4j 的 CircuitBreaker。

具体实现 ​

多模型注册 ​

java
@Configuration
public class MultiModelConfig {

    /**
     * 通义千问 - 轻量级，适合简单问答
     */
    @Bean("turboModel")
    public ChatModel turboModel() {
        DashScopeApi api = DashScopeApi.builder()
                .apiKey(System.getenv("DASHSCOPE_API_KEY"))
                .build();
        return new DashScopeChatModel(api,
                DashScopeChatOptions.builder()
                        .withModel("qwen-turbo")
                        .withTemperature(0.3)
                        .withMaxTokens(1024)
                        .build());
    }

    /**
     * 通义千问 - 标准级，适合一般业务对话
     */
    @Bean("plusModel")
    public ChatModel plusModel() {
        DashScopeApi api = DashScopeApi.builder()
                .apiKey(System.getenv("DASHSCOPE_API_KEY"))
                .build();
        return new DashScopeChatModel(api,
                DashScopeChatOptions.builder()
                        .withModel("qwen-plus")
                        .withTemperature(0.5)
                        .withMaxTokens(2048)
                        .build());
    }

    /**
     * 通义千问 - 旗舰级，适合复杂分析任务
     */
    @Bean("maxModel")
    public ChatModel maxModel() {
        DashScopeApi api = DashScopeApi.builder()
                .apiKey(System.getenv("DASHSCOPE_API_KEY"))
                .build();
        return new DashScopeChatModel(api,
                DashScopeChatOptions.builder()
                        .withModel("qwen-max")
                        .withTemperature(0.2)
                        .withMaxTokens(4096)
                        .build());
    }
}

踩坑点：每个 DashScopeApi 实例实际上可以复用，不需要每个模型都 new 一个。但如果后续需要给不同模型配置不同的 timeout 或 proxy，分开创建更灵活。

模型路由器 ​

java
public enum ModelLevel {
    TURBO,  // 轻量
    PLUS,   // 标准
    MAX     // 旗舰
}

@Service
@Slf4j
public class ModelRouterService {

    private final Map<ModelLevel, ChatModel> modelMap;

    public ModelRouterService(
            @Qualifier("turboModel") ChatModel turboModel,
            @Qualifier("plusModel") ChatModel plusModel,
            @Qualifier("maxModel") ChatModel maxModel) {
        this.modelMap = Map.of(
                ModelLevel.TURBO, turboModel,
                ModelLevel.PLUS, plusModel,
                ModelLevel.MAX, maxModel
        );
    }

    /**
     * 根据场景路由到合适的模型
     */
    public ChatModel route(ModelLevel level) {
        ChatModel model = modelMap.get(level);
        if (model == null) {
            log.warn("【模型路由】未找到指定级别 {}，使用默认 plus", level);
            return modelMap.get(ModelLevel.PLUS);
        }
        return model;
    }

    /**
     * 根据问题复杂度自动选择模型
     * 简单规则：输入长度 + 关键词判断
     */
    public ChatModel autoRoute(String userMessage) {
        // 简单问答：长度短、无专业关键词
        if (userMessage.length() < 50 && !containsComplexKeywords(userMessage)) {
            log.info("【自动路由】简单问题，使用 turbo");
            return modelMap.get(ModelLevel.TURBO);
        }
        // 复杂分析：长度长或含专业关键词
        if (userMessage.length() > 500 || containsAnalysisKeywords(userMessage)) {
            log.info("【自动路由】复杂分析，使用 max");
            return modelMap.get(ModelLevel.MAX);
        }
        log.info("【自动路由】一般问题，使用 plus");
        return modelMap.get(ModelLevel.PLUS);
    }

    private boolean containsComplexKeywords(String text) {
        return text.contains("分析") || text.contains("对比") || text.contains("总结");
    }

    private boolean containsAnalysisKeywords(String text) {
        return text.contains("报告") || text.contains("方案") || text.contains("策略");
    }
}

自动降级 ​

java
@Service
@Slf4j
public class ResilientChatService {

    private final ModelRouterService routerService;
    private final CircuitBreakerRegistry circuitBreakerRegistry;

    public ResilientChatService(ModelRouterService routerService) {
        this.routerService = routerService;
        this.circuitBreakerRegistry = CircuitBreakerRegistry.of(
                CircuitBreakerConfig.custom()
                        .failureRateThreshold(50)           // 失败率超50%触发熔断
                        .waitDurationInOpenState(Duration.ofSeconds(30))  // 熔断30秒
                        .slidingWindowSize(10)              // 最近10次调用统计
                        .build());
    }

    /**
     * 带降级的模型调用
     * 优先用指定模型，熔断后自动降级
     */
    public String chatWithFallback(ModelLevel level, String userMessage) {
        // 先尝试主模型
        try {
            return doChat(level, userMessage);
        } catch (Exception e) {
            log.warn("【降级触发】{} 调用失败: {}，尝试降级", level, e.getMessage());
        }

        // 降级链：MAX -> PLUS -> TURBO
        ModelLevel[] fallbackChain = getFallbackChain(level);
        for (ModelLevel fallback : fallbackChain) {
            try {
                log.info("【降级执行】切换到 {}", fallback);
                return doChat(fallback, userMessage);
            } catch (Exception ex) {
                log.warn("【降级失败】{} 也失败了: {}", fallback, ex.getMessage());
            }
        }

        // 所有模型都失败
        log.error("【降级结束】所有模型均不可用");
        return "抱歉，AI服务暂时不可用，请稍后再试。";
    }

    private String doChat(ModelLevel level, String message) {
        CircuitBreaker cb = circuitBreakerRegistry.circuitBreaker(level.name());
        return cb.executeSupplier(() -> {
            ChatModel model = routerService.route(level);
            return model.call(new Prompt(message))
                    .getResult()
                    .getOutput()
                    .getContent();
        });
    }

    private ModelLevel[] getFallbackChain(ModelLevel level) {
        return switch (level) {
            case MAX -> new ModelLevel[]{ModelLevel.PLUS, ModelLevel.TURBO};
            case PLUS -> new ModelLevel[]{ModelLevel.TURBO};
            case TURBO -> new ModelLevel[]{};
        };
    }
}

对外接口 ​

java
@RestController
@RequestMapping("/api/chat")
@RequiredArgsConstructor
@Slf4j
public class SmartChatController {

    private final ModelRouterService routerService;
    private final ResilientChatService resilientChatService;

    /**
     * 指定模型级别的对话接口
     */
    @PostMapping("/smart")
    public Map<String, Object> smartChat(@RequestBody SmartChatRequest request) {
        ModelLevel level = ModelLevel.valueOf(request.level().toUpperCase());
        String answer = resilientChatService.chatWithFallback(level, request.message());
        return Map.of("answer", answer, "requestedLevel", request.level());
    }

    /**
     * 自动路由的对话接口
     */
    @PostMapping("/auto")
    public Map<String, Object> autoChat(@RequestBody ChatRequest request) {
        ChatModel model = routerService.autoRoute(request.message());
        String answer = model.call(new Prompt(request.message()))
                .getResult().getOutput().getContent();
        return Map.of("answer", answer);
    }

    public record SmartChatRequest(String message, String level) {}
    public record ChatRequest(String message) {}
}

踩坑总结 ​

  1. 不要过度设计路由策略：我们一开始想用 LLM 自动判断问题复杂度来选模型，但那本身就多了一次模型调用，延迟和成本都上去了。最后用简单的规则引擎（输入长度 + 关键词）反而效果最好。

  2. CircuitBreaker 的 slidingWindowSize 要调：默认值在模型调用这种高延迟场景下不太合适。建议至少设 10 次以上，否则一次偶发超时就可能触发熔断。

  3. 模型级别的降级不代表回答质量线性下降：qwen-turbo 在某些简单问题上可能跟 qwen-max 差别不大，但在需要多步推理的场景差距很明显。降级后最好在回答里注明"当前使用的是轻量模型"。

  4. DashScope 有并发限制：免费版的 QPS 限制比较低，多个模型共享同一个 API Key 的话，总 QPS 是叠加的。如果流量大，要么申请提升配额，要么用多个 Key 做轮询。

  5. 配置热更新有局限：DashScopeChatModel 的一些参数（比如 model name）在实例创建后就不能改了。要真正实现热切换，得重建 Bean 然后刷新 Spring 容器，成本比较高。我们最后采用的方案是：预创建所有模型实例，运行时只做路由切换。
