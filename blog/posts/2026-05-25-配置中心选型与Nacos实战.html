没有配置中心的痛 ​

2024 年初我们还是用 application.yml 管理配置，加上 Spring profiles 做环境隔离。问题一个接一个：

  • 修改一个 Redis 连接地址，要重新打包部署，等 5 分钟的 CI/CD
  • 生产环境某个阈值配错了，紧急回滚要 10 分钟
  • 不同环境的配置文件越来越多，YAML 之间互相覆盖，调试一下午才发现是 profile 优先级问题
  • 配置变更没有审计记录，谁改了什么无从查起

这些痛点推动我们引入配置中心。

Apollo vs Nacos 选型 ​

Apollo（携程开源）和 Nacos（阿里开源）是最主流的两个选择。我们最终选了 Nacos，以下是详细对比：

维度ApolloNacos
核心定位专注配置管理配置管理 + 服务注册二合一
配置界面功能完善，有灰度发布、变更审计、回滚基础功能够用，灰度和审计相对弱
多环境管理通过 Env + Cluster + Namespace 三级结构通过 Namespace + Group + Data ID 三级结构
配置推送长轮询，实时性好长轮询，默认延迟 30ms 内
部署复杂度4 个组件（Admin Service、Config Service、Portal、DB）单体部署（standalone 或 cluster）
生态Spring Boot Starter 支持好和 Spring Cloud Alibaba 深度集成
社区携程维护，更新频率一般阿里维护，更新活跃

选 Nacos 的理由：

  1. 我们已经在用 Nacos 做服务注册发现，不需要额外部署一套配置中心
  2. 团队技术栈是 Spring Cloud Alibaba，Nacos 是默认选择
  3. Apollo 的灰度发布和审计确实更好，但对中小厂来说不是刚需

Nacos 多环境管理 ​

架构设计 ​

Namespace（环境隔离）
  ├── dev     - 开发环境
  ├── staging - 预发环境
  └── prod    - 生产环境
       ├── Group: ORDER_GROUP（订单服务的配置）
       │   ├── Data ID: order-service.yml
       │   └── Data ID: order-service-datasource.yml
       ├── Group: PRODUCT_GROUP
       │   └── Data ID: product-service.yml
       └── Group: DEFAULT_GROUP（公共配置）
            └── Data ID: common.yml

Spring Boot 接入 ​

yaml
# bootstrap.yml
spring:
  application:
    name: order-service
  profiles:
    active: ${SPRING_PROFILES_ACTIVE:dev}
  cloud:
    nacos:
      config:
        server-addr: ${NACOS_ADDR:127.0.0.1:8848}
        # 环境隔离 - 不同环境用不同 namespace
        namespace: ${NACOS_NAMESPACE:dev}
        group: ORDER_GROUP
        file-extension: yml
        # 共享配置
        shared-configs:
          - data-id: common.yml
            group: DEFAULT_GROUP
            refresh: true
        # 坑：username 和 password 必须配，否则 nacos 2.x 会鉴权失败
        username: nacos
        password: ${NACOS_PASSWORD:nacos}

踩坑：shared-configs 的优先级低于主配置，如果 common.yml 和 order-service.yml 有同名 key，后者会覆盖前者。这个逻辑不直觉，踩过好几次。

配置热更新原理 ​

Nacos 配置热更新的底层机制：

  1. 客户端启动时拉取配置，建立长轮询连接
  2. 配置变更时服务端推送变更通知
  3. 客户端收到通知后拉取最新配置
  4. 通过 Spring 的 @RefreshScope 或 EnvironmentChangeEvent 触发 Bean 刷新
java
/**
 * 方式一：@RefreshScope + @Value
 * 坑：@RefreshScope 会导致 Bean 每次请求都重新创建，如果有状态要注意
 */
@RefreshScope
@RestController
@RequestMapping("/api/config")
public class ConfigTestController {

    @Value("${order.timeout:5000}")
    private int orderTimeout;

    @GetMapping("/timeout")
    public Result<Integer> getTimeout() {
        return Result.ok(orderTimeout);
    }
}

/**
 * 方式二：@ConfigurationProperties（推荐）
 * 自动刷新，不需要 @RefreshScope
 */
@Data
@Component
@ConfigurationProperties(prefix = "order")
public class OrderProperties {
    /** 下单超时时间（毫秒） */
    private int timeout = 5000;
    /** 最大重试次数 */
    private int maxRetry = 3;
    /** 是否开启库存预占 */
    private boolean inventoryPreempt = true;
}

// 使用
@Service
@RequiredArgsConstructor
public class OrderService {
    private final OrderProperties orderProperties;

    public void createOrder(OrderDTO dto) {
        // 读取的是最新配置值，修改 Nacos 后无需重启
        int timeout = orderProperties.getTimeout();
        // ...
    }
}

踩坑：@RefreshScope 和 @Async 一起用会出问题。@RefreshScope 的 Bean 默认是 SCOPE_PROTOTYPE，但 @Async 的代理目标在 Bean 创建时就固定了。刷新后 @Async 的方法可能调用到旧实例。解决方案是统一用 @ConfigurationProperties。

灰度配置 ​

Nacos 原生的灰度能力比较弱（相比 Apollo），但我们可以通过元数据实现简单的灰度：

按实例灰度 ​

java
/**
 * 监听配置变更事件，根据实例元数据决定是否采用新配置
 */
@Component
@Slf4j
public class GrayConfigListener {

    @NacosConfigListener(dataId = "order-service.yml",
        groupId = "ORDER_GROUP", timeout = 3000)
    public void onConfigChange(String newConfig) {
        // 读取实例元数据中的灰度标识
        String grayTag = System.getProperty("gray.tag", "stable");
        NacosConfigProperties properties = parseYaml(newConfig);

        if ("canary".equals(grayTag)) {
            // 灰度实例使用特殊配置
            log.info("【灰度配置】应用灰度配置: {}", properties);
        } else {
            log.info("【稳定配置】使用稳定配置");
        }
    }
}

更实用的方案：Nacos + 本地配置回退 ​

java
/**
 * 配置优先级：Nacos > 环境变量 > 本地 yml > 默认值
 * 如果 Nacos 不可用，自动回退到本地配置
 */
@Configuration
public class ConfigFallbackConfig {

    @Bean
    public NacosConfigManager nacosConfigManager(
            NacosConfigProperties properties) {
        try {
            return new NacosConfigManager(properties);
        } catch (NacosException e) {
            log.warn("【配置中心】Nacos 不可用，使用本地配置", e);
            return null;
        }
    }
}

生产环境经验 ​

  1. 配置变更必须有审计：Nacos 自带的操作日志可以记录谁在什么时候改了什么配置，但功能比较基础。我们的做法是把配置变更事件推送到企业微信，关键配置变更需要审批。

  2. 敏感配置加密：数据库密码、API Key 等敏感信息不要明文放在 Nacos 里。我们用 Jasypt 加密：

yaml
# application.yml - 加密后的密码
spring:
  datasource:
    password: ENC(加密后的密文)
java
// 启动时传入解密密钥
// java -Djasypt.encryptor.password=your-secret-key -jar app.jar

  1. 配置回滚：Nacos 的配置历史功能可以回滚到之前的版本。但更安全的做法是把配置文件也纳入 Git 管理，Nacos 作为运行时配置源，Git 作为配置的"真相源"。

  2. 配置加载顺序：bootstrap.yml > Nacos 远程配置 > application.yml。Spring Boot 3.x 默认不加载 bootstrap.yml，需要额外引入 spring-cloud-starter-bootstrap 依赖。

  3. 集群部署：生产环境至少 3 个 Nacos 节点 + MySQL 存储。我们用的是 Nacos 2.3 + MySQL 8.0，3 个节点跑了半年多很稳定。注意 Nacos 2.x 的 gRPC 端口（默认 9848/9849）要在防火墙放行。
