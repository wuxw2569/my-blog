AI 应用开发的核心概念与术语速查手册 ​

前言 ​

作为开发者，当你开始接触 AI 应用开发时，会遇到大量新概念和术语。很多文章要么不解释这些术语（默认你懂），要么解释得太学术（看了等于没看）。

这篇文章就是一份速查手册，用最通俗的方式把 AI 应用开发中最核心的概念一次性讲清楚。你可以把它收藏起来，遇到不懂的术语随时翻看。

基础概念篇 ​

Token（词元） ​

是什么： LLM 处理文本的最小单位。不是字符，不是单词，而是介于两者之间的"子词"。

为什么要了解它： 因为 LLM 的计费、上下文限制、响应速度都以 Token 为单位。

经验值：

  • 1 个英文单词 ≈ 1-2 个 token
  • 1 个中文字 ≈ 1-2 个 token
  • 1000 个 token ≈ 750 个英文单词 ≈ 500 个中文字

工具： 可以用 tiktoken（OpenAI）或各厂商提供的 tokenizer 来计算 token 数。

Context Window / Context Length（上下文窗口） ​

是什么： 模型一次能处理的最大 token 数。包括输入（prompt + 历史对话）和输出（生成内容）的总和。

类比： 相当于模型的"工作记忆"容量。窗口越大，能同时"看到"的信息越多。

当前主流值：

  • GPT-4o: 128K tokens
  • Claude 4 Sonnet: 200K tokens
  • Gemini 2.5 Pro: 1M-2M tokens
  • DeepSeek-V3: 128K tokens

注意： 上下文窗口大不等于模型能完美利用所有内容。研究表明，模型对上下文开头和结尾的内容关注度高于中间部分。

Prompt（提示词） ​

是什么： 你发给 LLM 的输入文本。

分类：

  • User Prompt：用户的直接输入
  • System Prompt：系统级的指令，设定模型的角色和行为规则
  • Assistant Prompt：模型之前的回复（用于多轮对话）

类比： Prompt 就是你给 AI 下的"工作指令"。指令越清晰，结果越好。

System Prompt（系统提示词） ​

是什么： 在对话开始前设定模型角色、规则和限制的指令。用户通常看不到 System Prompt。

python
messages = [
    {"role": "system", "content": "你是一个专业的法律顾问。只回答与劳动法相关的问题。如果问题不相关，礼貌地拒绝回答。回答时引用具体法条。"},
    {"role": "user", "content": "公司不给我加班费怎么办？"}
]

最佳实践：

  • 明确角色定义
  • 设定回答边界（什么该答、什么不该答）
  • 指定输出格式
  • 给出行为准则

参数调优篇 ​

Temperature（温度） ​

是什么： 控制输出随机性的参数。范围通常是 0-2。

  • Temperature → 0：输出最确定、最"保守"。同样的输入几乎总是产生同样的输出。适合代码生成、数据提取。
  • Temperature = 1：默认值。在创造性和准确性之间平衡。
  • Temperature > 1：输出更随机、更有"创意"。适合头脑风暴、创意写作。

Top-P（核采样/Nucleus Sampling） ​

是什么： 另一种控制输出随机性的参数。范围 0-1。

  • Top-P = 0.1：只从概率最高的前 10% 候选词中采样
  • Top-P = 1：从所有候选词中采样

和 Temperature 的区别： 两者都是控制随机性，但机制不同。Temperature 改变概率分布的"形状"，Top-P 截断候选词的"范围"。一般建议只调其中一个。

Max Tokens（最大输出长度） ​

是什么： 限制模型最多生成多少个 token 的输出。

为什么要设置： 防止模型"话痨"，控制成本，避免超出上下文窗口。

Stop Sequences（停止序列） ​

是什么： 指定当模型输出出现某个特定字符串时立即停止生成。

用途： 比如你希望模型生成到"END"就停止：stop=["END"]

进阶技巧篇 ​

Few-shot（少样本学习） ​

是什么： 在 prompt 中给出几个"输入→输出"的示例，让模型通过"照猫画虎"来完成任务。

python
prompt = """将用户评论分类为正面、负面或中性。

评论：这个产品太棒了，值得购买！
分类：正面

评论：质量太差了，用了一天就坏了
分类：负面

评论：还行吧，凑合能用
分类：中性

评论：{用户输入的评论}
分类："""

类比： 就像给新员工看几个案例，让他知道该怎么做。

最佳实践：

  • 示例数量：通常 3-5 个就够
  • 示例要有代表性：覆盖典型和边界情况
  • 示例顺序有影响：把最相似的放在最后（靠近用户输入）

CoT（Chain-of-Thought，思维链） ​

是什么： 让模型在给出最终答案之前，先展示推理过程。

python
# 不用 CoT
prompt = "一个球拍和一个球一共 1.10 美元，球拍比球贵 1 美元。球多少钱？"

# 用 CoT
prompt = """一个球拍和一个球一共 1.10 美元，球拍比球贵 1 美元。球多少钱？
请一步一步思考。"""

为什么有效： 强制模型"慢思考"，减少跳步导致的错误。在数学题、逻辑推理、复杂分析任务中效果显著。

变体：

  • Zero-shot CoT：只需加一句"请一步一步思考"
  • Few-shot CoT：给出带推理过程的示例

Function Calling（函数调用） ​

是什么： 让 LLM 能够"调用"你定义的函数/工具。模型不会真正执行函数，而是告诉你"我需要调用这个函数，参数是什么"。你的代码负责真正执行。

python
# 定义函数
tools = [{
    "type": "function",
    "function": {
        "name": "get_weather",
        "description": "查询指定城市的天气",
        "parameters": {
            "type": "object",
            "properties": {
                "city": {"type": "string", "description": "城市名称"}
            },
            "required": ["city"]
        }
    }
}]

# 模型可能返回:
# {"function": "get_weather", "arguments": {"city": "北京"}}
# 你执行函数后把结果返回给模型

和 Agent 的关系： Function Calling 是 Agent 的基础能力。有了 Function Calling，LLM 就能"做事"而不仅仅是"说话"。

Streaming（流式输出） ​

是什么： 模型不是等全部生成完才返回，而是生成一个 token 就返回一个 token。

为什么重要：

  • 用户体验：用户能看到文字"一个一个蹦出来"，而不是等很久后突然显示一大段
  • 感觉上更快：虽然总耗时可能一样，但感知延迟大幅降低
python
# 流式调用示例
from openai import OpenAI

client = OpenAI()
stream = client.chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "写一首诗"}],
    stream=True
)

for chunk in stream:
    if chunk.choices[0].delta.content:
        print(chunk.choices[0].delta.content, end="")

应用架构篇 ​

RAG（Retrieval-Augmented Generation，检索增强生成） ​

是什么： 先用检索系统找到相关文档，再把文档内容和用户问题一起发给 LLM 生成回答。

一句话理解： 给 AI 开卷考试。

核心流程： 文档切分 → Embedding → 向量存储 → 检索 → 拼接 Prompt → LLM 生成

Embedding（向量嵌入） ​

是什么： 把文本转换成一组数字（向量），使得语义相似的文本对应的向量也相似。

用途： 语义搜索、RAG 检索、聚类、分类。

Vector Database（向量数据库） ​

是什么： 专门存储和检索向量的数据库。支持"找最相似的向量"这种查询。

常用产品： Milvus、Pinecone、Chroma、Weaviate、pgvector。

Agent（智能体） ​

是什么： 能自主规划、执行多步任务的 AI 系统。和普通聊天的区别是 Agent 能调用工具、做决策、处理中间结果。

Multi-modal（多模态） ​

是什么： 能同时处理多种类型信息（文本、图片、音频、视频）的 AI 能力。

Hallucination（幻觉） ​

是什么： 模型生成了看起来合理但实际错误的内容。比如编造不存在的论文、虚构历史事件、生成错误的代码。

怎么应对：

  • 使用 RAG 引入可靠数据源
  • 让模型在不确定时说"我不知道"
  • 对关键输出做事实验证
  • 使用更可靠的模型

模型相关篇 ​

Pre-training（预训练） ​

是什么： 在大规模通用数据上训练模型，让它学会语言的基本规律和广泛知识。这是模型的"基础教育"阶段。

Fine-tuning（微调） ​

是什么： 在预训练模型的基础上，用特定任务的数据进行额外训练，让模型更适合特定场景。

LoRA（Low-Rank Adaptation） ​

是什么： 一种轻量级微调方法。不修改原始模型参数，只训练一小部分额外参数。

优势： 显存需求低、训练快、可叠加。

Quantization（量化） ​

是什么： 把模型参数从高精度（如 FP32）压缩为低精度（如 INT8、INT4），减少显存占用和推理成本。

类比： 把高清照片压缩为低分辨率，文件变小了但基本不影响"看"。

Inference（推理） ​

是什么： 使用训练好的模型对新输入进行预测/生成的过程。也就是你日常调用 API 时模型做的事情。

Token/s（生成速度） ​

是什么： 模型每秒生成的 token 数。数值越高，用户体验越好。

参考值： GPT-4o 约 80-100 tokens/s，GPT-4o-mini 约 120+ tokens/s，本地小模型可能 20-50 tokens/s。

总结 ​

以上就是 AI 应用开发中最核心的概念和术语。作为开发者，你不需要深入了解每个概念的数学原理，但需要理解它们的含义、用途和限制。
