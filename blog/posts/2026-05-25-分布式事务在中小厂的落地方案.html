问题场景 ​

我们的订单服务需要在一个请求里完成三件事：扣减库存（库存服务）、创建订单（订单服务）、扣减账户余额（账户服务）。三个操作分布在三个不同的数据库里。

没有分布式事务之前，代码是这样的：

java
public void createOrder(OrderRequest request) {
    // 1. 扣库存
    stockService.deduct(request.getSkuId(), request.getQuantity());
    // 2. 创建订单
    orderService.create(request);
    // 3. 扣余额
    accountService.deduct(request.getUserId(), request.getAmount());
}

问题显而易见：如果步骤2成功但步骤3失败，库存扣了、订单创建了，但余额没扣。数据不一致。

分析思路 ​

1. 先问一个问题：真的需要分布式事务吗？ ​

这是我们团队内部讨论最久的一个问题。分布式事务引入的复杂度是巨大的，在拍板之前，我们先梳理了业务场景：

必须强一致性的场景：支付扣款 + 订单状态变更。这种钱相关的，不能用最终一致性。

可以用最终一致性的场景：订单创建后扣库存、发短信通知、推消息到ES。这些允许短暂不一致，最终能补偿就行。

结论：我们80%的场景不需要分布式事务，只有20%的核心链路需要。

2. 方案对比 ​

方案一致性性能侵入性适用场景
Seata AT最终一致（自动补偿）中等低（加注解）常规CRUD场景
Seata TCC强一致高高（写三个接口）核心链路、高并发
可靠消息（事务消息）最终一致高中等异步场景
本地消息表最终一致高中等简单场景

3. Seata AT 模式：我们的主力方案 ​

Seata AT 模式的核心原理：拦截业务SQL，自动生成反向补偿SQL。比如执行了 UPDATE account SET balance = balance - 100，Seata 会自动记录前镜像和后镜像，回滚时生成 UPDATE account SET balance = balance + 100。

引入依赖：

xml
<dependency>
    <groupId>io.seata</groupId>
    <artifactId>seata-spring-boot-starter</artifactId>
    <version>1.7.1</version>
</dependency>

配置：

yaml
seata:
  enabled: true
  application-order-service: order-service
  tx-service-group: my_tx_group
  service:
    vgroup-mapping:
      my_tx_group: default
    grouplist:
      default: 10.0.1.200:8091
  # AT模式必须建 undo_log 表
  data-source-proxy-mode: AT

每个参与分布式事务的数据库都要建 undo_log 表：

sql
CREATE TABLE undo_log (
    id BIGINT NOT NULL AUTO_INCREMENT,
    branch_id BIGINT NOT NULL,
    xid VARCHAR(128) NOT NULL,
    context VARCHAR(128) NOT NULL,
    rollback_info LONGBLOB NOT NULL,
    log_status INT NOT NULL DEFAULT 0,
    log_created DATETIME NOT NULL,
    log_modified DATETIME NOT NULL,
    PRIMARY KEY (id),
    UNIQUE KEY ux_undo_log (xid, branch_id)
) ENGINE=InnoDB;

业务代码：

java
@Service
public class OrderService {

    @GlobalTransactional(name = "create-order", timeoutMills = 30000)
    public void createOrder(OrderRequest request) {
        // 1. 扣库存（远程调用库存服务）
        stockFeignClient.deduct(request.getSkuId(), request.getQuantity());

        // 2. 创建订单（本地操作）
        Order order = new Order();
        order.setUserId(request.getUserId());
        order.setAmount(request.getAmount());
        order.setStatus(OrderStatus.CREATED);
        orderMapper.insert(order);

        // 3. 扣余额（远程调用账户服务）
        accountFeignClient.deduct(request.getUserId(), request.getAmount());

        // 如果任何一步失败，Seata 自动回滚所有已执行的操作
    }
}

AT 模式的坑：

  1. 全局锁的性能问题。Seata AT 在提交时需要获取全局锁，高并发下会成为瓶颈。我们的订单服务在大促时就遇到了——大量请求排队等全局锁，RT翻倍。
  2. undo_log 表会膨胀。事务提交后需要清理 undo_log，如果清理不及时会占大量空间。
  3. 不支持含有外键的表。Seata AT 解析 SQL 时对有外键约束的表支持不好，我们把所有外键都删了，改成代码层面保证数据完整性。

4. Seata TCC 模式：核心链路的选择 ​

AT 模式的全局锁在高并发下性能不好，所以核心支付链路我们用了 TCC 模式。

TCC 需要手动写三个接口：Try（预留资源）、Confirm（确认）、Cancel（取消）。

java
@LocalTCC
public interface AccountTccAction {

    @TwoPhaseBusinessAction(
        name = "deductAccount",
        commitMethod = "confirm",
        rollbackMethod = "cancel"
    )
    boolean tryDeduct(
        BusinessActionContext context,
        @BusinessActionContextParameter(paramName = "userId") Long userId,
        @BusinessActionContextParameter(paramName = "amount") BigDecimal amount
    );

    boolean confirm(BusinessActionContext context);

    boolean cancel(BusinessActionContext context);
}

@Component
public class AccountTccActionImpl implements AccountTccAction {

    @Override
    @Transactional
    public boolean tryDeduct(BusinessActionContext context, Long userId, BigDecimal amount) {
        // Try阶段：冻结金额（不是直接扣减）
        int rows = accountMapper.freezeAmount(userId, amount);
        return rows > 0;
    }

    @Override
    @Transactional
    public boolean confirm(BusinessActionContext context) {
        Long userId = (Long) context.getActionContext("userId");
        BigDecimal amount = new BigDecimal(context.getActionContext("amount").toString());
        // Confirm阶段：扣减冻结金额
        accountMapper.deductFrozenAmount(userId, amount);
        return true;
    }

    @Override
    @Transactional
    public boolean cancel(BusinessActionContext context) {
        Long userId = (Long) context.getActionContext("userId");
        BigDecimal amount = new BigDecimal(context.getActionContext("amount").toString());
        // Cancel阶段：解冻金额
        accountMapper.unfreezeAmount(userId, amount);
        return true;
    }
}

TCC 的核心思想：Try 阶段只预留资源不实际扣减，Confirm 和 Cancel 分别做实际操作和反向操作。好处是没有全局锁，性能更好。坏处是业务侵入性强，每个参与者都要写三个接口，还要处理空回滚、悬挂、幂等这些异常场景。

5. 事务消息：异步场景的最终一致性 ​

订单创建后需要通知积分服务加积分、通知物流服务，这些不需要强一致性，用 RocketMQ 的事务消息：

java
@Service
public class OrderMessageService {

    @Autowired
    private RocketMQTemplate rocketMQTemplate;

    public void sendOrderCreatedEvent(Order order) {
        // 发送半消息（half message）
        rocketMQTemplate.sendMessageInTransaction(
            "TOPIC_ORDER_EVENT",
            MessageBuilder.withPayload(new OrderEvent(order)).build(),
            order
        );
    }

    @RocketMQTransactionListener
    public class OrderTransactionListener implements RocketMQLocalTransactionListener {

        @Override
        public RocketMQLocalTransactionState executeLocalTransaction(Message msg, Object arg) {
            try {
                Order order = (Order) arg;
                // 检查订单是否真正创建成功
                Order dbOrder = orderMapper.selectById(order.getOrderId());
                if (dbOrder != null) {
                    return RocketMQLocalTransactionState.COMMIT;
                }
                return RocketMQLocalTransactionState.UNKNOWN;
            } catch (Exception e) {
                return RocketMQLocalTransactionState.ROLLBACK;
            }
        }

        @Override
        public RocketMQLocalTransactionState checkLocalTransaction(Message msg) {
            // 回查机制：检查本地事务是否完成
            OrderEvent event = JSON.parseObject(new String((byte[]) msg.getPayload()), OrderEvent.class);
            Order order = orderMapper.selectById(event.getOrderId());
            return order != null
                ? RocketMQLocalTransactionState.COMMIT
                : RocketMQLocalTransactionState.ROLLBACK;
        }
    }
}

踩坑总结 ​

  1. 能不用分布式事务就不要用。单库能解决的问题不要跨库。如果非要跨库，先考虑能不能把数据合到一个库里。
  2. Seata AT 适合常规场景，加个注解就行。但全局锁在高并发下是瓶颈，核心链路用 TCC。
  3. TCC 的三个接口必须都实现幂等。Confirm 和 Cancel 可能被重试调用多次。
  4. TCC 要处理好三个异常场景：空回滚（Try 没执行但 Cancel 被调了）、悬挂（Cancel 先于 Try 执行）、幂等。
  5. 事务消息是最轻量的方案，适合"写完发消息通知下游"的场景。不需要引入 Seata，RocketMQ 原生支持。
  6. 分布式事务的超时时间要合理设置。太短的话正常的业务处理就被回滚了，太长的话资源被占用。我们的经验值是：普通场景30秒，核心支付场景15秒。
  7. undo_log 表要定期清理。我们有次忘了清理，表涨到50GB，差点把磁盘撑爆。
