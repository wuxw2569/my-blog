问题场景 ​

我们的核心业务表 t_order 单表已经到了3000万行，t_order_detail 更是到了5000万。每天新增订单15万+，查询RT逐渐劣化，虽然加了索引、做了读写分离，但单表的数据量瓶颈已经很难通过优化SQL来解决了。

老板说"加机器"，但单机MySQL的瓶颈不在CPU和内存，而在于：单表数据量过大后B+树层数增加、索引维护成本升高、DDL操作（加字段加索引）耗时以小时计。

是时候分库分表了。

分析思路 ​

1. 选型：ShardingSphere 还是自己搞？ ​

我们评估了三个方案：

方案优点缺点
ShardingSphere-JDBC客户端分片，无中间件开销；社区活跃；Spring Boot 集成好侵入性强，每个服务都要引入依赖
ShardingSphere-Proxy对应用透明，像连普通MySQL多一层网络跳转，运维复杂
自研分表路由完全可控开发维护成本高，中小厂没必要

我们最终选了 ShardingSphere-JDBC 5.4.1。原因很直接：团队30人，后端8人，没有专门的中间件团队，需要一个成熟、文档齐全、Spring Boot 一键集成的方案。ShardingSphere-Proxy 多一层网络跳转，出了问题排查链路更长，不适合我们这个规模。

2. 分片键的选择——这是最关键的决策 ​

分片键选错了，后面全是坑。我们的核心查询场景：

  • 用户查自己的订单（按 user_id）
  • 运营查某天的订单（按 create_time）
  • 财务对账查订单号（按 order_id）

最终选择 user_id 作为分片键。理由：

  1. 95%+ 的查询都带 user_id 条件，走分片键可以直接路由到单库单表
  2. 同一用户的订单数据都在同一个分片，事务不需要跨库
  3. 按时间分片的问题是：老数据不活跃但集中在某个分片，数据分布不均匀

分片规则：user_id % 64 分到64张表，分布在4个库中（每个库16张表）。

yaml
# application-sharding.yml
spring:
  shardingsphere:
    datasource:
      names: ds0,ds1,ds2,ds3
      ds0:
        type: com.zaxxer.hikari.HikariDataSource
        jdbc-url: jdbc:mysql://10.0.1.100:3306/order_db_0
        username: order_user
        password: ${DB_PASSWORD}
      # ds1, ds2, ds3 省略...
    rules:
      sharding:
        tables:
          t_order:
            actual-data-nodes: ds$->{0..3}.t_order_$->{0..63}
            table-strategy:
              standard:
                sharding-column: user_id
                sharding-algorithm-name: t-order-inline
            database-strategy:
              standard:
                sharding-column: user_id
                sharding-algorithm-name: db-inline
        sharding-algorithms:
          t-order-inline:
            type: INLINE
            props:
              algorithm-expression: t_order_$->{user_id % 64}
          db-inline:
            type: INLINE
            props:
              algorithm-expression: ds$->{user_id % 64 / 16}

3. 分布式主键方案 ​

分库分表后自增主键不能用了，必须用全局唯一ID。我们用了雪花算法，但踩了个坑：

java
// 直接用 ShardingSphere 内置的雪花算法
spring.shardingsphere.rules.sharding.tables.t_order.key-generate-strategy.column=order_id
spring.shardingsphere.rules.sharding.tables.t_order.key-generate-strategy.key-generator-name=snowflake

踩坑：服务器时钟回拨会导致雪花ID重复。我们的运维做了一次NTP同步，导致两台服务器时钟回拨了3ms，产生了重复主键，虽然概率极低但确实发生了。后来加了时钟回拨检测：

java
@Component
public class SafeSnowflakeIdGenerator {
    private final Snowflake snowflake;

    public SafeSnowflakeIdGenerator() {
        // 使用 Hutool 的雪花算法，自带时钟回拨等待
        this.snowflake = IdUtil.getSnowflake(1, 1);
    }

    public long nextId() {
        return snowflake.nextId();
    }
}

具体实现 ​

数据迁移——最危险的环节 ​

分库分表最大的风险不是代码，而是数据迁移。我们的方案是双写+追数据：

第一步：全量迁移

java
// 用 Spring Batch 做全量数据迁移
@Bean
public Step migrateOrderStep() {
    return stepBuilder.get("migrateOrderStep")
        .<Order, Order>chunk(1000)
        .reader(orderJdbcReader())  // 从老库读
        .processor(orderProcessor()) // 计算分片路由
        .writer(orderJdbcWriter())   // 写入新库
        .build();
}

第二步：增量同步

全量迁移耗时约4小时，期间新产生的数据用 Canal 订阅 binlog 同步到新库：

java
@CanalEventListener
public class OrderSyncListener {

    @InsertListenPoint
    public void onInsert(CanalEntry.RowData rowData) {
        Order order = convertToOrder(rowData);
        // 写入新的分库分表
        orderShardingService.insert(order);
    }

    @UpdateListenPoint
    public void onUpdate(CanalEntry.RowData rowData) {
        Order order = convertToOrder(rowData);
        orderShardingService.updateById(order);
    }
}

第三步：双写验证

切换期间同时写新老两套库，用定时任务对比数据一致性：

java
@Scheduled(cron = "0 0/5 * * * ?")
public void verifyDataConsistency() {
    // 抽样对比老库和新库的数据
    List<Long> sampleIds = oldOrderDao.sampleOrderIds(100);
    for (Long id : sampleIds) {
        Order oldOrder = oldOrderDao.selectById(id);
        Order newOrder = orderShardingService.selectByOrderId(id);
        if (!Objects.equals(oldOrder, newOrder)) {
            log.error("【数据不一致】orderId={}, old={}, new={}", id, oldOrder, newOrder);
        }
    }
}

跨库查询的解决方案 ​

分库分表后最大的痛点：不带 user_id 的查询怎么办？

比如运营要查某天所有订单、财务要按订单号查流水，这些查询无法直接路由到具体分片。

方案一：基因法（推荐）

把 user_id 的低几位编码到 order_id 中，这样通过 order_id 反推出分片：

java
public class ShardingKeyExtractor {
    /**
     * 从 order_id 的低6位提取 user_id 的分片信息
     * order_id = 雪花ID + 分片后缀
     */
    public static int extractShardIndex(long orderId) {
        return (int)(orderId % 64);
    }
}

方案二：异构索引表

单独建一张 t_order_mapping 表，记录 order_id 到 user_id 的映射，这张表不分片或用其他方式分片：

sql
CREATE TABLE t_order_mapping (
    order_id BIGINT PRIMARY KEY,
    user_id BIGINT NOT NULL,
    create_time DATETIME NOT NULL
) ENGINE=InnoDB;

查询时先查 mapping 表拿到 user_id，再路由到具体分片。

方案三：对于复杂查询场景，直接走 ES

我们在 Canal 同步数据到新库的同时，也同步了一份到 Elasticsearch。运营和财务的复杂查询走 ES，简单查询走分库分表。

踩坑总结 ​

  1. 分片键选定后基本不可变。重新分片等于再来一次数据迁移，所以一定要基于查询场景仔细分析，宁可多花一周调研也不要匆忙定分片键。
  2. 分片数量要预估3-5年的数据量。我们选64张表是因为预估5年后总量到5亿。如果一开始只分8张表，两年后又要扩容，极其痛苦。
  3. 扩容方案提前设计。我们用的翻倍扩容法：64表扩到128表，本质就是偶数号表不动、奇数号表从偶数号表拆分出来。前提是分片数是2的幂。
  4. 分布式事务是绕不开的问题。跨库的写操作要用 Seata 或者改成最终一致性。我们大部分业务通过合理的分片键设计避免了跨库事务。
  5. DDL 变更要提前规划。分库分表后加字段要改64张表，用 Flyway 或 Liquibase 管理变更脚本，千万别手动改。
  6. 不要过早分库分表。单表2000万以下，优先考虑优化索引、读写分离、归档冷数据。分库分表的复杂度是永久的，不到万不得已不要上。
