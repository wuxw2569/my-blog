RAG 检索增强生成技术入门 ​

前言 ​

如果你只记住一个 AI 应用开发的技术名词，那应该是 RAG。

RAG（Retrieval-Augmented Generation，检索增强生成）是目前企业落地 AI 应用最主流的架构模式。几乎所有"让 AI 读你的文档来回答问题"的应用，背后都是 RAG。

为什么 RAG 如此重要？先从一个问题说起。

为什么需要 RAG？ ​

LLM 有三个致命缺陷：

  1. 知识有截止日期：模型的训练数据有时间边界，不知道最新的信息
  2. 不了解你的私有数据：模型没读过你公司的内部文档、产品手册、业务数据
  3. 会"幻觉"：当模型不确定时，它可能会编造一个看起来很合理但实际错误的回答

你可能会想：那我直接把所有文档塞到 prompt 里不就行了？

问题在于：

  • 文档可能有几十万字，超出上下文窗口
  • 即使塞得进去，模型对长文本中间部分的关注度很低
  • 每次调用都塞大量文本，token 成本会爆炸

RAG 的核心思想就是：先帮模型"查资料"，再让它"回答问题"。 就像开卷考试一样，先翻书找到相关章节，再回答问题。

RAG 的基本流程 ​

一个标准的 RAG 系统分为两个阶段：离线索引阶段 和 在线查询阶段。

阶段一：离线索引（数据准备） ​

原始文档（PDF/Word/网页/数据库）
  ↓
[文档解析] 提取纯文本
  ↓
[文本切分] 把长文档分成小块（chunk）
  ↓
[Embedding] 每个文本块变成向量
  ↓
[向量存储] 向量 + 原文一起存入向量数据库

这一步是离线完成的，相当于给你的知识库建"索引"。

阶段二：在线查询（用户提问时） ​

用户提问："我们的退货政策是什么？"
  ↓
[Embedding] 把问题变成向量
  ↓
[向量检索] 在向量数据库中找到最相关的几个文本块
  ↓
[构建 Prompt] 把检索到的文本块 + 用户问题 组装成 prompt
  ↓
[LLM 生成] 让 LLM 基于检索到的内容生成回答
  ↓
返回给用户："根据公司政策，7天内无理由退货..."

各环节关键决策 ​

1. 文档解析 ​

不同格式的文档需要不同的解析方式：

格式常用工具
PDFPyPDF2、pdfplumber、Unstructured
Wordpython-docx
网页BeautifulSoup、Jina Reader
Markdown直接读取
数据库SQL 查询导出

2. 文本切分（Chunking） ​

这是 RAG 中最容易被低估但影响最大的环节。切分策略直接决定检索质量。

常用切分方式：

python
# 递归字符切分（最常用）
from langchain.text_splitter import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,        # 每块最大 500 字符
    chunk_overlap=50,      # 相邻块重叠 50 字符，避免切断上下文
    separators=["\n\n", "\n", "。", "！", "？", "，", " "]
)

chunks = splitter.split_text(long_document)

切分的最佳实践：

  • chunk_size 一般在 300-1000 个 token 之间，需要根据实际测试调优
  • 保持适当的 overlap（重叠），避免关键信息被切断
  • 如果文档有明确结构（标题、段落），按结构切分效果更好
  • 可以为每个 chunk 保留元数据（标题、来源、页码等）

3. 检索策略 ​

基础的向量检索之外，还有进阶策略：

混合检索（Hybrid Search）：结合向量检索和关键词检索，取长补短。向量检索擅长理解语义，关键词检索擅长精确匹配（如产品型号、错误代码）。

python
# 混合检索的伪代码
def hybrid_search(query, vector_db, keyword_db, alpha=0.7):
    # alpha 控制两种检索的权重
    vector_results = vector_db.search(query)      # 语义检索
    keyword_results = keyword_db.search(query)    # 关键词检索
    
    # 加权融合排序
    final_results = merge_and_rerank(
        vector_results, keyword_results, alpha
    )
    return final_results

重排序（Rerank）：先用向量检索召回 20-50 个候选结果，再用专门的 Rerank 模型精排，选出最相关的 Top 3-5。这一步能显著提升检索质量。

多路召回：同时用多种方式检索（向量、关键词、知识图谱等），合并去重后再排序。

4. Prompt 构建 ​

把检索到的内容组装进 prompt 时，有一些技巧：

你是一个专业的客服助手。请根据以下参考资料回答用户问题。
如果参考资料中没有相关信息，请直接说"我没有找到相关信息"，不要编造。

## 参考资料：
[1] {检索到的文本块1}
[2] {检索到的文本块2}
[3] {检索到的文本块3}

## 用户问题：
{用户的问题}

关键点：

  • 明确告诉模型"基于参考资料回答"，减少幻觉
  • 让模型在找不到信息时坦诚说"不知道"
  • 如果有多个检索结果，标上编号方便追溯

RAG vs 微调（Fine-tuning） ​

很多人会问：我想让模型了解我的数据，是用 RAG 还是微调？

维度RAG微调
成本低，不需要训练高，需要 GPU 和标注数据
更新频率实时，改文档就行需要重新训练
可追溯性好，知道答案来自哪段文档差，无法追溯
适合场景知识问答、文档检索风格迁移、格式调整
复杂度中等较高

简单来说：RAG 是给模型"开卷考试"，微调是给模型"上课"。

大多数场景下，RAG 是更好的选择。只有当你需要改变模型的"能力"（而不仅仅是"知识"）时，才需要微调。

一个最小可行的 RAG 系统 ​

如果你想快速体验 RAG，以下是一个用 LangChain + Chroma 搭建的最简示例：

python
from langchain_community.document_loaders import TextLoader
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain_community.vectorstores import Chroma
from langchain_openai import OpenAIEmbeddings, ChatOpenAI
from langchain.chains import RetrievalQA

# 1. 加载文档
loader = TextLoader("my_docs.txt")
documents = loader.load()

# 2. 切分
splitter = RecursiveCharacterTextSplitter(chunk_size=500, chunk_overlap=50)
chunks = splitter.split_documents(documents)

# 3. Embedding + 存储
vectorstore = Chroma.from_documents(chunks, OpenAIEmbeddings())

# 4. 构建 QA 链
qa_chain = RetrievalQA.from_chain_type(
    llm=ChatOpenAI(model="gpt-4o"),
    retriever=vectorstore.as_retriever(search_kwargs={"k": 3}),
)

# 5. 提问
answer = qa_chain.invoke("退货政策是什么？")
print(answer["result"])

RAG 的常见踩坑点 ​

  1. 检索质量不好：大部分 RAG 系统效果差的原因不是 LLM 不行，而是检索没找对文档。优先优化切分策略和检索方式。

  2. Chunk 太大或太小：太大导致包含太多无关信息，太小导致丢失上下文。建议从 500 token 开始调。

  3. 忽略元数据：给每个 chunk 附加来源、标题、日期等元数据，检索时可以做过滤和加权。

  4. 幻觉问题没解决：即使用了 RAG，模型仍可能基于检索结果编造内容。需要在 prompt 中明确要求"基于给定内容回答，不要编造"。

  5. 没有评估机制：需要建立一套评估 pipeline，用标注数据集定期评估检索质量和回答质量。

总结 ​

RAG 是当前最实用的 AI 应用架构模式。核心思想很简单：先检索，再生成。但要做好一个高质量的 RAG 系统，需要在文档解析、文本切分、Embedding 选择、检索策略、Prompt 工程等每个环节下功夫。

作为开发者，建议从最简单的 RAG 流程开始跑通，然后逐步优化每个环节。下一篇我们会讲 AI Agent——一种比 RAG 更进一步的 AI 应用形态。
